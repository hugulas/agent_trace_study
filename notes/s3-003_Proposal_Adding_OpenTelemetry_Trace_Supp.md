# [Proposal] Adding OpenTelemetry Trace Support to MCP

## 核心信息

- **标题**: [Proposal] Adding OpenTelemetry Trace Support to MCP
- **作者**: altryne（提案发起人，GitHub 社区成员）
- **发布日期**: 2025 年 4 月 2 日
- **来源类型**: GitHub Discussion（modelcontextprotocol/modelcontextprotocol #269）
- **URL**: https://github.com/modelcontextprotocol/modelcontextprotocol/discussions/269
- **PDF**: `[s3-003]-proposal-adding-opentelemetry-trace-support-to-mcp.pdf`
- **证据质量**: high
- **关联概念**: Model Context Protocol（MCP）、OpenTelemetry（OTel）、分布式追踪、traceToken、通知机制、协议能力协商（capability negotiation）、OTLP/JSON

## 内容摘要

这篇 GitHub Discussion 是由社区成员 altryne 于 2025 年 4 月发起的一份正式提案，旨在为 Model Context Protocol（MCP）引入标准化的 OpenTelemetry（OTel）追踪能力，以解决当前 Agent 工作流中 MCP 服务器内部操作对客户端不可见的"黑箱"问题。随着 MCP 被誉为"Agent 的 HTTP"，基于该协议构建的应用复杂度正在迅速攀升，开发者越来越依赖 Weights & Biases Weave、LangSmith、Braintrust、Arize Phoenix 等可观测性平台来理解和调试多步工具调用与 LLM 交互的执行流程。然而，当前当 Agent（MCP 客户端）通过 MCP 服务器调用工具或资源时，客户端只能看到请求和最终响应，服务器内部的执行细节完全不可见，这为性能瓶颈定位、错误根因诊断、复杂交互路径还原以及端到端可观测性带来了根本性障碍。

提案的核心机制是：在保留 MCP 现有协议结构的前提下，新增一种专用的服务器到客户端通知类型 `notifications/otel/trace`，使 MCP 服务器能够将其内部生成的 OTel trace span 数据回传给调用方客户端。该方案选择 OTel 作为标准格式，因其厂商中立、广泛采纳且拥有丰富的语义约定（包括不断演进的 GenAI 相关约定）。在传输机制上，提案主张利用 MCP 已有的异步通知通道（类似于现有的 `notifications/message` 日志机制），而非要求服务器直接推送数据到外部 OTel Collector。这一设计简化了服务器实现（无需配置导出器或处理外部后端认证），同时将数据流向控制权保留在客户端手中，客户端可决定如何处理和转发接收到的追踪数据。

为了实现请求与追踪数据的可靠关联，提案借鉴了现有的 `progressToken` 模式，引入 `traceToken` 概念：客户端在发起请求时于 `params._meta` 中嵌入一个唯一的 `traceToken`，支持 `otel.traces` 能力的服务器在处理该请求后，必须在返回的追踪通知中原样回传该 token，从而使客户端能够将服务器端的 span 与客户端侧的起始 span 进行父子关联和时序拼接。服务器需在初始化阶段通过 `capabilities.otel.traces: true` 宣告其支持该功能。提案还详细讨论了安全考量（追踪数据可能包含敏感信息，服务器实现者有责任进行脱敏处理）、向后兼容性（纯增量式扩展，不影响现有功能）以及三种替代方案的权衡：服务器直接推送到 Collector（因增加配置复杂度被否决）、复用现有日志通知机制（因语义混淆被否决作为长期方案）、以及客户端到服务器的 W3C TraceContext 传播（被推迟到未来提案）。

该 Discussion 引发了社区的高度关注和激烈辩论，累计 21 条回复、16 位参与者。微软的 samsp-msft、OpenTelemetry 维护者 lmolkova、Pydantic 作者 samuelcolvin 等重量级开发者均参与了讨论。反对方的主要论点包括：该设计与 OTel 的标准实践（各节点自主导出到 Collector）存在显著偏离；可能暴露服务器内部实现细节，引发安全和竞争担忧；在 MCP 服务器链式调用的场景下可能导致响应体体积爆炸；以及客户端作为 OTel 代理/Collector 的角色与传统分布式追踪架构不符。支持方则认为，MCP 服务器本质上是客户端工作流中的"工具调用"，由客户端统一收集和转发追踪数据更符合其架构逻辑，且能避免为每个服务器单独配置 OTLP 端点的运维负担。支持方还指出，对于托管型 MCP 服务器注册表（如 MCP.run）而言，服务器根本无法获知每个客户端的私有 OTel 后端地址，因此将 span 回传给客户端是唯一可行的方案。codefromthecrypt 等社区成员建议优先聚焦于 trace context 传播（如 traceparent/baggage），先解决关联问题，再逐步探索数据格式和传输机制的细化方案。

## 关键要点

1. **MCP 黑箱问题的诊断困境**

   当前 MCP 客户端调用服务器工具时，无法判断性能瓶颈出现在客户端逻辑、网络传输还是服务器内部执行（如慢速 API 调用或低效处理）。提案将这一问题归纳为四个具体挑战：调试瓶颈、错误诊断、复杂交互理解和可观测性不完整。

2. **专用通知类型的语义清晰性**

   提案力主新增 `notifications/otel/trace` 专用通知类型，而非复用现有的 `notifications/message` 日志机制。理由包括：语义清晰（明确区分追踪信号与通用日志）、客户端易于路由处理、以及为未来扩展 Metrics（`notifications/otel/metrics`）和 Logs（`notifications/otel/logs`）提供清晰的模式。

3. **traceToken 的关联机制设计**

   `traceToken` 借鉴了 MCP 已有的 `progressToken` 设计模式，作为客户端请求与服务器追踪通知之间的关联纽带。客户端生成唯一 token 嵌入请求元数据，服务器在处理完成后回传该 token，确保在并发场景下也能无歧义地将服务器 span 与正确的客户端请求进行匹配和父子关联。

4. **服务器能力协商的协议扩展**

   该提案遵循 MCP 的能力协商范式：服务器通过 `capabilities.otel.traces: true` 宣告支持，客户端据此判断是否请求追踪数据。这种设计确保了向后兼容性——不支持该能力的服务器和客户端可以互不干扰地继续运行。

5. **OTLP/JSON 作为序列化标准**

   追踪数据负载采用 OpenTelemetry 的 OTLP/JSON 格式进行序列化，具体包含 `resourceSpans` 数组，内部嵌套 `scopeSpans` 和具体的 `spans` 对象，涵盖 traceId、spanId、parentSpanId、name、kind、时间戳、属性和状态等标准字段。

6. **社区分歧的核心焦点：OTel 架构正统性 vs MCP 场景特殊性**

    debate 的本质并非反对可观测性，而是对实现路径的分歧。反对方（lmolkova、samuelcolvin、samsp-msft）认为各服务自主导出到 Collector 是 OTel 的标准模式，MCP 不应例外；支持方（altryne、dmontagu）则认为 MCP 服务器的"工具化"特性使其更适合由客户端代理收集。

7. **安全与隐私的深层张力**

   samuelcolvin 和 LucaButBoring 指出，如果服务器默认将所有内部追踪数据发送给客户端，可能导致 SQL 查询、内部 API 调用等敏感信息泄露。即便是同一信任边界内的部署，不同厂商的服务器也可能需要同时维护两套并行的 OTel 系统（内部 Collector 一套、客户端一套），增加了实现复杂度和出错风险。

8. **数据量与性能的现实约束**

   Cirilla-zmh 等人担忧，在 MCP 服务器构成 DAG（有向无环图）的链式调用场景中，追踪数据在服务器之间来回传递可能导致网络流量激增。相比之下，每个服务器独立导出到统一 Collector 的设计更为简洁。

9. **增量式采纳与向后兼容**

   提案强调该扩展是纯增量式的：不支持 `otel.traces` 的客户端和服务器将忽略相关消息和能力标志，现有功能完全不受影响。服务器可以逐步、独立地采纳该特性，无需协议版本升级或全局协调部署。

10. **传播优先的务实建议**

   codefromthecrypt 和 armorer-labs 建议将问题拆分为两层：第一层是 trace correlation（通过 traceparent/baggage 传播关联上下文），第二层是 Agent 运行记录（更高层次的执行事实，如模型 ID、工具注册表快照、审批状态等）。优先解决传播问题，可以为后续的任何数据交换机制奠定基础。

10. **上下文收据（Context Receipts）的替代思路**

    caioribeiroclw-pixel 提出了一种隐私安全的替代方案：不发送原始内部 span，而是生成紧凑的上下文收据（包含哈希、计数、分类决策和显式审计缺口），通过 runId/sessionId/turnId/toolCallId/traceparent 与 OTel 追踪关联。这种分层设计既回答了"哪些组件参与了执行"，又以隐私安全的方式回答了"哪些信息跨越了 Agent 上下文边界"。

11. **提案的局限与待解决问题**

    提案本身也承认若干开放性问题：OTLP/JSON 序列化格式的最终确认、MCP 操作（如 `tools/call`、`resources/read`）的标准 OTel 语义约定定义、客户端向服务器请求特定追踪详细程度的机制，以及 OTel Metrics 和 Logs 的后续扩展路径。这些未决问题表明该提案仍处于概念验证和社区讨论的早期阶段。

12. **现实世界的类比：从 LLM SDK 仪器化中汲取经验**

    anuraaga 在评论中指出，现有的 LLM SDK 仪器化（如 OpenAI 仪器化）已经采用了类似模式：客户端 span 捕获总体耗时，并通过响应中的 usage 字段进行 enrich，而服务器（如 OpenAI 的 API）本身是黑箱。只有当通过 vLLM 等自托管方案连接时，才能获得更详细的遥测。这一类比说明，MCP 的追踪问题并非全新，而是与现有 LLM 遥测实践一脉相承。

从协议演进的角度看，该提案直接触及了 MCP 作为"Agent 基础设施协议"走向生产就绪所必须解决的核心问题：如何在保持协议简洁性的同时，满足企业级可观测性需求。综述在讨论 Agent 系统可观测性架构时，可以引用该提案作为"协议层追踪"方案的代表性设计，并与传统的"应用层自主导出"方案（如 lmolkova 所主张的标准 OTel 模式）形成对比论证。

在分布式追踪的语境下，本文揭示了一个深层的设计哲学张力：通用分布式追踪框架（如 OTel）的设计假设是每个服务节点是自主运行的微服务，而 MCP 的架构假设是服务器是客户端工作流中的"可调用工具"。这两种模型对"谁应该拥有追踪数据"和"数据应该流向何处"给出了截然不同的答案。综述在构建 Agent 可观测性的理论框架时，可以将这种张力作为一个核心分析维度，探讨 Agent 系统是否需要一套区别于传统微服务的追踪语义和架构模式。

从社区治理的角度，该 Discussion 还反映了开源协议标准化过程中多方利益相关者（协议维护者、OTel 官方、可观测性厂商、应用开发者、安全专家）之间的博弈与协商。这对于综述理解技术标准的形成机制，以及为什么某些看似合理的技术提案最终可能无法被采纳，提供了生动的案例素材。

此外，Discussion 中提出的 context receipts 和分层可观测性思路，为综述中"Agent 可解释性与隐私保护的平衡"章节提供了前沿的解决方案参考。特别是在金融、医疗等受监管行业中，如何在提供足够调试信息的同时避免泄露敏感内部实现细节，是一个具有普遍意义的挑战。

从可观测性技术的产业格局来看，该 Discussion 的参与者背景极具代表性：既有 OTel 官方维护者（lmolkova），也有主流 AI 可观测性平台从业者（Weights & Biases、Arize、Braintrust 等），还有核心基础设施开发者（Pydantic、Dagger、Elastic 等）。这种多元利益相关者的参与格局意味着，任何关于 MCP 可观测性的标准化决策都必须平衡技术正确性、安全合规性、生态互操作性和商业可行性等多重维度。

值得特别关注的是，shykes 指出 Dagger 在同时实现 MCP 和 OTel 的情况下，无需扩展任一协议即可获得完整的工具可观测性。这一经验性观察对提案的前提假设构成了直接挑战：如果统一运行时层（如 Agent SDK 和框架）或注入 OTel Collector 配置即可解决问题，那么协议层面的扩展是否真的是必需的？综述在讨论"协议层扩展 vs 运行时层解决"这一架构选择时，可以引用 Dagger 的实践作为反事实参照。

## 我的笔记

这篇 GitHub Discussion 是我所读到的关于 MCP 可观测性最深入、最富张力的原始技术资料之一。其价值不仅在于提案本身的设计细节，更在于围绕该提案展开的多元观点碰撞，这些碰撞暴露了当前 Agent 可观测性领域许多尚未被充分讨论的深层问题。

我认同 altryne 对 MCP 黑箱问题的诊断：当 Agent 调用外部工具时，如果无法看到工具内部的执行细节，调试效率将大打折扣。文中以 Sarah 调试客服 Agent 调用 Notion 工具的场景为例，生动地展示了 `notion_api_request` span 如何帮助开发者将瓶颈定位到 Notion API 本身而非 MCP 服务器逻辑或网络层。这一用例具有很强的现实说服力。

然而，我也认为反对方的论点同样有力，甚至在某些方面更为根本。lmolkova 指出，如果服务器不真正参与分布式追踪（即不向下游传播 trace context），端到端追踪链就会断裂；而即便解决了传播问题，让服务器手工构造要回传给客户端的 span 也极为困难，自动仪器化将完全失效。这一观察击中了提案的一个关键软肋：该方案可能只能提供一层额外的"服务器摘要 span"，而无法替代真正的端到端分布式追踪。

samuelcolvin 从安全角度提出的五点反对意见尤其值得重视。他认为，任何在请求处理过程中产生的 OTel 仪器化数据默认发送给客户端，对于 GitHub、银行等拥有敏感内部实现的服务提供者来说是不可接受的。我注意到，altryne 对此的回应是"服务器可以选择是否发送追踪数据"，但这实际上意味着可观测性将变成服务器的一项可选特性而非协议保证，从而削弱了整个方案的价值主张。

dmontagu 提出的"MCP Client 作为 OTel 代理"概念化框架非常有启发性。他将客户端定位为服务器追踪数据的代理/收集器，这种视角转换有助于理解提案的设计动机。但同时，他自己也承认这种设计在安全问题（意外泄露实现细节）和数据量问题（客户端可能成为瓶颈）方面存在现实障碍。

codefromthecrypt 的建议——先聚焦于 trace context 传播（如 W3C traceparent），获取实际运行经验后再决定是否需要自定义数据格式——在我看来是最为务实的路径。事实上，如果客户端和服务器能够通过标准化的传播机制共享 trace context，那么服务器完全可以使用标准的 OTel 仪器化并将数据发送到客户端也能访问的 Collector，从而在不引入 MCP 特定机制的情况下实现追踪关联。

armorer-labs 和 caioribeiroclw-pixel 提出的"分层可观测性"思路——将 trace correlation（追踪关联）与 context receipts（上下文收据）分离——可能是解决当前僵局的最有前景的方向。这种设计既保留了分布式追踪在延迟调试方面的价值，又通过隐私安全的收据机制回答了更高层次的 Agent 治理问题（如"哪个 Agent 运行、在何种工具和政策状态下导致了该行为"）。

最后，我注意到该 Discussion 中反复出现的一个主题是"信任边界"（trust boundary）。当 MCP 客户端和服务器属于同一组织时，两种模型（客户端代理 vs 服务器自主导出）的行为差异并不大；但当它们跨越组织边界时，安全风险和配置复杂度的差异将被急剧放大。这一观察对于综述讨论 Agent 系统的安全架构和跨组织协作模式具有重要参考价值。未来 MCP 可观测性的标准化进程，很可能需要在"同信任边界"和"跨信任边界"两种场景下给出差异化的推荐方案。

从更长远的视角看，这场辩论实际上预示了 Agent 基础设施发展的一个普遍规律：当一个新的抽象层（如 MCP）开始承载生产级工作负载时，它必然会与现有成熟标准（如 OTel）产生摩擦和重新协商。这种摩擦不是技术失败的信号，而是技术成熟度的标志。回顾微服务架构早期，服务网格（service mesh）与语言原生 SDK 之间也经历过类似的架构之争，最终行业选择了多种共存方案。MCP 可观测性的未来很可能同样不会收敛到单一方案，而是形成"协议层传播 + 运行时层收集 + 应用层收据"的多层协作格局。
