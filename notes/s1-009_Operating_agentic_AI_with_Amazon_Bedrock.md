# Operating agentic AI with Amazon Bedrock AgentCore and Datadog LLM Observability: Lessons from NTT DATA

- **标题**: Operating agentic AI with Amazon Bedrock AgentCore and Datadog LLM Observability: Lessons from NTT DATA
- **作者**: Tohn Furutani（NTT DATA SRE 工程师）

- **来源类型**: 技术博客 / 企业案例分享

## 内容摘要

这篇由 NTT DATA SRE 工程师 Tohn Furutani 撰写的客座博文，系统分享了 NTT DATA 在内部 AI 试验平台 GenAI Tech Hub 中，将 Amazon Bedrock AgentCore 与 Datadog LLM Observability 结合使用的第一手实践与决策洞察。

文章开宗明义地指出，过去一年间生成式 AI 的讨论重心已经发生根本性转移：从单次交互场景（如文本摘要、问答系统、聊天界面）逐步演进为能够基于上下文做出自主决策、规划多步行动、调用外部工具并根据环境变化自适应调整的 agentic AI 系统。这种范式转换意味着 AI 系统从"被动响应"走向"主动行动"，从 Stateless 的简单问答走向 Stateful 的复杂会话，从单步推理走向多步规划与执行。对于长期与企业客户合作的系统集成商而言，这一转变带来的不仅是技术架构的升级，更是运维理念的重塑。

NTT DATA 在 GenAI Tech Hub 的持续验证中很快意识到，agentic AI 最困难的部分并非让概念验证（PoC）在实验环境中跑通，而是让它在生产环境中稳定、可靠、可解释地运行。企业客户对智能体系统的需求已经超越了单纯的准确性与响应速度，他们同时需要安全性保障、系统决策过程的可视性、一致性的行为表现，以及面向持续改进的清晰路径。这四项需求构成了文章后续所有技术决策的核心框架，也定义了 NTT DATA 选择 Bedrock AgentCore 与 Datadog 的组合逻辑。

文章随后回顾了早期 PoC 阶段的典型做法：将应用容器化、暴露 API 端点、逐一实现 IAM 认证和外部工具集成。这种"从零搭建"的方式在单一场景下可以快速验证可行性，但随着验证范围扩大、用例复杂度提升，每次新增智能体工作流都需要重复处理运行环境配置、身份认证、工具连接、会话状态管理等基础问题。团队用于优化智能体推理质量和终端用户体验的时间被严重挤压。更关键的是，企业客户在讨论中反复提出一些根本性问题：智能体在哪运行？工具调用如何受控与审计？不同环境如何隔离？版本如何管理与回滚？这些问题从 scratch 逐一回答显然无法规模化，因此 NTT DATA 决定引入 Amazon Bedrock AgentCore 作为标准化的智能体执行平台。

Bedrock AgentCore 提供了 Runtime 执行环境、Gateway 工具治理层、Memory 与 Identity 集成，以及版本管理等企业级能力。这让团队能够将精力聚焦于智能体逻辑本身的改进，而不必为每个新项目重新搭建基础设施。在解决了执行平台标准化的问题之后，NTT DATA 遇到了第二个更微妙的挑战：传统的应用性能监控（APM）方法难以完整理解智能体 AI 的行为模式。智能体工作流并非简单的请求-响应模式，而是由 LLM 调用、工具调用、条件分支、循环迭代等层层步骤构成的复杂链条。LLM 行为的非确定性意味着相同输入未必产生相同的行为输出。传统 APM 擅长定位语法层面的错误（如崩溃、超时、异常），但智能体 AI 的核心挑战往往从"哪里出错了"转向"决策质量如何"——如果不理解哪一步推理导致了哪个工具选择、什么上下文影响了最终决策，就难以建立可靠的持续改进循环。文章特别指出，在很多情况下系统表面上看起来完全成功（HTTP 200、外部 API 调用成功、无异常堆栈），但用户体验却仍不符合预期。这种 silent failure（静默失败）现象是传统可观测性工具几乎无法捕捉的盲区。

为此，NTT DATA 引入 Datadog LLM Observability，核心目标是在同一上下文中追踪智能体推理（LLM 决策）与智能体行动（工具执行）的完整链条。在实现层面，团队使用了默认支持 OpenTelemetry 的 Strands Agents SDK，该 SDK 可自动发出符合 GenAI 语义约定（v1.37 及更高版本）的属性（如 gen_ai.*）。Datadog 能够自动识别这些属性并接入 LLM Observability，无需自定义格式或专用转换层。这让团队能够将实施重点放在运维 essentials 上，例如应用一致的元数据标签（service.name、env、version）以及跨边界传播 trace 上下文，以便在 Datadog 中完整分析智能体行为。

文章最后总结了 NTT DATA 的三项核心实施重点：第一，对齐遥测数据与 GenAI 语义约定，通过设置 OTEL_SEMCONV_STABILITY_OPT_IN 环境变量实现，并确保跨组件的 trace 上下文一致传播；第二，通过 Datadog AWS 集成将 CloudWatch 日志和指标导入 Datadog，实现平台级信号与应用级信号的关联分析；第三，将评估纳入运维工作流，定义工具选择适当性、工具参数正确性、提示注入韧性、目标完成度等评估标准，并在类生产环境中持续检查。NTT DATA 正将这一组合在 GenAI Tech Hub 中模板化为可复用方案，以加速从 PoC 到生产的过渡，并通过技术文章、与 Datadog 和 AWS 的技术反馈、以及 2026 年 3 月的联合网络研讨会等方式向社区分享经验，目标是帮助企业建立可信赖的、可解释的智能体 AI 生态系统。

## 关键要点

**1. 执行平台标准化是智能体规模化的必要前提**

Bedrock AgentCore 通过 Runtime、Gateway、Memory、Identity 和版本管理，解决了重复搭建基础设施的问题，使团队能专注于智能体逻辑优化。每一次新增工作流都不需要重新回答"在哪运行""如何认证""如何连接工具""如何管理状态"等基础问题。这种标准化对于系统集成商尤其关键，因为他们需要为不同客户快速交付多样化的智能体解决方案。没有这种标准化，每个新项目都会消耗大量时间在基础设施搭建上，严重挤压业务逻辑开发与用户体验优化的空间。AgentCore 的 Gateway 层还提供了统一的工具治理能力，使得外部 API 的调用可以被集中审计、限流和监控，这是生产环境不可或缺的治理机制。

**2. 智能体可观测性的核心挑战是决策质量，而非传统意义上的错误定位**

传统 APM 关注崩溃和延迟，但智能体系统可能返回 HTTP 200 且所有外部 API 调用成功，用户体验却仍不符合预期。这种 silent failure 要求运维人员能够追踪每一步推理、每一次工具选择以及上下文如何影响了最终决策，才能理解"为什么做出了这个决策"。这种从"语法正确"到"语义正确"的转变，是智能体可观测性与传统软件可观测性的本质区别。文章将这种现象称为"visible but not fully understandable AI"，即系统行为可见但决策逻辑难以充分理解，这正是 LLM Observability 要解决的核心痛点。

**3. OpenTelemetry GenAI 语义约定是跨平台可观测性集成的关键技术纽带**

通过 Strands Agents SDK 默认发出的 gen_ai.* 属性，Datadog 可自动识别并接入 LLM Observability，无需额外转换层。这体现了"约定优于配置"的设计理念，大幅降低了多组件集成的工程成本。但前提是各组件都必须遵循同一语义约定版本，否则会出现属性缺失或语义不一致的问题。NTT DATA 选择 v1.37+ 版本，说明他们在实施时已经考虑到了标准稳定性与向前兼容性。这一实践也为其他企业提供了可直接复制的集成路径。

**4. Trace 上下文跨边界传播是构建完整工作流视图的必要条件**

智能体系统通常横跨多个组件（如编排器、AgentCore Gateway、MCP 服务等），必须在各边界间一致传播 trace ID，否则无法获得完整的单条工作流视图。单个组件的遥测数据无法拼凑出全局画面。对于基于 LLM 的系统可观测性，traces 和 attributes 必须被结构化，使得整个工作流呈现为单一连贯的 trace。这对分布式系统的可观测性工程提出了更高要求，特别是在涉及多语言和异构服务的环境下，trace 上下文的丢失会导致严重的观测盲区。

**5. CloudWatch 与 Datadog 的集成实现了平台级与应用级信号的统一关联**

通过 Datadog AWS 集成，Bedrock AgentCore Runtime 和 Gateway 的日志、指标可与应用遥测和 LLM 遥测在同一界面并排分析。这种统一视图避免了在多工具之间来回切换的低效排查方式，使得运维人员能够将平台行为（如 Gateway 的认证延迟、Runtime 的资源使用）与应用行为（如工具选择错误、LLM 响应偏差）快速关联。这种关联能力是缩短平均故障排查时间（MTTR）的关键技术支撑。

**6. 评估必须嵌入日常运维闭环，而不能停留在离线测试阶段**

仅建立可见性不够，还需定义工具选择适当性、工具参数正确性、提示注入韧性、目标完成度等评估维度，并在类生产流程中持续检查。这些标准帮助团队避免将失败视为孤立事件，而是提供了一套可用于定义质量标准并持续改进的一致框架。Datadog 支持在同一平台管理可观测性（APM、日志管理、LLM Observability）和运维（监控、笔记本、云成本管理），从而更容易将评估结果直接嵌入运维工作流，形成"观测-评估-行动"的完整闭环。这种闭环是智能体运维成熟度的重要标志。

**7. Strands Agents SDK 的 OpenTelemetry 原生支持大幅降低了集成工程成本**

NTT DATA 选择 Strands Agents SDK 的一个重要原因是其默认支持 OpenTelemetry，能够自动发出符合 GenAI 语义约定的属性。这意味着团队无需在应用中手动植入追踪代码或编写自定义格式转换器，只需配置环境变量和元数据标签即可接入 Datadog LLM Observability。这种"即插即用"的集成体验对于希望快速验证智能体可观测性价值的企业尤为重要，它降低了尝试新技术的门槛，使得团队可以将有限的工程资源集中在业务逻辑优化上。

**8. 成本归因是可观测性不可忽视的维度**

AgentCore 提供的 token 用量和成本归因指标，使企业能够将智能体运营成本与具体工作流、具体用户甚至具体工具调用关联起来。这种精细化的成本可见性对于控制 AI 支出至关重要，特别是在智能体调用频繁、涉及多个付费 API 的场景中。Datadog 的 Cloud Cost Management 进一步将这些数据纳入统一视图，使成本优化成为运维闭环的自然组成部分，而非事后的财务审计工作。

**9. 安全与韧性是生产运维的基础要求**

文章虽然没有深入展开安全细节，但多次提到企业需要"安全性保障"和"提示注入韧性"。这表明在 NTT DATA 的客户场景中，安全不是可选项而是必选项。AgentCore 的 Gateway 层提供了统一的工具调用审计和访问控制，这是安全治理的基础。而 Datadog 的评估功能中专门包含 prompt injection resilience 检查，说明安全评估正在从传统的渗透测试向持续的自动化评估演进。这一趋势对于综述中"智能体安全与治理"章节具有直接参考价值。

**10. 从 PoC 到生产需要模板化与社区协作的双重推进**

NTT DATA 将 Bedrock AgentCore 与 Datadog 的整合方案在 GenAI Tech Hub 中持续验证，并模板化为标准演示方案。这种"先内部验证、后客户推广"的路径降低了新客户用例的接入成本，同时也确保了方案在真实场景中的可靠性。社区分享（技术文章、网络研讨会、厂商技术反馈）进一步放大了这一价值，使得行业能够共同学习、共同改进，加速整个智能体运维领域的成熟度提升。模板化不仅是技术层面的封装，更是知识管理和组织学习的机制，它将隐性经验转化为显性可复用的资产。

## 与综述的关联

本来源直接对应综述中"生产运维与云平台"方向的 AWS / Amazon Bedrock AgentCore 章节，为综述提供了来自一线系统集成商的真实生产证据。

综述引用该案例说明 AgentCore 发出的遥测数据符合 OpenTelemetry 规范，内置指标覆盖 agent 会话、gateway 与 memory 三个层面。这种多层次指标设计使得运维人员能够从宏观（会话量、错误率）到微观（单次工具调用延迟、token 消耗）多个粒度观察系统健康状态。在综述的技术架构图中，这三个层面的指标被对应到"会话可观测性""网关可观测性""记忆可观测性"三个模块，形成完整的分层观测体系。

CloudWatch Trajectory view 以时序方式展示 agent 决策流，每个 span 包含工具调用详情与执行参数。这种可视化能力对于理解复杂多步工作流的行为模式至关重要，特别是在调试意外行为时，能够直观呈现决策分叉点与上下文变化。综述在讨论"智能体决策可视化"时引用了这一功能作为当前业界最佳实践的代表，并指出这是从"调用链追踪"向"决策链追踪"演进的关键界面创新。

X-Ray Transaction Search 进一步支持跨服务边界的 trace 检索，将 orchestrator、Gateway 与 MCP 服务之间的 trace 上下文连贯传播。这对于诊断涉及多个外部工具的复杂工作流尤为关键，因为传统的单服务日志往往无法呈现完整的调用链条。综述将这一能力与 OpenTelemetry 的跨服务传播机制进行了对比分析，指出虽然 X-Ray 提供了 AWS 生态内的深度集成，但跨云场景仍需要依赖 OpenTelemetry 作为通用标准。

NTT DATA 通过 OTEL_SEMCONV_STABILITY_OPT_IN 对齐 OpenTelemetry GenAI 语义约定 v1.37+，并将 CloudWatch 日志与指标经 AWS 集成管道导入 Datadog，实现跨云统一观测。这一实践证明了语义约定对齐加上平台集成的工程可行性，为其他企业提供了可直接参考的实施路径。综述在"实施建议"章节中引用了这一具体配置作为推荐的集成方式，并强调了 metadata 一致性（service.name、env、version）在避免观测数据碎片化中的关键作用。Metadata 不一致是企业在多团队、多环境部署智能体时最常见的问题之一，它会导致同一服务在不同环境中被识别为不同实体，从而破坏趋势分析和告警关联的准确性。NTT DATA 的经验表明，metadata 治理应在智能体项目启动时就纳入规范，而非等到问题暴露后再补救。

该案例支撑了综述关于"四信号"可观测性模型的核心论点：传统可观测性的 metrics、logs、traces 三元组在智能体场景下需要扩展为包含质量评估（evaluation）的四元组，而四个信号只有同时启用时才能释放完整价值。NTT DATA 的实践表明，缺少评估信号的可观测性体系只能回答"发生了什么"，无法回答"做得对不对"。综述将这一洞察提升为智能体可观测性的核心理论贡献之一。此外，NTT DATA 案例也为综述中"成本可观测性"子主题提供了支持：AgentCore 提供的 token 用量和成本归因指标，使企业能够将智能体运营成本与具体工作流关联，实现精细化的成本管理。Datadog 的 Cloud Cost Management 进一步将这一数据纳入统一视图，使成本优化成为运维闭环的一部分。

同时，该来源与 s1-004（Google ADK 与 AWS AgentCore 对比分析）形成直接对照：Google ADK 深度绑定 Google Cloud 原生服务（Cloud Trace + Cloud Monitoring），而 AWS AgentCore 深度绑定 AWS 服务（CloudWatch + X-Ray），两者都面临供应商锁定与多云对接的挑战，都需要第三方可观测性平台（如 Datadog）作为桥接。NTT DATA 案例为这一对比提供了具体的落地证据，展示了 AWS 生态用户如何通过 Datadog 桥接实现跨平台统一观测。此外，该来源与 s1-008（AWS AgentCore 官方文档）和 s1-010（CloudWatch GenAI Observability 文档）形成互补。

官方文档侧重功能描述和配置指南，而 NTT DATA 案例侧重真实生产环境中的痛点、决策过程和实施细节。这种"官方文档 + 用户案例"的组合为综述提供了既有技术深度又有实践温度的证据基础。

避免了单纯依赖厂商文档可能带来的偏向性。在综述写作中，这种证据组合增强了论证的可信度和可读性，使读者既能理解技术原理，又能感知工程实施的复杂性和现实约束。

## 我的笔记

这篇案例的核心价值在于它并非产品宣传文档，而是来自 NTT DATA 这一大型系统集成商在真实客户场景中的验证经验。作者身份（SRE 工程师）决定了文章视角偏向运维可靠性与工程实践，而非算法创新或模型优化，这与综述关注"agentic AI 如何落地"的基调高度一致。系统集成商的视角尤为珍贵，因为他们需要同时理解平台技术能力、客户业务需求和运维现实约束，这种"三角视角"是单一厂商案例或学术研究难以提供的。在综述写作中，应当充分利用这种来自一线运维人员的声音，以平衡技术理想与工程现实之间的张力。

最值得关注的洞察是文章对"从 PoC 到生产"鸿沟的刻画。文章明确指出，PoC 阶段可以靠容器化加 API 加独立 IAM 集成快速验证，但进入多工作流、多租户、多版本的企业场景后，执行平台的标准化（AgentCore 的 Runtime + Gateway + Memory + Identity + 版本管理）成为必要条件而非可选优化。这与软件工程中"先让代码工作，再让代码正确，最后让代码快速"的经典路径相似，但在智能体场景下，"正确"和"快速"的定义被扩展为"决策可解释"和"行为可复现"。这种扩展意味着智能体系统的成熟度模型需要全新的评估维度，不能简单套用传统软件的 DevOps 成熟度框架。

关于可观测性，文章提出了一个容易被忽视但极其关键的观点：传统 APM 的"错误"概念在智能体场景下部分失效。HTTP 200、API 调用成功、无异常堆栈——这些传统健康指标无法保证用户体验符合预期。这让我想到综述中提到的"语义漂移"问题：智能体的失败往往是语义层面的（选择了错误工具、给出了错误参数、误解了用户意图），而非语法层面的（崩溃、超时、异常）。Datadog LLM Observability 的 trace view 正是为了解决这种"语义级调试"需求，它将 LLM 调用、工具执行、条件分支以可视化方式串联，使运维人员能够追踪"决策链"而非仅仅是"调用链"。这种从"调用链"到"决策链"的视角转换，可能是智能体可观测性领域最重要的认知升级，也是未来可观测性产品竞争的差异化焦点。

OpenTelemetry GenAI 语义约定 v1.37+ 的出现是一个重要的标准化信号。它意味着社区正在形成跨平台的智能体遥测标准，类似于 HTTP 协议对 Web 服务的统一作用。Strands SDK 默认支持这一标准，Datadog 自动识别，这种"约定优于配置"的设计显著降低了集成成本。但也要注意，这一标准目前仍处于早期阶段（v1.37），随着智能体架构的演进（如 MCP 协议的普及、multi-agent 协作模式的成熟），语义约定本身也需要持续迭代。未来可能出现专门针对 multi-agent 协作、long-horizon task、dynamic tool-calling 等场景的扩展语义。综述在引用该标准时，应注明其版本号与发布时间，以便读者了解标准的演进状态与稳定性风险。

评估嵌入运维工作流的做法值得深思。文章列举的四个评估维度（工具选择适当性、工具参数正确性、提示注入韧性、目标完成度）实际上定义了智能体系统的"质量门"。这与传统软件测试中的单元测试、集成测试、端到端测试形成类比，但评估标准更偏向语义正确性而非功能正确性。Datadog 将 APM、日志、LLM Observability 与监控、笔记本、云成本管理统一在同一平台，使得评估结果可以直接驱动运维决策（如触发告警、回滚版本、调整 prompt、优化工具描述）。这种"观测-评估-行动"闭环是智能体运维成熟度的重要标志。在综述中，可以将这一闭环与传统 DevOps 的"监控-告警-响应"闭环进行对比，突出智能体场景下"评估"环节的新增必要性。传统 DevOps 的闭环主要处理已知的故障模式，而智能体运维的闭环还需要处理未知的决策偏差，这要求评估机制具备更强的语义理解能力和动态适应能力。

从社区生态角度看，NTT DATA 将验证经验通过技术文章、与 Datadog 和 AWS 的技术反馈、以及 2026 年 3 月的联合网络研讨会等方式向外部分享，这种"内修外联"的策略对行业具有示范意义。智能体运维作为一个新兴领域，最佳实践尚未固化，企业间的经验分享和社区协作是推动领域成熟的关键动力。NTT DATA 的 GenAI Tech Hub 模式——内部持续验证、模板化复用、外部知识共享——可以作为其他系统集成商或大型企业构建自身智能体运维体系的参考蓝图。综述在讨论"企业级智能体落地路径"时，可以将 GenAI Tech Hub 作为一个典型案例进行分析。值得注意的是，这种"验证-模板化-分享"的三段式路径与开源软件社区中"dogfooding + upstreaming"的文化有异曲同工之妙，它既保证了内部实践的质量，又通过外部反馈循环加速了迭代改进。

最后，文章发布时间（2026 年 4 月）距离当前分析时间很近，这意味着其中描述的技术方案（Bedrock AgentCore + Datadog LLM Observability）代表了当前业界最前沿的生产实践。然而，也正因为其前沿性，这些方案在长期稳定性、成本效益和大规模扩展性方面仍有待更多案例验证。综述在引用该来源时，应当注意其"高时效性、低样本量"的特点，将其作为趋势指示而非成熟定论。建议综述在结论部分指出，虽然 NTT DATA 案例展示了可行的技术路径，但企业级智能体运维的最佳实践仍需要更多来自不同行业、不同规模组织的验证数据来支撑，特别是在金融、医疗、制造等强监管行业的应用场景中，安全性和合规性要求会对当前方案提出额外的挑战。

从更宏观的视角看，NTT DATA 案例揭示了一个正在形成的行业共识：智能体系统的生产化不仅需要执行框架和可观测性工具，还需要一套完整的"运维范式"。

这一范式包括标准化平台、统一遥测语义、决策质量评估和持续改进机制四个核心支柱。它与传统 DevOps 和 MLOps 既有继承关系，又在语义调试、非确定性评估、多步决策追踪等维度上提出了全新要求。

综述可以将 NTT DATA 的实践作为这一新兴范式的早期实例，并呼吁学术界和工业界共同探索智能体运维（AgentOps）的标准化框架和成熟度模型。

这一研究方向不仅具有理论价值，也将直接影响未来智能体平台的产品设计和企业的技术选型决策。谁率先在 AgentOps 领域建立标准和最佳实践，谁就可能在企业级智能体市场中获得显著的竞争优势。

另一个值得记录的细节是文章中提到的"hands-on implementation details"（实践 implementation 细节）。

NTT DATA 在日本市场发表的技术文章包含了大量的实操细节，这种区域性知识共享对于非英语市场的企业尤为重要。

综述在讨论全球智能体运维实践时，应注意到当前英文技术博客 dominance 的现状，并主动纳入来自日本、欧洲等非英语市场的案例，以避免视角偏差。

NTT DATA 作为日本最大的系统集成商，其经验对于亚洲企业具有特殊的参考价值，因为这些企业往往面临类似的组织架构、合规要求和文化环境。

GenAI Tech Hub 作为 NTT DATA 的内部验证平台，其设计思路也值得借鉴。

它不是简单地将多个 PoC 项目堆放在一起，而是建立了一套包含环境隔离、版本管理、评估基准和知识沉淀的标准化流程。

这种"平台化验证"的思维，比传统的"项目制验证"更能积累可复用的组织资产。

对于计划建立类似验证平台的企业而言，NTT DATA 的经验提供了一个经过实践检验的参考模板。
