## 原文摘要翻译

在大语言模型多智能体系统中，故障归因——即识别导致任务失败的智能体和具体步骤——为系统调试提供了关键线索。但该领域尚未得到充分探索，且高度依赖人工劳动。本文提出并形式化了一个新研究方向：面向 LLM 多智能体系统的自动化故障归因。为支持该研究，作者构建了 Who-When 数据集，包含来自 127 个 LLM 多智能体系统的详细失败日志，并附有细粒度标注，将失败与特定智能体及决定性错误步骤关联起来。基于 Who-When，作者开发并评估了三种自动化故障归因方法，总结了各自的优势与局限。最优方法在识别故障责任智能体上的准确率为 53.5%，但在定位故障步骤上的准确率仅为 14.2%，部分方法甚至低于随机水平。即便使用 OpenAI-o1 和 DeepSeek-R1 等最先进的推理模型，也未能达到实用水平。这些结果凸显了该任务的复杂性以及进一步研究的必要性。

## 创新点

1. **提出新研究问题**：首次将 LLM 多智能体系统中的自动化故障归因形式化为独立研究问题，填补了评估与系统改进之间的鸿沟。
2. **构建专用数据集**：发布 Who-When 基准数据集，涵盖 127 个多智能体系统（含算法生成与人工精心设计两类）的失败日志，并提供细粒度的人工标注（故障责任智能体、决定性错误步骤、自然语言解释）。
3. **系统评估三种基线方法**：设计了全局法、逐步法和二分法三种故障归因方法，并在多种模型、上下文长度、有无标签等条件下进行全面对比实验。
4. **揭示任务内在难度**：通过实验表明，即使 SOTA 推理模型也难以实现实用的故障归因精度，为该领域后续研究指明了方向。

## 一句话总结

本文首次形式化并系统研究了 LLM 多智能体系统的自动化故障归因问题，构建了 Who-When 数据集，发现即便最优方法在定位故障步骤上的准确率也远低于实用门槛，揭示了该任务的根本性困难。

## 研究问题

随着 LLM 多智能体系统（如 AutoGen、CAMEL、MetaGPT 等）在编码、科学发现、复杂现实世界问题求解等领域展现出巨大潜力，系统开发通常遵循一个迭代循环：在基准上评估系统表现，然后手动进行故障归因，最后针对性地改进系统。然而，故障归因——即识别直接导致任务失败的系统组件——目前几乎完全依赖人工完成。开发人员需要分析复杂的历史执行日志，理解每个智能体的行为逻辑，并判断哪些动作是正确的、哪些动作误导了整个求解过程。随着系统复杂度增加，组件数量增多，手动故障归因变得愈发困难且耗时。

现有研究主要通过构建更细粒度的基准（如 DevAI）来辅助故障归因，但这本质上仍只是提供更多参考指标，未能从根本上解决"评估结果如何映射到具体故障组件"的问题。因此，本文提出一个核心问题：**能否利用 LLM 的能力自动识别多智能体系统中导致任务失败的智能体（哪个）和具体步骤（何时）？**

## 数据与任务定义

### 问题形式化

本文采用回合制 LLM 多智能体协议。系统包含 $N$ 个智能体，

$$\mathcal{M} = \big\langle \mathcal{N}, S, A, P, \phi \big\rangle$$

其中 $N$ 为智能体数量。各符号含义如下：

- $S$ 为状态集合
- $A$ 为全局动作集合
- $\phi(t)$ 表示时间步 $t$ 的执行智能体
- $P(\cdot)$ 为状态转移概率

$$P(s_{t+1} \mid s_t, a_t, \phi(t))$$

对于一条失败轨迹 $\tau$（即 $Z(\tau) = 1$），若将智能体 $i$ 在时间 $t$ 的动作进行修正，

$$a_t \rightarrow \tilde{a}_t$$

轨迹变为成功，则称该二元组为一个决定性错误。定义决定性错误指示函数：

$$\Delta_{i,t}(\tau) = \begin{cases} 1, & \text{if } Z(\tau) = 1 \text{ and } Z(\tau^{(i,t)}) = 0 \\ 0, & \text{otherwise} \end{cases}$$

实际中一条轨迹可能包含多个决定性错误，本文选取时间最早的一个作为失败主因，即：

$$(i^*, t^*) = \arg\min_{(i,t) \in \mathcal{C}(\tau)} t, \quad \mathcal{C}(\tau) = \{(i,t) \mid \Delta_{i,t}(\tau) = 1\}$$

其中 $i^*$ 为故障责任智能体，$t^*$ 为决定性错误步骤。研究目标即为自动识别该二元组。

### Who-When 数据集

为支持自动化故障归因研究，本文构建了 Who-When 数据集，包含以下要素：

- **查询（Query）**：来自 GAIA 或 AssistantBench 的真实世界问题。
- **失败日志（Failure log）**：特定系统未能解决该查询时的完整对话记录。
- **系统信息（System info）**：对于算法生成系统，包括系统提示词、可用工具、智能体名称等。
- **标注（Annotations）**：故障责任智能体、错误发生的具体步骤、以及失败原因的自然语言解释。

数据集涵盖两类系统：

1. **算法生成系统**：使用 AG2 库中的 CaptainAgent 算法自动为每个查询生成多智能体配置，共 126 个系统。所有智能体及 CaptainAgent 本身均基于 GPT-4o（2024-08-01-preview 版本）。
2. **人工精心设计系统**：采用 Magnetic-One 通用智能体系统，包含 5 个专业智能体（如 WebSurfer、FileSurfer、Orchestrator 等）。

标注过程由三位 AI 智能体领域专家分三轮完成：第一轮各自独立标注并标记不确定项；第二轮就不确定项进行讨论并达成共识；第三轮进行交叉验证，确保标注质量。

![Figure 2 标注过程统计分析](p-022_Which_Agent_Causes_Task_Failures_and_Whe/images/fig2a_time_cost.png)

三位标注者分别花费了 30.9、30.2 和 23.2 小时完成标注，不确定标注比例在 15% 到 30% 之间，且存在显著的分歧率。这表明即便是领域专家，手动故障归因也是极其耗时且困难的任务。

### 评估指标

- **智能体准确率（Agent-Level Accuracy）**：正确预测故障责任智能体的比例。
- **步骤准确率（Step-Level Accuracy）**：正确定位决定性错误步骤的比例。
- **带容差的步骤准确率**：允许一定容差范围的步骤定位准确率。

## 方法主线

本文提出并评估了三种自动化故障归因方法，分别对应不同的信息接收策略。所有方法均基于以 LLM 为裁判的范式，由判断模型对失败日志进行分析。

### 机制流程

三种方法的执行流程可概括如下：

1. **输入准备**：将用户查询与失败日志送入判断 LLM。根据实验设置，可选择性地提供最终答案作为参考信号。
2. **日志分析**：判断 LLM 对失败日志进行推理分析，提取关键错误线索。
   - 全局法：一次性接收完整日志，输出故障责任智能体及错误步骤。
   - 逐步法：逐步接收日志，每一步判断当前步是否出错，若发现错误则终止并返回。
   - 二分法：先判断错误发生在日志的上半部分还是下半部分，然后迭代缩小范围直至定位单一步骤。
3. **结果生成**：返回预测的二元组，即故障责任智能体及其错误步骤。
4. **性能评估**：将预测结果与 Who-When 的人工标注进行对齐，计算各项准确率指标。

### 实验设置

实验主要在两种场景下进行：

- **有标签（With Ground Truth）**：提供查询的最终正确答案，模拟开发阶段使用带标签数据进行调试的场景。
- **无标签（Without Ground Truth）**：不提供最终答案，模拟部署后对真实用户查询进行自反思的场景。

主实验使用 GPT-4o 作为判断模型，同时测试了多款开源与闭源模型（如 Llama 系列、Qwen 系列、GPT 系列），以及两款先进推理模型（OpenAI-o1 与 DeepSeek-R1）。

| Hand Crafted | With GT | 12.00 | 4.16 | 55.17 | 5.26 | 34.48 | 7.02 | 51.72 | 6.90 |

核心发现如下：

- **智能体准确率依赖大上下文**：全局法在智能体准确率上显著优于其他两种方法（最高达 55.17%），因为它能一次性获取完整对话上下文，更准确地判断哪个智能体的行为导致了失败。二分法次之，逐步法最差。
- **步骤准确率依赖细粒度处理**：逐步法在步骤准确率上表现最好（最高 25.51%），因为它逐步处理上下文，能更聚焦地定位错误步骤。全局法在步骤准确率上表现最差，部分设置甚至低于随机基线，这体现了"大海捞针"问题——LLM 难以从长上下文中精确定位特定信息。
- **标签信息的作用**：三种方法在有标签场景下的表现均优于无标签场景，说明最终答案作为参考信号能有效辅助故障归因。

## 深度分析

### 任务难度的根源

本文通过人工标注实验充分证明了故障归因的内在难度。三位专家在标注过程中不仅需要阅读冗长复杂的日志，还需要理解每个智能体的求解逻辑，判断某个动作是否正确、是否误导了整个过程。例如，当智能体使用浏览器搜集信息时，标注者必须检查浏览器历史并访问每个网站，才能判断失败是因为网站信息不可用，还是智能体未能正确检索。此外，多个智能体可能同时犯错，需要从中筛选出最严重的那个——这种严重程度的判断有时甚至带有主观性。

### 方法设计的权衡

三种方法体现了上下文范围与定位精度之间的根本权衡：

- **全局法**：最大化上下文范围，有利于智能体级别判断，但面临"大海捞针"问题，步骤级别极差。
- **逐步法**：最小化单次处理的上下文，有利于步骤级别定位，但由于决策时信息不完整，智能体级别表现较差。
- **二分法**：在两者间折中，但并未在任何单项指标上取得绝对优势。

### 统计有效性与实例精度

本文的一个重要发现是：虽然实例级故障归因准确率有限，但在聚合统计层面，方法能有效识别出最容易出错的智能体类型。这对于实际系统优化非常有价值——开发者可以优先改进最薄弱的环节，而不必精确到每一次失败的具体步骤。

### 推理模型的局限

强推理模型（o1、R1）未能在该任务上取得突破，说明故障归因不仅是推理能力的问题，更可能涉及：

1. **对多智能体交互动态的理解**：需要理解智能体之间的协作与冲突模式。
2. **对工具使用效果的判断**：需要评估外部工具（如浏览器、代码执行器）返回结果的质量。
3. **对错误传播路径的追踪**：需要判断一个早期错误如何在后续步骤中放大或收敛。

这些能力可能超越了当前 LLM 通过纯文本推理所能达到的范畴。

## 局限

1. **准确率远未达到实用门槛**：最优方法在步骤级别上的准确率最高仅约 25%（算法生成系统，有标签），在手工设计系统上甚至不足 15%，距离实际可用的自动化诊断仍有巨大差距。
2. **数据集规模与覆盖范围**：该数据集主要基于 GAIA 与 AssistantBench 的验证集，且手工设计系统仅包含 Magnetic-One，可能不足以覆盖所有类型的多智能体协作模式。
3. **标注主观性**：尽管经过多轮讨论，但故障严重程度的判断仍可能存在主观偏差，特别是对于多个智能体均犯错的实例。
4. **成本考量**：混合方法虽性能最优，但 token 消耗显著增加，在实际部署中需要权衡精度与成本。
5. **潜在安全风险**：将故障归因能力集成到可修改外部环境（如执行代码）的系统中，可能带来不可预期的后果。

## 我的笔记

本文的工作为 agentic trace analysis 领域提供了一个重要的新维度——从"评估系统表现"到"定位故障根源"。与 AgentRx 等专注于轨迹可视化与交互式诊断的工具不同，本文试图完全自动化这一过程。从结果来看，自动化故障归因的难度被严重低估了。

一个值得深入思考的问题是：当前方法将故障归因视为一个纯判别任务，但本质上它更接近因果推断。具体而言，需要判断：若某智能体在特定步骤采取不同动作，整个系统是否会成功？这提示未来工作可以探索反事实推理或干预分析的方法。

此外，Who-When 数据集作为该领域首个专用基准，具有很高的复用价值。后续研究可以考虑：（1）引入过程奖励模型的思路，训练专门用于多智能体故障归因的判别器；（2）结合程序分析技术，对工具调用和代码执行进行白盒追踪；（3）探索人机协作的半自动故障归因流程，在自动化与准确性之间取得平衡。
