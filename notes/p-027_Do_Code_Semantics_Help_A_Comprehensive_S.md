# Do Code Semantics Help? A Comprehensive Study on Execution Trace-Based Information for Code Large Language Models

## 核心信息

- **标题**: Do Code Semantics Help? A Comprehensive Study on Execution Trace-Based Information for Code Large Language Models
- **作者**: Jian Wang, Xiaofei Xie, Qiang Hu, Shangqing Liu, Yi Li
- **年份**: 2025
- **来源**: Findings of the Association for Computational Linguistics: EMNLP 2025
- **DOI**: 10.18653/v1/2025.findings-emnlp.548

## 原文摘要翻译

代码大语言模型以其卓越的能力开启了编程的新纪元。然而，近期研究揭示了它们在推理程序运行时行为和理解程序实际功能方面存在严重局限，这对其后训练阶段和实际部署构成了重大挑战。具体而言，这类模型面临两个核心问题：（1）缺乏对程序执行行为进行推理的能力，难以解释程序在运行时的实际行为；（2）现有方法中语义信息（如执行轨迹）的表示方式不一致且碎片化，阻碍了模型的泛化和有效推理。为应对这些挑战，本文引入了一个通用框架，支持将语义信息整合到与代码任务相关的提示中，并据此开展了一项全面研究，以探索语义信息在增强代码大模型推理能力方面的作用。具体而言，研究重点是基于轨迹的语义信息在提升监督微调和测试时推理扩展方面的效用。实验结果与先前工作截然相反，表明语义信息对监督微调与测试时计算扩展的帮助非常有限。

## 创新点

1. **首个支持多种语义表示的统一框架**。本文提出了一个通用框架，能够生成和适配多种代码执行轨迹表示（Scratchpad、NExT、SemCoder、CodeExecutor 以及本文设计的 Concise），并支持将这些表示用于后训练、一次性推理和测试时计算扩展。此前工作通常只采用单一表示，缺乏系统性的对比工具。

2. **高质量多表示数据集的开源构建**。基于 APPs 数据集，作者通过代码相似度匹配、测试用例筛选、执行轨迹提取等步骤，构建了一个包含 27.8K 条样本的程序修复数据集，并集成了五种执行轨迹表示、单元测试和程序描述。该数据集填补了现有数据集在表示多样性和可复现性上的空白。

3. **对 SFT 与测试时扩展的全面实证研究**。本文首次在同一框架下系统评估了执行轨迹语义信息对 Code LLMs 监督微调和推理阶段的影响，覆盖了程序修复、代码合成和代码推理三类任务，实验涉及 3 个开源基座模型和 7 个推理模型。

4. **与先前结论相悖的负面发现**。与 NExT、SemCoder 等先前工作声称轨迹信息显著提升模型能力不同，本文的实验一致表明：将现有语义信息整合到输入提示中，对 SFT 和测试时扩展的增益非常有限，甚至在多数情况下无显著提升。

## 一句话总结

本文构建了一个支持多种执行轨迹表示的通用框架，通过大规模实验发现现有代码语义信息对 Code LLMs 的监督微调和测试时扩展几乎没有显著帮助，为未来设计更适配模型处理方式的语义表示提供了新的基线。

## 研究问题

本文围绕以下两个核心子问题展开：

1. **基于语义信息的监督微调**：如何在程序修复任务的微调数据中最优地编码执行轨迹推理信号，使得代码大模型能够习得更好的代码生成能力？给定微调数据集，其中输入为缺陷代码片段与执行轨迹语义信息，输出为修复后的代码，核心问题是什么样的轨迹表示最有利于模型学习。

2. **基于语义信息的测试时推理扩展**：在固定的推理计算预算 $N$ 下，当提示被不同轨迹表示增强时，Code LLM 能在多大程度上提升性能？形式化地，对于查询 $q$，最终输出 $y$ 从以下分布中采样：

$$
y \sim \operatorname{Target}\bigl(\theta,\; q,\; N,\; \mathrm{Verify}\bigr)
$$

其中 $\operatorname{Target}$ 是一个测试时扩展框架，在预算 $N$ 内迭代地进行候选生成、验证、剪枝和精化。最优策略 $\theta_q^*(N)$ 定义为最大化生成正确答案期望概率的策略：

$$
\theta_{q}^*(N) = \argmax_\theta \Bigl( \mathbb{E}_{y \sim \operatorname{Target}(\theta, N, q, \mathrm{Verify})} \bigl[\mathds{1}_{y = y^*(q)}\bigr] \Bigr)
$$

## 数据与任务定义

### 任务类型

本文涉及三类代码相关任务：

- **程序修复**：给定缺陷代码片段、失败测试用例和执行轨迹，模型输出修复后的代码。评估基准包括 HumanEvalPack（HE-R）和本文从 EvalPlus MBPP 构造的 MBPP-R。
- **代码合成**：根据自然语言描述生成可执行代码。评估基准包括 HumanEval、MBPP、LiveCodeBench（easy 子集）和 BigCodeBench（full）。
- **代码推理（Code Reasoning）**：评估模型对代码运行时行为的理解能力，包括输入预测（CRUXEval-I）和输出预测（CRUXEval-O）。

## 方法主线

### 机制流程

本文框架的核心执行链可分为以下四个步骤：

1. **输入：执行行为数据构建**
   - 从 APPs 数据集中通过 UniXcoder 嵌入相似度匹配（阈值 0.8）收集缺陷代码与修复代码对 $(B_X, P_Y)$。
   - 执行两份代码并保留在缺陷代码上失败、在修复代码上通过的测试用例。
   - 利用 Trace-Tracker 收集运行时信息，并通过转换器生成多种轨迹表示。
   - **输出去向**：生成的训练样本 $\{(b, r, a)\}$ 进入微调数据池。

2. **关键中间变换：轨迹适配器**
   - 轨迹适配器将原始执行轨迹转换为五种表示之一：Scratchpad（行内注释式）、NExT（变量变更行内注释）、SemCoder（自然语言逐行描述）、CodeExecutor（独立的状态变更记录）、Concise（CodeExecutor 的简化变体，忽略未变更变量）。
   - **输出去向**：格式化后的轨迹文本 $r$ 被嵌入到输入提示 $x=(b, r)$ 中。

3. **输出去向一：监督微调（SFT）**
   - 模型以缺陷代码 $b$ 和轨迹 $r$ 为输入，学习生成修复代码 $a$。
   - 支持全参数微调和参数高效微调（LoRA8、LoRA64）。
   - **输出**：微调后的模型权重，用于下游任务评估。

4. **输出去向二：测试时推理扩展**
   - **Sequential Scaling**：模型每轮生成 $N$ 个候选程序，外部检查器（Python 解释器）验证每个候选。若全部失败，将轨迹诊断信息追加到提示中，进入下一轮自调试循环，最多 $R_{\max}$ 轮。
   - **Parallel Scaling**：并行生成 $N$ 个候选，由 LLM-as-a-Judge（phi-4）根据执行输出和轨迹表示进行排序投票，选择最优解。
   - **输出**：最终通过所有隐藏测试用例的代码。

### 轨迹表示方法

本文系统评估了五种执行轨迹表示：

- **NExT**：将执行轨迹以行内注释形式嵌入代码，每行代码后追加该行中变量的变更情况。
- **SemCoder**：使用自然语言逐行描述代码执行状态，包括执行状态、变量变更和输入输出关系。
- **CodeExecutor**：与 NExT 类似记录每行变量状态变更，但将轨迹与代码分离展示，强调代码与执行状态的清晰区分。
- **Concise（本文提出）**：CodeExecutor 的简化变体，仅记录每行中值发生变化的变量，忽略未变更变量，从而降低表示冗余。

## 深度分析

### 为什么轨迹信息对微调帮助有限？

本文的负面发现与 NExT、SemCoder 等先前工作的积极结论形成了鲜明反差。一个可能的解释是：**现有轨迹表示与人类开发者的直觉一致，但与代码大模型的信息处理方式并不对齐**。这类模型以自回归方式处理长序列，而 NExT 的行内注释式表示会严重破坏代码的词元结构；SemCoder 的自然语言描述虽然人类可读，但引入了跨域信息（自然语言 + 代码），增加了学习难度。Concise 通过忽略未变更变量来降低冗余，在测试时扩展中表现相对稳健，这一结果暗示**表示的简洁性可能比信息完备性更重要**。

### 模型架构对训练策略的敏感性

全参数微调 vs LoRA 的有效性高度依赖模型架构：DeepSeek-Coder 适合全参数微调，而 LLaMA 和 Gemma2 更适合 LoRA。这提醒我们，在复现或扩展本文工作时，不能简单地套用单一训练配置。21.6 个百分点的性能差距说明，**训练策略的选择本身就是比轨迹表示更显著的变量**。

### 测试时扩展的启示

测试时扩展（尤其是顺序扩展）带来的提升远超过语义信息表示带来的差异。这一结果支持了近期推理时计算研究的核心观点：对于代码生成这类需要反复试错和验证的任务，**计算预算的重新分配（从训练转向推理）可能比训练信号的精心设计更有效**。顺序扩展本质上是将人类调试的“编码—运行—修复”循环自动化，其成功关键在于外部检查器提供的即时反馈，而非轨迹表示的精细程度。

### 与相关工作的关系

本文与 NExT（Ni 等, 2024）和 SemCoder（Ding 等, 2024）构成了直接对话。后两者分别通过执行轨迹微调取得了显著增益，但本文在统一框架下的系统性对比表明，这些增益可能源于特定模型、特定任务或特定训练配置的耦合，而非轨迹信息本身的普适价值。本文的发现更贴近 Chen 等（2024）关于代码大模型在运行时行为推理方面存在根本局限的观察。

## 局限

1. **编程语言覆盖有限**：当前框架仅支持 Python，未覆盖 Java、C++ 等其他主流语言。虽然作者认为发现具有跨语言启示，但缺乏实证验证。

2. **模型规模受限**：由于计算资源限制，实验主要在约 7B 参数规模的模型上进行，更大规模模型（如 30B+）上的结论可能不同。

3. **评估指标单一**：本文主要使用 Pass@1 功能性指标，未评估非功能性属性（运行时间复杂度、内存占用、安全性等）。正确性提升不等于生成代码更安全或更高效。

4. **对抗鲁棒性未验证**：未测试恶意构造的轨迹是否能诱导模型生成脆弱或有害代码，在安全敏感场景中这一风险需要额外关注。

5. **轨迹选择启发式**：当前轨迹注入采用固定策略（最短失败路径等），未探索基于学习或问题难度的自适应轨迹选择机制。

## 我的笔记

### 复现思路

本文开源了完整代码和数据集（https://github.com/tracewise-probing/tracewise_probing），复现门槛较低。关键复现点包括：

1. **数据构造流程**：基于 APPs 的相似度匹配、测试用例筛选和 Trace-Tracker 轨迹提取。需要注意 APPs 数据集的授权协议和去污染流程。
2. **轨迹适配器实现**：五种轨迹表示的转换器逻辑。Concise 作为 CodeExecutor 的简化版，实现上只需在状态变更记录中过滤未变更变量。
3. **训练配置**：DeepSpeed ZeRO-3 全参数微调（学习率 2e-5，批次大小 8，bfloat16 精度，最大序列长度 2048，训练 2 轮）与 LoRA（秩 8/64）的对比需要精确复现。
4. **顺序扩展推理循环**：公开测试上的验证、轨迹诊断信息追加到提示模板的逻辑、最大轮数 $R_{\max}$ 和温度系数的设置。

### 与 Agent 轨迹研究的联系

本文标题中的执行轨迹与智能体轨迹研究有概念交集：程序执行轨迹可以视为代码智能体的一种内部状态记录。但本文的发现——细粒度状态信息对大语言模型帮助有限——对智能体轨迹设计有警示意义。如果代码级别的执行状态都难以被模型有效利用，那么更高级别的智能体行动轨迹（如工具调用序列、环境观测）可能需要更精心的抽象和压缩，而非直接原始转储。

### 后续研究方向

1. **架构层面的语义融合**：本文仅在输入提示层面整合轨迹，未来可探索在模型架构层面（如专用编码器、交叉注意力）融合执行语义。
2. **自适应表示学习**：让模型自身决定需要哪些执行信息、以何种粒度呈现，而非使用固定的启发式表示。
3. **多语言验证**：将框架扩展至 Java/C++，验证结论的跨语言稳定性。
4. **与 RL/Process Reward 结合**：本文的测试时扩展依赖简单的验证器反馈，结合过程奖励模型（PRM）可能进一步提升扩展效率。
