# View-oriented Conversation Compiler for Agent Trace Analysis

- **标题**: View-oriented Conversation Compiler for Agent Trace Analysis
- **作者**: Lvmin Zhang, Maneesh Agrawala
- **机构**: Stanford University
- **年份**: 2026
- **来源**: ArXiv.org
- **DOI**: 10.48550/arXiv.2603.29678
- **arXiv ID**: 2603.29678
- **PDF**: `[p-015]-vcc-view-oriented-conversation-compiler-for-agent-trace-anal.pdf`
- **证据质量**: high

## 原文摘要翻译

智能体轨迹在智能系统与上下文工程中承载着日益增长的解析价值，然而大多数现有工作将对话格式视为微不足道的实现细节。
现代智能体会话却包含深层结构化内容——嵌套工具调用与结果、思维链推理块、子智能体调用、上下文窗口压缩边界以及工具注入的系统指令——其复杂度远超简单的用户与助手交换。
将此类轨迹以纯文本、JSON、YAML 或 grep 形式馈送给反射器或其他分析机制，会实质性地降低分析质量。
本文提出 VCC（View-oriented Conversation Compiler）。
该编译器遵循完整编译器流水线，将原始智能体 JSONL 日志转化为一系列结构化视图：
全视图（lossless transcript，作为规范行号坐标系）、UI 视图（重建用户实际感知到的交互过程）以及自适应视图（由相关性谓词支配的结构保留投影）。
在 AppWorld 上的上下文工程实验中，仅将反射器的输入格式从原始 JSONL 替换为 VCC 编译后的视图，即可在全部三种模型配置下获得更高的通过率，同时将反射器 token 消耗削减一半到三分之二，并生成更简洁的学习记忆。
这些结果表明，消息格式是上下文工程的基础设施，而非偶然的实现选择。

## 创新点

1. **编译器范式引入轨迹分析**：将编译器理论（lex、parse、IR、lower、emit）系统性地应用于智能体轨迹处理，提出完整的四阶段编译流水线。
2. **三种结构化视图与行号一致性**：设计全视图、UI 视图与自适应视图，所有视图共享同一套行号坐标系，指针解析由构造保证而非事后验证。
3. **投影 versus 抽象的明确区分**：VCC 仅执行投影（选择并标注现有内容），不生成摘要、嵌入或知识图谱节点等抽象产物，保证无损可还原。
4. **上下文工程层面的格式验证**：通过控制实验表明，仅改变反射器的输入格式即可同时提升任务通过率、降低 token 消耗并压缩记忆文件体积。

## 一句话总结

VCC 是一个面向智能体轨迹的视图编译器，通过将原始 JSONL 日志编译为结构化视图，在不改变学习算法的前提下显著提升了反射器的分析效果与效率。

## 研究问题

上下文学习研究长期关注高层架构选择：生成器、反射器与策展人的角色分解，层次化记忆的组织与检索，工具使用，以及从智能体轨迹中提取经验并泛化。
然而，消息本身的格式——即智能体轨迹如何被表示并呈现给大语言模型进行分析——在很大程度上被当作微不足道的实现细节，纯文本、JSON、YAML 与 grep 成为默认惯例。
对于简单的用户与助手交换，这些惯例或许足够。
但当代轨迹具有完全不同的复杂度：像 Claude Code 或 Codex 这样的编码智能体单次会话即可产生超过一万行的 JSONL 日志，其中工具调用参数、工具结果体、思维链块、子智能体调用、上下文压缩边界、base64 编码媒体以及工具注入的系统指令相互交错。
工具结果体单独即可占据 token 预算的大部分，将实际决策点淹没在大量文件内容与终端输出之下。
已有实证工作支持格式并非中性的观点：相同内容以不同格式呈现可导致性能差异高达 40 个百分点；智能体轨迹中 40% 到 60% 的 token 被识别为无用、冗余或过期信息，其移除并不会降低任务性能。
由此可见，瓶颈并非缺乏有效的学习方法，而是输入质量本身。

## 数据与任务定义

实验在 AppWorld 基准上进行，该基准提供真实的 API 交互环境以评估自主智能体。
数据集包含三个划分：train（90 个任务，用于学习）、test_normal（168 个任务）与 test_challenge（416 个任务），覆盖三个难度级别（d1 简单、d2 中等、d3 困难）。
研究采用基于轮次的上下文工程协议：
生成器执行训练集中的每个任务，随后由 AppWorld 工具链评估；反射器读取评估结果以判断成败，分析生成器的执行轨迹，并编辑共享记忆文件 MEMORY.md 以添加规则。
失败任务对应 `[CRITICAL]` 规则（指出根因与预防措施），成功任务对应风险提示规则。
整个过程运行两轮（90 个任务 × 2 轮 = 180 次分析），迭代精炼规则集。
训练时并行运行 32 个任务，每批结束后通过基于大语言模型的差异合并步骤整合各工作者的记忆编辑。
学习得到的 MEMORY.md 随后在测试划分上执行单次评估，不再更新记忆。
实验的唯一变量是反射器分析阶段所接收的轨迹格式：VCC 条件使用预编译的结构化视图，JSON 条件直接读取原始 JSONL。
测试了三种模型配对：最强配置、中等配置与较弱配置。
最强配置使用 Opus 同时担任生成器与反射器。
中等配置使用 Sonnet 兼任两者。
较弱配置使用 Haiku 作为生成器、Sonnet 作为反射器。
这是由于 Haiku 自身不具备反射器能力。

## 方法主线

VCC 将原始智能体 JSONL 日志编译为结构化视图。
其核心并非单层格式转换（如 JSON 转 Markdown），而是执行完整的编译器流水线，并包含大量轨迹特定的变换：
- 工具调用参数从转义 JSON 编译为可读 YAML，使用块标量处理多行值。
- 工具注入的 XML 标记被识别并移除，完全由此类标记构成的用户轮次被隐藏。
- 因上下文压缩而分割在两条 JSONL 记录中的助手消息片段被重新组装为单一节。
- 不含对话内容的记录类型在词法阶段即被过滤。
- base64 编码的内联图片被解码为独立文件并替换为占位符。

### 全视图

全视图是恒等投影。
设

$$\mathcal{I} = (n_1, n_2, \ldots, n_N)$$

为解析和行号分配后的 IR 节点序列，则全视图对每个节点逐字输出：

$$V_{\text{full}} = \bigoplus_{i=1}^{N} \text{emit}(n_i)$$

其中 $\bigoplus$ 表示带结构空白行插入的拼接运算。
该渲染是双射的：可从 $V_{\text{full}}$ 重建源文件（除序列化空格外）。
全视图作为规范参考面，所有其他视图的行号均解析至此。

### UI 视图

UI 视图重建人类用户在原始交互会话中的感知体验。
在基于终端的编码智能体中，用户极少看到原始工具输入、工具输出或思维链推理；用户观察到的是助手文本、单行工具调用摘要以及对话轮次结构。
UI 视图对 IR 应用四项变换：
- 隐藏内部内容（工具调用体、结果体、思维链块、系统消息）。
- 将每个工具调用折叠为单行摘要，并标注指向全视图的行范围指针。
- 合并相邻的助手回复段（中间无用户轮次插入），以反映用户对连续响应的感知。
- 过滤掉工具注入的标记噪音（IDE 选择上下文、系统提醒、钩子输出等）。

### 自适应视图

自适应视图通过相关性谓词对会话做结构保留的投影。
给定谓词如下：
$\rho: \text{line} \to \{0, 1\}$
该谓词保留会话的结构骨架（节标题、轮次边界、块定界符），同时仅输出匹配的内容块。

$$V_{\text{adapt}}(b) = \begin{cases} \text{match\_lines}(b, \rho) & \text{if } \exists\, l \in [s,e]: \rho(l) = 1 \\ \varnothing & \text{otherwise} \end{cases}$$

其中 $b$ 在全视图中跨越行 $[s, e]$。
match_lines 返回匹配行并标注其在全视图中的行号。
每行前面加上块级指针。
无匹配内容的节被完全省略。
谓词 $\rho$ 是可替换参数：当前实现为正则匹配，但框架也支持 BM25 评分、嵌入相似度或基于大语言模型的相关性判断，均无需改动结构投影逻辑。
自适应视图支持两种输出模态：
- 文档导向模态保留对话的时序结构，按节组织并省略非匹配块。
- 索引导向模态丢弃结构支架，输出扁平的匹配块序列，每块标注语义角色。
底层数据完全相同，仅迭代顺序不同——按节行优先 versus 按匹配列优先。

### 与多层记忆和 grep 的区别

VCC 的多个视图不是 MemGPT、A-MEM 或 RAPTOR 意义上的多级记忆层次。
这些系统维护静态的预计算产物（摘要、嵌入、知识图谱节点），通过持久边连接；查询意味着遍历预存拓扑，层级独立于具体查询而存在。
VCC 的视图则是动态、查询时的投影，基于单一扁平 IR 重新计算，不存在预计算摘要、持久图或存储的嵌入。
更根本的是，多级记忆执行抽象（生成源中不存在的新内容，引入有损压缩，其忠实度需经验评估），而 VCC 执行投影（选择并排列源中已有内容，标注精确坐标以恢复未选中的余下部分）。

VCC 的自适应视图也与扁平 grep 有本质区别。
在对话轨迹上做扁平 grep 返回一组孤立的匹配行，每行被剥离其会话身份：读者无法判断匹配出现在用户话语、助手思维链、工具调用输入还是工具结果中。
标为工具调用编辑的匹配（意图）与标为工具结果读取的匹配（观察）承载不同含义，而 grep 将意图、观察与推理坍缩为统一的行袋。
自适应视图保留块级角色注释与范围指针，能够表达扁平搜索无法表达的语义区分。
此外，块级指针携带范围信息而非仅点位置：第 50 行的匹配若位于跨越 46–56 行的块内，则告知读者周围存在十行可解引用的上下文。

### 机制流程

VCC 编译器将原始 JSONL 日志转化为结构化视图的机制可分为四个阶段：

1. **词法分析**：输入原始 JSONL 记录流，提取带类型的词法单元，过滤掉不含对话内容的记录类型。
2. **语法解析**：将词法单元编码为带语义类型的中间表示 IR，为每个节点标注其内容行与结构元数据（节索引、块索引）。
3. **行号分配**：对 IR 节点执行单次顺序遍历，生成单调递增的行号坐标；该坐标系作为所有视图共享的规范参考面，一旦分配不再变更。
4. **视图降维**：将共享 IR 分别投影到全视图、UI 视图与自适应视图；此阶段仅执行选择、截断与注释操作，绝不重排或重编号任何内容。

三种视图由此构成渐进式披露的工作流：分析师首先通过 UI 视图扫描会话整体轨迹，随后通过自适应视图定位特定内容块，最后跟随指针进入全视图以恢复该块及其周围环境的完整原文。

$$V_{\text{ui}} \xrightarrow{\text{pointer}} V_{\text{full}}[s:e], \qquad V_{\text{adapt}} \xrightarrow{\text{pointer}} V_{\text{full}}[s:e]$$

**通过率**：VCC 在全部三种模型配置的两种测试划分上均取得更高的 aggregate task_goal_completion：
Opus 配置在 test_normal 上为 94.0% 对比 JSON 的 92.9%，在 test_challenge 上为 86.3% 对比 84.7%。Sonnet 配置分别为 92.9% 对比 90.5% 与 74.3% 对比 71.0%。Haiku 配置分别为 75.6% 对比 71.4% 与 54.7% 对比 53.5%。
在更严格的 scenario_goal 指标上，Opus 于 test_normal 达到 91.1%，较 JSON 的 87.5% 高出 3.6 个百分点。

**Token 消耗**：VCC 将反射器 token 消耗削减一半到三分之二：
Opus 配置从 22.1M 降至 7.6M，Sonnet 从 20.5M 降至 8.8M，Haiku 从 23.3M 降至 12.6M。
单次分析成本也呈相同趋势（Opus 从 124K 降至 43K）。
这一缩减是 UI 视图压缩的直接后果：反射器阅读简洁的会话概览，而非在数千行原始 JSONL 中导航，仅在需要详细检查时才跟随指针跳转。

**记忆文件体积**：VCC 训练得到的记忆文件 consistently 更小：
Opus 配置为 12.0K 字符对比 12.6K，Sonnet 为 15.5K 对比 23.4K，Haiku 为 37.9K 对比 43.3K。
更小的记忆与更高的通过率相关，说明 VCC 使反射器能够生成更聚焦的规则，而非积累冗长或冗余条目。

**弱模型收益**：Haiku 配置显示出最大的绝对增益——
test_normal 的 task_goal 从基线 39.9% 提升至 VCC 的 75.6%，增幅达 35.7 个百分点。
VCC 较 JSON 的优势在 test_normal 的同一指标上也最大，差值为 4.2 个百分点。
这可能是因为较弱的生成器产生更多错误轨迹，其原始形式更难分析，从而使得结构化视图对反射器更有价值。

## 深度分析

VCC 的核心洞见在于将问题从「需要更好的学习算法」重新框定为「需要更好的输入格式」。
实验设计中，学习算法、模型与任务集均未改变，唯一变量是轨迹的呈现方式。
然而这一单点替换同时推动了通过率上升、token 消耗下降与记忆体积压缩三个指标向同一方向移动，暗示它们可能测量的是同一个潜变量——反射器注意力被引导至正确信息的程度。

从信息论角度看，VCC 的三种视图实现了不同精度的信息呈现：
全视图提供无损的规范表面，UI 视图执行语义压缩以匹配人类感知，自适应视图执行查询驱动的选择性投影。
这与传统编译器的多层级 IR 理念一致：不同优化遍在不同抽象层级上操作，但共享同一套命名与坐标系统。
行号一致性作为构造不变量（而非测试验证的性质）尤其值得注意——这类似于编译器中静态单赋值形式通过构造保证 use-def 一致性。

VCC 与多层记忆系统的区分具有方法论意义。
当前许多上下文工程方案追求「更聪明的压缩」（生成摘要、嵌入、知识图谱），但这类抽象引入有损压缩，其忠实度必须经验验证。
VCC 选择另一条路径：不做抽象，只做投影，并通过精确坐标保证每条被省略的信息都可回溯。
这在需要审计、调试或法律可追溯性的场景中可能更具优势。

另一个有趣的发现是 JSON 在部分难度级别上偶尔超过 VCC（如 Haiku d2 test_normal task_goal 79.2% 对比 75.0%）。
作者给出的解释是 JSON 反射器每次分析消耗更多 token，偶尔能捕捉到 VCC 反射器基于指针的导航所遗漏的细节，代价是整体大得多的 token 预算。
这提示结构化视图与原始格式之间可能存在精度与效率的权衡空间，值得未来探索自适应切换策略。

## 局限

当前评估仅限于 AppWorld 单一基准。
向其他智能体基准（SWE-bench、TAU-bench、WebArena、AgentBench）以及其他轨迹学习方法的扩展仍是开放方向。
此外，VCC 的前端目前针对 Claude Code 的 JSONL 方言设计。
支持其他运行时（如 Codex）需要编写新的前端。
IR 与视图降维阶段可完全复用。
自适应视图的相关性谓词当前使用正则匹配，更复杂的谓词（BM25、嵌入、LLM 判断）尚未在实验中进行系统比较。

## 我的笔记

VCC 为智能体轨迹分析领域带来了一个简洁而有力的视角转换：将格式视为基础设施而非装饰。
这一视角与软件工程中「数据即代码」的理念相呼应——当输入数据的结构得到充分尊重时，下游处理逻辑可以显著简化。
从可复现性角度看，VCC 的编译器隐喻具有很强的可扩展性：为新的智能体运行时添加支持仅需替换前端（lexer/parser），后端视图系统保持不变。

一个值得关注的实践方向是将 VCC 集成到现有的智能体可观测性平台（如 LangSmith、Langfuse）中，作为日志预处理层。
当前这些平台主要提供检索与可视化功能，而 VCC 提供的语义投影与指针导航能力可进一步增强人类分析师对复杂轨迹的理解效率。
此外，VCC 的行号坐标系统为自动化调试工具提供了稳定的引用框架，可能降低多智能体轨迹中故障归因的难度。

在上下文工程日益成为独立研究领域的背景下，VCC 的结果强调了一个基础性问题：在追逐更复杂的记忆架构与学习算法之前，应先确保输入格式本身不会成为瓶颈。
这一原则对于资源受限场景（弱模型、长轨迹、高频率调用）尤其重要。
