## 原文摘要翻译

本文围绕 llm-d 项目的 KV 缓存感知路由展开，介绍了其设计原理、实现细节与实测效果。
llm-d 是一款面向容器编排生态的开源大语言模型推理框架，由 IBM、Google、Red Hat 及社区共同维护。
文章首先指出传统推理网关缺乏缓存状态感知能力的问题，随后阐述了一种基于外部处理 Pod 与网关推理扩展的智能路由架构。
文中详细拆解了 KV 缓存索引器的读写路径与核心组件，并给出了生产部署所需的配置示例。
最后，文章通过持续集成管道测试展示了 87.4% 的缓存命中率和 88% 的首词元延迟优化，同时分析了成本节省、能效提升与典型应用场景。

## 创新点

1. **外部处理 Pod 与网关 API 推理扩展的协同路由**
   将外部处理 Pod 嵌入网关 API 推理扩展，使网关在转发请求前能够查询全局 KV 缓存状态并执行智能打分。这打破了传统网关只能做轮询或粘性会话的限制，将缓存感知能力从模型服务器上推到流量入口层。

2. **KV 缓存索引器的双路径设计**
   读路径通过前缀存储、词元处理器、块索引和打分器四级结构实现亚毫秒级 pod 打分。写路径通过 ZMQ 事件流实时同步各 vLLM pod 的块存储与块移除事件。两条路径独立运行，兼顾了查询低延迟与状态一致性。

3. **前缀感知的最长连续匹配打分策略**
   打分器并非简单判断缓存是否命中，而是计算从提示词开头开始的最长连续 KV 块匹配长度，并据此对 pod 排序。这种细粒度策略比整句命中判断更能挖掘部分前缀复用的潜力。

4. **会话亲和路由**
   在前缀感知打分之外引入会话亲和性，确保同一会话的连续请求被导向同一 pod，避免会话内上下文在 pod 间频繁迁移导致的缓存碎片化。

5. **云原生一键化部署**
   通过官方社区 Helm chart 自动配置网关、外部处理 Pod、推理池、推理模型以及 EnvoyFilter，将复杂的分布式推理调度封装为标准的 Kubernetes 资源，降低了生产落地门槛。

## 一句话总结

llm-d 通过网关 API 推理扩展与外部处理 Pod 将 KV 缓存状态感知上推到网关层，利用前缀匹配打分和会话亲和把请求导向缓存最热的 pod，在 Tekton 测试中实现了 87.4% 的缓存命中率和 88% 的首词元延迟降低。

## 研究问题

在分布式大语言模型推理场景中，即使底层模型服务器启用了前缀缓存，流量网关对 GPU 内存中的 KV 缓存状态仍然一无所知。这导致以下问题：

- 轮询或粘性会话策略无法根据实际缓存分布做智能调度；
- 高并发或共享上下文场景下出现大量重复前缀计算；
- GPU 内存与计算资源被冗余占用，延迟和成本双双上升。

本文要解决的问题是：如何在 Kubernetes 原生架构中构建一个全局、近实时的 KV 缓存状态感知与请求路由系统，使得新到达的提示词能够被导向缓存内容最充分的 pod，从而最大化前缀复用、降低首词元延迟并提升整体吞吐。

## 方法主线

### 系统架构概览

llm-d 的 KV 缓存感知路由建立在三个云原生组件之上。

第一，**网关 API 推理扩展**：扩展标准 Kubernetes Gateway API，使其支持面向大语言模型推理的专用路由语义。

第二，**外部处理 Pod**：作为网关 API 推理扩展的外部处理单元，拦截请求后执行智能 pod 打分，再将决策返回给网关。

第三，**KV 缓存索引器**：位于外部处理 Pod 内部，维护全局、近实时的 vLLM pod 缓存块索引。

在数据平面上，客户端请求先到达网关。Envoy 的 ext_proc 过滤器将请求头和请求体转发给外部处理 Pod。外部处理 Pod 调用索引器的读路径得到候选实例排序。网关根据排序结果将请求转发到最优解码实例。与此同时，各 vLLM 实例通过 ZMQ 将缓存事件推送给索引器的写路径，保持索引与真实缓存状态同步。

### 机制流程

llm-d 处理单次推理请求的核心执行链可分为以下四步。

**步骤 1：请求拦截与词元化**
- **输入**：HTTP 推理请求，请求体包含提示词文本与模型名。
- **动作**：Envoy 的 ext_proc 过滤器拦截请求，将请求头和请求体转发给外部处理 Pod。外部处理 Pod 提取提示词，调用前缀存储查询该提示词前缀是否已被词元化。
- **输出去向**：若命中前缀存储，返回已缓存的词元标识序列；否则返回待词元化的原始提示词。两者均进入词元处理器。

**步骤 2：KV 块键生成与索引查询**
- **输入**：词元标识序列（来自步骤 1）。
- **动作**：词元处理器按 vLLM 的 chunk size（每块 16 个词元）对序列分块，并对每块做哈希生成块键。随后查询块索引，返回持有各块键的 pod 列表。
- **输出去向**：pod 到块键集合的映射表进入打分器。

**步骤 3：前缀感知打分与 pod 排序**
- **输入**：pod 到块键集合的映射表、原始提示词词元序列。
- **动作**：打分器采用最长连续前缀匹配策略，对每个 pod 计算从提示词开头开始的连续命中块数量，命中越长得分越高。若开启会话亲和性，同一会话的历史 pod 选择记录会进一步调整排序。
- **输出去向**：按得分降序排列的 pod 候选列表返回给网关路由器。

**步骤 4：请求下发与缓存状态同步**
- **输入**：选定的目标 pod、原始推理请求。
- **动作**：网关将请求转发至最优解码 pod。vLLM 完成推理后，将新生成的 KV 块状态变更事件通过 ZMQ 发布。索引器的写路径接收事件、解码 msgpack 载荷并更新内存中的块索引。
- **输出去向**：推理结果流返回客户端，全局索引更新后进入下一轮查询就绪状态。

### KV 缓存索引器深度拆解

KV 缓存索引器是外部处理 Pod 内部的核心子系统，由六个组件组成。

| 组件 | 职责 | 实现方式 |
|---|---|---|
| 缓存索引器 | 总协调器 | 统筹所有内部模块，响应打分请求 |
| 事件池 | 事件摄取 | 基于 ZMQ 的分片工作线程池，实时消费 vLLM 事件 |
| 块索引 | 核心数据存储 | 内存中的两级 LRU，块哈希到 pod 列表的映射，支持亚毫秒查询 |
| 前缀存储 | 前缀缓存 | LRU 缓存已词元化的提示词前缀，避免重复词元化开销 |
| 词元处理器 | 词元转换 | 与 vLLM 对齐的分块加哈希算法，生成确定性块键 |
| 打分器 | pod 打分 | 最长连续前缀匹配策略，按命中长度排序 |

写路径的事件处理流水线如下。

1. vLLM pod 通过 ZMQ 发布缓存事件。
2. 事件按 `kv@podid@model` 主题格式被接收。
3. 以 pod ID 哈希做分片，确保同一 pod 的事件顺序处理。
4. 工作线程解码 msgpack 载荷中的事件批次。
5. 将变更应用到内存中的块索引。

这种设计避免了引入外部依赖，所有索引状态保存在外部处理 Pod 的内存中，依赖水平扩容外部处理 Pod 本身来提升容量与可用性。

## 关键结果

**缓存命中率与流量分布**
- 总查询数：4,776
- 总缓存命中数：4,176
- 整体缓存命中率：87.4%（较此前 86% 有所提升）
- 主 pod 接收流量：4,772 次查询（99.92%），命中率 87.5%
- 备 pod 接收流量：仅 4 次查询（0.08%，故障转移场景）

**首词元延迟对比**

| 场景 | 无缓存路由 | KV 缓存感知路由 | 提升幅度 |
|---|---|---|---|
| 冷启动推理 | 2,850 ms | 2,850 ms | 基准线 |
| 热缓存命中 | 2,850 ms（最坏情况） | 340 ms | 快 88% |

**延迟与资源**
- 缓存命中请求的响应延迟低于 400 ms
- 冷启动推理响应延迟为 3 至 5 秒
- GPU 内存利用率保持在 90%

**成本与能效**
- 重复提示的计算时间减少 70%，对应 GPU 小时数减少 70%
- 缓存命中的能耗比完整推理低 90%
- 在同等硬件条件下，可支持 3 倍并发用户

## 深度分析

**前缀匹配策略的工程权衡**

最长连续前缀匹配策略看似直观，但在实际系统中面临几个隐式假设。vLLM 的块大小为 16，最大模型长度为 4096，这意味着单条请求最多占用 256 个块。当提示词长度远小于 4096 时，前缀匹配的收益取决于提示词之间的重叠度。文章测试场景为共享前缀的重复请求，天然对前缀匹配友好。若测试改为完全随机的独立提示词，命中率将大幅下降。因此 87.4% 的数值应被理解为特定工作负载下的上限参考，而非通用保证。

**会话亲和的极端集中效应**

99.92% 的流量被导向主 pod。这说明会话亲和性与前缀感知的叠加会产生极强的亲和性。虽然这有利于缓存复用，但也带来单点负载倾斜风险。主 pod 的 GPU 利用率、显存占用和故障面都显著高于其他实例。文章提到 GPU 内存利用率维持在 90%，接近饱和。一旦主 pod 出现内存耗尽或重启，缓存状态将全部丢失。备实例的 0.08% 流量几乎无法提供有效的负载分担。生产环境中需要考虑更细粒度的负载均衡或缓存预热机制。

**与 vLLM 前缀缓存的互补关系**

llm-d 并不替代 vLLM 的前缀缓存，而是将其从单个实例级别扩展到整个集群级别。vLLM 负责单个 GPU 上的块分配与驱逐。llm-d 负责在多个实例之间做全局调度。两者之间的接口是 ZMQ 事件流。这是一种松耦合设计，但也意味着事件延迟或丢包会导致索引与真实状态短暂不一致。文章未报告事件管道的延迟分布和一致性边界，这是后续可深挖的方向。

**部署复杂度的实际成本**

文章列出了较长的前置依赖清单。对于已经运行在云原生栈上的团队，这些依赖是自然的。清单中包括容器编排平台及其 GPU 算子、服务网格、Gateway API 自定义资源、包管理器配置和模型仓库令牌等。但对于使用传统负载均衡或裸机部署的用户，迁移成本不可忽视。此外，外部处理 Pod 作为请求路径上的额外跳数，其自身延迟虽然被描述为亚毫秒，但在高并发场景下是否会成为瓶颈，文章未给出具体数据。

## 局限

1. **文章性质限制**：本文是 Red Hat Developer Blog 的技术文章，并非经过同行评审的学术论文，缺乏严格的实验设计说明、显著性检验和可复现性细节。

2. **缺少对照系统**：文章仅报告了 llm-d 自身在 KV 缓存感知路由开启前后的对比，未与其他 KV 缓存路由方案进行横向比较，例如基于全局调度器的启发式路由或纯粘性会话基线。

3. **测试规模有限**：测试仅包含 4,776 次查询，虽然足以演示功能正确性，但不足以支撑大规模生产系统的性能承诺。高并发、多模型、多租户混合负载下的行为未被覆盖。

4. **无消融实验**：文章未拆解各个组件的独立贡献。例如，前缀感知打分与会话亲和性各自对 87.4% 命中率的贡献是多少。去掉会话亲和性后流量分布和延迟如何变化，这些问题没有答案。

5. **未讨论安全与隔离**：在多租户场景下，共享 KV 缓存索引可能泄露提示词前缀的哈希分布信息。不同租户对同一模型但不同适配器的请求是否会被错误地路由到缓存内容不匹配的 pod，文章未涉及。

6. **故障与退化行为缺失**：当外部处理 Pod 不可用、ZMQ 事件流中断或块索引出现不一致时，系统的降级策略是什么。EnvoyFilter 中设置了失败时允许继续路由，但这只是继续转发，未说明此时路由质量如何保证。

## 我的笔记

**与通用大语言模型服务调度器的关系**

llm-d 的定位类似于容器编排平台原生版的缓存感知请求路由器。它与纯调度器不同，不是基于资源配额做调度，而是基于缓存内容做调度。这更接近于面向推理的专用负载均衡器，类似于某些专有云平台的路由策略。llm-d 的开源实现让这一能力对社区可见，但其强绑定于网关接口、服务网格和 vLLM 的技术栈，意味着通用性受限于云原生生态。

**对检索增强生成与智能体场景的实际价值**

文中提到的检索增强生成、客户支持和代码生成都是前缀重复度高的场景。在我自己的经验中，检索增强生成系统的系统提示词加检索上下文往往占词元总数的 80% 以上，如果这部分能被稳定命中缓存，首词元延迟的改善确实是决定性的。然而，检索得到的片段若每次不同，则只有系统提示词和固定指令部分能复用，实际可命中前缀长度可能远低于理想假设。因此落地前应根据实际提示词模板做前缀重叠度分析，再评估引入 llm-d 的投入产出比。

**可关注的后续方向**

- 将块索引的内存 LRU 扩展为分布式缓存，以支持跨可用区的全局索引；
- 引入基于命中概率的加权负载均衡，避免 99.92% 流量单点倾斜；
- 对 ZMQ 事件流增加至少一次投递语义和延迟监控，量化索引一致性窗口；
- 探索与投机解码或分块预填充的联合优化。
