## 内容摘要

本文是 Galileo 发布的一篇关于 AI agent 护栏框架的深度技术指南，核心目标是帮助企业理解为何传统软件安全措施无法应对自主 agent
的失效模式，并提供可落地的实施框架、架构模式与工具选型建议。文章开篇即引用 Forrester 通过 VentureBeat 披露的研究数据：AI agent
在需要多步推理与工具使用的真实企业任务中失败率高达 70–90%，具体案例包括金融 AI 系统批准欺诈性交易、agent
通过工具使用范围扩张访问未授权数据等。这些失败并非源于团队缺乏专业能力，而是自主 agent
的失效方式与传统软件存在本质差异——它们基于概率性模型做出决策，其输出空间远大于传统软件的状态空间，导致编译期检查和单元测试等传统手段完全失效。

文章首先定义了 AI agent 护栏的核心概念：护栏是系统级安全控制，用于将自主行为约束在可接受的操作边界内。与传统软件依赖编译期检查与确定性执行不同，agent
护栏必须应对概率性决策带来的动态、不可完全预见的风险。NIST 框架将护栏整合到 AI
生命周期的四个核心功能中：Govern（建立组织策略与治理结构）、Map（识别系统上下文与风险类别）、Measure（实施持续评估与度量）、Manage（执行缓解策略与风险处置）。Stanford
AI Safety Seminar
则进一步将护栏控制划分为三个关键运行阶段：部署前（Pre-deployment，包括数据审计、模型验证、全面安全测试）、训练期（Training-time，包括
RLHF、偏见缓解技术与对齐优化）、部署后（Post-deployment，包括实时行为监控、输出过滤与持续性能评估）。这两个框架的结合，为企业在时间与组织两个维度上建立了完整的护栏坐标系。

在企业安全必要性方面，文章用极具冲击力的场景开场：假设团队投入数月构建 agent，通过内部测试后上线生产，却在 72 小时内发现它正在批准违反公司政策的交易。Forrester
的实测数据显示，生产环境中的 AI 模型约有 60% 的失效率，这些不是理论预测，而是来自真实企业部署的测量结果。具体的真实失效模式包括：采购 agent
因优化函数错位而订购了功能错误的组件；金融系统因遗漏细微模式而批准了欺诈交易；具有管理权限的 agent 因优先级误解而延迟了关键安全补丁。MIT CSAIL
的综合风险分析进一步补充了四大关键风险类别：幻觉影响自主决策、对抗性提示注入攻击、agent-工具交互导致的隐私泄露，以及不可预测生产环境中的操作失效。

为应对不同自主程度的需求，文章引入了 Stanford 大学针对企业场景改编的 SAE（Society of Automotive Engineers）框架，将 agent
自主级别划分为三个层次。Level 0–1（无/最小自动化）中，AI 仅提供建议，人类做出所有最终决策，所需的控制包括输入验证（提示注入预防与输入消毒）、输出过滤（内容
moderation 机制）以及全面的日志记录与审计追踪（用于合规与事后审查）。Level 2（部分自动化）中，agent
执行特定任务但需人类监督，并在采取行动前获得明确批准；OpenAI 的实施指南规定该级别必须包含
human-in-the-loop（HITL）审批所有自主动作、严格的作用域限制（尽可能使用只读数据库访问）、回滚机制以撤销有问题的执行，以及针对预期模式的实时监控。Level
3（条件自动化）中，agent 在限定条件下自主运行；Anthropic
的框架规定该级别必须具备上下文边界检测（识别能力边界）、置信度阈值（低置信度时升级至人类）、行为漂移检测（监控行为变化）、关键动作的多因素认证，以及防止级联故障的速率限制。

在护栏分类体系上，文章从三个维度建立了全面的分类框架：按保护目的、按流水线阶段、按实现方式。按目的分为伦理护栏（NIST AI RMF 定义，确保人口统计 parity 与
equalized odds；Google Responsible AI Toolkit 实时筛查歧视性输出，并通过 SynthID 进行内容水印与溯源）、安全护栏（NIST
Generative AI Profile 识别提示注入、数据泄露与未授权系统访问为关键缓解目标；Anthropic ASL-3  safeguards
部署实时分类器检测越狱尝试与对抗性提示，并实施基于角色的访问控制与细粒度认证）与技术护栏（MIT CSAIL Dynamo AI 案例研究提出四大核心组件：评估模块评估 agent
行为、修复工作流响应检测问题、实时监控基础设施追踪性能指标、全面审计日志记录；具体实现包括延迟追踪、token
用量限制防止资源耗尽、模型版本控制确保行为一致性，以及用于合规与调试的审计追踪）。

按流水线阶段分为输入校验（OpenAI Guardrails Python 框架提供第一层防御，在输入到达模型前进行消毒；arXiv LlamaFirewall
论文详述检测提示注入模式的静态分析技术；Google Guardrails API 实施多类别安全筛查与 PII 检测）、执行控制（arXiv
多层护栏论文记录全面的执行控制框架，涵盖基于策略的授权机制、带隔离容器的沙箱执行环境、带熔断器的实时监控，以及基于能力安全的工具访问控制）与输出过滤（Anthropic 研究显示
15–20% 的策略违规发生在工具执行阶段、输出生成之前，因此输出过滤虽必要但不充分；ASL-3 Deployment Safeguards
实施实时分类器检测越狱、对抗性提示与上下文操纵攻击，并结合输出过滤器在送达用户前筛查有害内容）。按实现方式分为基于策略的规则、ML
分类器与混合方法，企业应根据自身数据量、延迟要求与可解释性需求进行选择。

文章进一步总结了 AI agent 安全框架的四大核心组件。第一是访问与范围控制：agent
需要生产访问权限，但传统用户权限存在危险缺口，企业安全框架通过四层授权层级实现细粒度控制——安全主体（托管身份）、角色定义（特定 AI
操作权限）、范围（订阅或资源组级别的工作负载隔离）、角色分配（将主体绑定到定义范围内的角色）。第二是输入输出验证：输出验证需要复杂的模式匹配，包括响应 schema
校验（强制结构与类型约束）、剥离潜在 XSS 载荷、URL 白名单校验、敏感数据模式脱敏，以及 JSON 转义以实现安全渲染。第三是 human-in-the-loop
模式：生产级 HITL 工作流编排为高 stakes 决策提供必要安全控制，同时维持运营速度，需要四大组件——定义触发干预条件的人工审查工作流、为审查者提供 UI
的工人任务模板、用于劳动力管理的工作团队配置、以及将人工反馈整合进 agent 系统的审查结果 API。然而规模化迅速成为问题：研究显示单 agent 每小时执行 1000
次以上动作，全面人工审查不可行，因此领先框架推荐分层审查系统（低风险动作自动检查、高影响决策人工审查、风险评分标记超出置信度阈值的决策）。第四是监控与故障安全：企业可观测性指南规定三支柱监控——metrics（包括
P50/P95/P99 延迟与低于 1% 的错误率）、logging（结构化 JSON 格式、关联 ID、PII 脱敏）与 tracing（分布式追踪工具，span
注释包含模型名称与 token 数量）。熔断器实现需要三态机：closed 状态正常通行、open 状态故障超阈值后快速失败、half-open
状态以有限请求测试恢复；配置参数包括故障阈值、超时持续时间、预期异常类型与降级响应策略。

在实施路径上，文章提出了系统化的四步法。第一步评估风险与定义自主级别：McKinsey 提出五维评估法——识别 agent
能力与工具访问范围、映射潜在影响区域（财务与运营损失）、建立自主动作的风险阈值、定义 HITL 触发条件、记录合规要求；基于评估结果将系统分为 Tier
1（信息检索，仅需自动监控）、Tier 2（可逆动作，需要实时护栏）、Tier 3（金融交易，所有决策需 HITL）。第二步设计护栏架构：传统安全架构无法预测自主 agent
的工具链或强制执行动态边界，因此推荐微服务架构（护栏作为独立可扩展服务）、API 网关模式（在入口点集中策略执行）、Sidecar 模式（与 agent
容器并行部署护栏，实现每容器策略执行）、以及熔断器模式（三态机制防止级联故障）；架构选择取决于延迟要求（低延迟内联护栏 vs
异步验证）、规模要求（请求量）、安全态势（数据驻留）、团队专长与成本约束。第三步选择工具与框架：重点考虑三大因素——部署灵活性（本地部署、私有云
VPC、混合架构，对医疗与金融等监管行业至关重要）、评估成本结构（传统 LLM 评估月成本可达 20 万美元以上，专用评估模型如 Galileo Luna-2 可降低 95%+
同时保持准确性，使 100% 采样成为可能）、运行时延迟（内联护栏增加 2–3 秒会成为用户体验瓶颈，sub-200ms
的阻塞延迟是理想目标）。第四步发布、监控与迭代：关键面板指标包括护栏有效性（误报与漏报率）、agent
成功率（任务完成百分比）、人工干预率、平均恢复时间、以及按类型与严重程度分类的策略违规频率；需区分系统健康指标（延迟分位数、错误率、吞吐量、资源利用率）与工作流理解（端到端
trace 捕获、提示词日志、工具调用序列、护栏决策）；干预阈值需与组织风险容忍度对齐，立即停机条件包括护栏绕过、PII
暴露、财务损失超标或监管违规，降级模式条件包括错误率持续超过 5%、P95 延迟超过基线 2 倍、或护栏误报率超过 20%。

文章末尾介绍了 Galileo 自身的解决方案，虽带有商业推广色彩，但展示了行业前沿方向：CI/CD 集成（每次代码变更自动运行综合评估，拦截未达质量阈值的发布）、Luna-2
多维度响应评估（以传统 LLM 评估 97% 更低的成本评估正确性、毒性、偏见、遵循度等维度）、Agent Protect
实时运行时防护（扫描生产中的每次提示与响应，拦截有害输出并维护详细合规日志）、Insights Engine 智能故障检测（自动聚类相似故障、发现根因模式并推荐修复方案）、以及
CLHF（Continuous Learning via Human Feedback，将专家审查转化为可复用评估器）。

## 关键要点

1. **真实企业场景中的 agent 失败率极高**：Forrester 数据显示多步推理任务失败率 70–90%，生产环境 AI 模型失败率约 60%，说明实验室指标与生产表现之间存在巨大鸿沟，护栏不是锦上添花而是生存必需。
2. **护栏必须覆盖 AI 全生命周期**：NIST 的 Govern-Map-Measure-Manage 与 Stanford 的 Pre-deployment/Training-time/Post-deployment 双重视角，构成了时间维度上完整的护栏坐标系，避免仅在某一阶段部署控制。
3. **自主级别决定控制强度**：SAE Level 0–1 到 Level 3
的划分为企业提供了可操作的决策框架，避免一刀切地采用最昂贵或最严格的控制，实现风险与效率的动态平衡。
4. **按目的、阶段、方式三维分类护栏**：伦理/安全/技术 × 输入/执行/输出 × 规则/ML/混合的三维矩阵，帮助企业根据自身风险画像、合规要求与技术栈选择最优组合。
5. **15–20% 的策略违规发生在工具执行阶段**：Anthropic
的研究数据直接挑战了「仅靠输入输出过滤即可保证安全」的假设，强调执行阶段监控与干预的必要性，这也是本文对综述最核心的贡献之一。
6. **HITL 的规模化困境与分层审查**：单 agent 每小时执行 1000+
操作，全面人工审查不可行，必须建立自动化检查低风险动作、人工审查仅针对高影响决策、风险评分标记超阈值决策的分层系统。
7. **工具选型需权衡三大因素**：部署灵活性（数据驻留与合规）、评估成本（传统 LLM 评估月成本 20 万美元以上，专用模型可降低 95%+）、运行时延迟（内联护栏需控制在
200ms 以内，2–3 秒将成为用户体验瓶颈）。
8. **干预阈值需与组织风险容忍度严格对齐**：立即停机触发条件包括护栏绕过、PII 暴露、财务损失超标或监管违规；降级模式触发条件包括错误率持续超过 5%、P95 延迟超过基线
2 倍、护栏误报率超过 20%。
9. **技术护栏的四大核心组件**：评估模块、修复工作流、实时监控基础设施、全面审计日志，四者缺一不可，共同构成系统级控制闭环。
10. **成本降低使 100% 采样评估成为可能**：专用评估模型（如 Luna-2）将评估成本降低 95% 以上，从根本上改变了「抽样评估 +
事后复盘」的传统模式，使全量实时监控在经济上可行。

## 与综述的关联

本文是综述中「安全与治理」板块的核心非论文来源之一，提供了大量量化证据支撑综述关于「运行时安全护栏与 trace 分析双向闭环」的论点。综述直接引用了 c-017 中的
Forrester 70–90% 失败率、Anthropic 15–20% 工具执行阶段违规率、以及专用评估模型 95%+ 成本降低等数据，用于论证「生产级 agent
需要分层防御体系」与「执行阶段护栏不可替代」两个核心观点。这些数字来自产业调研与第三方研究，与学术论文中的实验数据形成互补，增强了综述论证的外部效度。

在机制层面，c-017 提出的输入-执行-输出三阶段护栏模型，与综述中 AgentVerify 的 LTL 运行时监控（Tier-1 每事件 O(1) 监控器，延迟开销仅
0.04%，灾难性失败检测率 100%）形成了理论与实践的互补：AgentVerify 从形式化验证角度证明运行时监控的可行性与正确性，而 Galileo
从产业数据角度证明其经济必要性与业务价值。两者共同支撑综述中「监控器产生的违规标记本质上是带因果注释的 trace 事件」这一关键洞察，并为「trace
分析系统对违规事件的统计聚合可直接优化护栏阈值」的双向闭环提供了逻辑基础。

此外，c-017 对 autonomy levels（SAE L0–L3）的系统梳理，为综述讨论「多 agent
系统的责任归属与人工介入策略」提供了分类学基础。综述中关于「何时需要人类审批、何时允许完全自主」的讨论，可直接援引本文的 tiered review 框架与 McKinsey
五维评估法作为工程实践参照。特别是在金融、医疗等高风险领域，Level 3
条件自动化所需的上下文边界检测、置信度阈值与漂移检测，与综述中讨论的「monitorability」与「interpretability」需求高度一致。

在工具与产业生态层面，本文对 Galileo Luna-2 评估模型与 Agent Protect 实时防护的介绍，与综述中 Observability 与 Guardrails

提供了来自厂商视角但包含客观数据的参考点。

最后，本文关于 HITL 规模化困境的讨论（单 agent 每小时 1000+ 动作使全面人工审查不可行），与综述中多 agent 系统故障归因的复杂性相互映照。当系统包含数十个
agent 并发执行时，人工审查的负担呈指数级增长，这进一步强化了自动化 trace 分析、智能故障聚类与根因推断（如 Galileo Insights Engine
所展示的方向）的研究必要性。

## 我的笔记

这篇指南的突出价值在于将分散在 NIST、Stanford、MIT CSAIL、Anthropic、OpenAI、McKinsey
等多处的标准、研究与产业实践整合为一个连贯的实施框架。对于正在撰写综述的笔者而言，c-017
提供了一条清晰的叙事线索：从「为什么需要护栏」（高失败率数据与真实案例）→「什么是护栏」（定义、分类与框架）→「如何实施护栏」（四步法与架构模式）→「用什么工具」（选型维度与产品示例）。这种从问题到解决方案的完整闭环，使其成为综述中连接学术研究与工程实践的关键桥梁。

其中关于「15–20% 违规发生在工具执行阶段」的数据尤为关键，也是本文对笔者启发最大的发现。这一数据意味着，传统的「输入过滤 + 输出
moderation」双层架构存在结构性盲区。即便输入经过严格消毒、输出通过内容审查，agent
在执行中间步骤时仍可能调用未授权工具、产生副作用或违反策略。这对综述中强调的「trace
分析必须覆盖中间步骤（工具调用、推理链、状态变更）」提供了强有力的外部证据。未来研究方向可聚焦于如何在工具调用层面实现实时语义拦截（semantic
interception），例如在执行数据库写入或 API 调用前，由轻量级策略引擎进行意图理解与权限校验，而非仅依赖事后 trace
分析。这一方向若取得突破，将从根本上改变当前「先执行、后审计」的被动安全模式。

另一个值得关注的论点是「评估成本的结构性下降」。传统 LLM-as-a-judge 方法每月可产生 20 万美元以上的评估开销，而专用小模型（如 Luna-2）可将成本降低 95%
以上，同时保持准确性。这一趋势如果持续，将使得「100% 采样评估」从理想变为现实，从而根本改变当前生产环境中普遍采用的「抽样评估 +
事后复盘」模式。综述在讨论「可观测性成本与经济性」时，应纳入这一技术经济视角，并探讨成本下降对评估频率、覆盖范围与响应速度的连锁影响。

最后，本文末尾的 Galileo 产品功能介绍虽带有商业推广色彩，但其提出的「将专家审查转化为可复用评估器（CLHF，Continuous Learning via Human
Feedback）」思路具有方法论意义。这与 RLHF（Reinforcement Learning from Human
Feedback）的理念类似，但应用于生产监控与护栏优化领域，可将偶发性的人工审查沉淀为系统性的自动规则。若能在学术研究中验证其跨领域泛化能力与长期稳定性，将有望成为连接「人类监督」与「自动监控」的重要桥梁，甚至催生出新的研究方向——Automated
Guardrail Learning from Human Feedback（AGLHF）。综述在展望「自适应护栏」或「自进化监控系统」时，可将此作为潜在技术路径加以讨论。

从代码、架构与部署角度回答「怎么做」与「用什么做」。两者共同为综述的安全与治理章节奠定了坚实的非论文证据基础。
