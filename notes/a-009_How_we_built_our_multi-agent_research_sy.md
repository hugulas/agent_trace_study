## 内容摘要

本文是 Anthropic 工程团队对其 Claude Research 多智能体研究系统从原型到生产全过程的深度技术复盘。Research 功能使 Claude 能够跨越网页、Google Workspace 以及各类第三方集成工具，自主执行复杂的研究任务。文章系统性地分享了在架构设计、工具设计、提示工程、评估方法和生产可靠性等方面的关键经验与教训。

### 多智能体系统的核心价值

研究工作天然具有开放性和路径依赖性，其所需步骤在事前极难预测，无法通过硬编码的固定流水线来完成。人类在调研时会根据新发现的信息持续调整方向、追踪衍生线索，这种动态性正是多智能体系统擅长的领域。单智能体的线性、一次性（one-shot）流程无法应对这种需要多轮自主决策的任务。

搜索的本质是信息压缩：从庞大的信息库中提炼出关键洞见。子智能体（subagents）通过在各自的上下文窗口中并行运作，同时探索问题的不同方面，然后将最重要的信息浓缩后返回给主导研究智能体（lead research agent）。每个子智能体还实现了关注点分离（separation of concerns）——拥有独立的工具集、提示词和探索轨迹——这减少了路径依赖，使得调查能够更加彻底和独立。

Anthropic 将多智能体系统的性能提升类比为人类社会的进步：虽然个体人类的智力在过去十万年间增长有限，但信息时代人类社会的整体能力呈指数级增长，核心驱动力正是集体智慧和协作能力。同样，即使单个智能体已经具备通用智能，其作为个体运行时仍存在极限，而智能体群组能够完成远为复杂的任务。

### 量化的性能证据

内部评估数据显示，多智能体研究系统在需要广度优先（breadth-first）探索的查询上表现尤为突出。在内部研究评估中，以 Claude Opus 4 为主导智能体、Claude Sonnet 4 为子智能体的多智能体系统，相比单智能体 Claude Opus 4 的性能提升了 90.2%。一个具体案例是：当要求"列举信息技术类 S&P 500 公司的全体董事会成员"时，多智能体系统通过将任务分解给子智能体成功找到了正确答案，而单智能体因顺序搜索速度过慢而未能完成。

更深层的统计分析揭示了一个关键洞察：在 BrowseComp 评估（测试浏览智能体查找难以发现信息的能力）中，有三个因素解释了 95% 的性能方差。其中，token 使用量单独解释了 80% 的方差，工具调用次数和模型选择是另外两个解释因素。这一发现有力验证了多智能体架构的核心假设——通过将工作分布在拥有独立上下文窗口的多个智能体上，可以有效扩展并行推理的容量。此外，升级到 Claude Sonnet 4 带来的性能增益，甚至超过了在 Claude Sonnet 3.7 上将 token 预算翻倍的增益，说明最新模型本身就是 token 使用效率的巨大乘数。

### 代价与适用边界

文章坦诚地指出了多智能体系统的现实代价：在实际数据中，智能体交互的 token 消耗约为普通对话的 4 倍，而多智能体系统整体则约为普通对话的 15 倍。这意味着多智能体系统只有在任务价值足够高、足以覆盖性能提升带来的额外成本时才具有经济可行性。

同时，并非所有任务都适合多智能体架构。当前两类任务尤其不合适：一是需要所有智能体共享同一上下文的任务，二是智能体之间存在大量依赖关系的任务。例如，大多数编码任务的可并行化程度低于研究任务，且 LLM 智能体在实时协调和委派方面的能力仍显不足。Anthropic 的经验表明，多智能体系统最适合的是以下三类高价值任务：涉及重度并行化的任务、信息超出单一上下文窗口容量的任务、以及需要与大量复杂工具交互的任务。

### 架构设计：Orchestrator-Worker 模式

Research 系统采用经典的 orchestrator-worker（调度器-工作者）架构。用户提交查询后，LeadResearcher（主导研究智能体）首先分析查询、制定研究策略，然后生成多个 Specialized Subagents（专业子智能体），让它们并行探索不同方面。子智能体作为智能过滤器，迭代使用搜索工具收集信息，然后将精炼后的发现返回给主导智能体，由其汇编最终答案。

与传统 RAG（Retrieval Augmented Generation，检索增强生成）的静态检索不同——传统 RAG 只是获取与输入查询最相似的文本块并用于生成回复——Research 架构采用动态多步搜索，能够根据新发现自适应调整查询策略、分析结果并构建高质量答案。

完整的工作流程包含多个精心设计的组件：主导智能体首先通过思考（thinking）过程规划方法，并将计划保存到 Memory 模块中持久化——这一点至关重要，因为当上下文窗口超过 20 万 token 时会被截断，必须确保研究计划不会丢失。随后创建具有特定研究任务的子智能体（数量可动态调整）。每个子智能体独立执行网页搜索、使用交错思考（interleaved thinking）评估工具结果，并将发现返回给主导智能体。主导智能体综合结果后判断是否需要进一步研究，如需则可创建额外子智能体或优化策略。当信息收集足够充分后，系统退出研究循环，将所有发现传递给 CitationAgent，由其处理文档和研究报告，识别具体的引用位置，确保所有主张都有恰当的来源归因。最终，附带完整引用的研究结果返回给用户。

### 提示工程的八大原则

多智能体系统与单智能体系统的一个关键差异在于协调复杂度的快速膨胀。早期智能体曾出现诸如为简单查询生成 50 个子智能体、在互联网上 endlessly 搜索不存在的来源、以及相互之间用过度频繁的更新干扰对方等错误行为。由于每个智能体都由提示词引导，提示工程成为改善这些行为的主要杠杆。

1. **像智能体一样思考（Think like your agents）**：要迭代优化提示词，必须理解它们的效果。团队使用 Console 构建模拟环境，用系统中的真实提示词和工具观察智能体逐步工作，立即发现了多种失败模式：智能体在已有足够结果时仍继续搜索、使用过于冗长的搜索查询、或选择错误的工具。有效的提示工程依赖于对智能体形成准确的心智模型。

2. **教会调度器如何委派（Teach the orchestrator how to delegate）**：主导智能体将查询分解为子任务并描述给子智能体。每个子智能体需要明确的目标、输出格式、工具和数据源使用指导，以及清晰的任务边界。缺乏详细任务描述时，智能体会重复工作、留下空白或找不到必要信息。团队早期允许主导智能体给出简短的指令如"研究半导体短缺"，但发现这类指令过于模糊，导致子智能体误解任务或执行完全相同的搜索。

3. **按查询复杂度缩放投入（Scale effort to query complexity）**：智能体难以判断不同任务的适当投入程度，因此团队在提示词中嵌入了缩放规则。简单的事实查找只需 1 个智能体和 3-10 次工具调用；直接对比可能需要 2-4 个子智能体各 10-15 次调用；复杂研究可能需要超过 10 个子智能体并有明确分工。这些显式指南帮助主导智能体高效分配资源，防止在简单查询上过度投入——这是早期版本的常见失败模式。

4. **工具设计与选择至关重要（Tool design and selection are critical）**：智能体-工具接口的重要性不亚于人机接口。使用正确的工具不仅高效，往往还是严格必要的。例如，如果某个信息只存在于 Slack 中，智能体却在网上搜索，那从一开始就注定失败。随着 MCP 服务器为模型提供外部工具访问，这个问题更加复杂，因为智能体会遇到质量参差不齐的陌生工具描述。团队为智能体提供了显式启发式规则：首先检查所有可用工具、将工具使用与用户意图匹配、对外部广泛探索使用网页搜索、以及优先使用专业工具而非通用工具。糟糕的工具描述会将智能体引向完全错误的方向，因此每个工具都需要明确的用途和清晰的描述。

5. **让智能体自我改进（Let agents improve themselves）**：Claude 4 模型在提示工程方面表现出色。给定一个提示词和失败模式，它们能够诊断智能体为何失败并提出改进建议。团队甚至创建了一个工具测试智能体——当给定一个有缺陷的 MCP 工具时，它会尝试使用该工具，然后重写工具描述以避免失败。通过数十次测试，这个智能体发现了关键细微差别和缺陷。改进工具人机工程学的流程使后续使用该新描述的智能体任务完成时间降低了 40%。

6. **先广后窄（Start wide, then narrow down）**：搜索策略应当模仿专家人类研究者的做法——在深入细节前先探索全局概况。智能体往往默认使用过长、过于具体的查询，导致返回结果稀少。团队通过在提示词中引导智能体从简短、广泛的查询开始，评估可用信息，然后逐步缩小关注范围来对抗这一倾向。这一策略有效避免了因查询过于狭窄而遗漏关键信息源的问题，确保研究覆盖面的完整性。

7. **引导思考过程（Guide the thinking process）**：扩展思考模式（extended thinking mode）引导 Claude 在可见的思考过程中输出额外 token，这可以作为一个可控的草稿本（scratchpad）。主导智能体使用思考来规划方法，评估哪些工具适合任务、确定查询复杂度和子智能体数量、定义每个子智能体的角色。测试显示扩展思考改善了指令遵循、推理能力和效率。子智能体也在规划后使用交错思考，在工具结果出来后评估质量、识别空白并优化下一个查询。

8. **并行工具调用改变速度和性能（Parallel tool calling transforms speed and performance）**：复杂研究任务天然需要探索多个来源。早期智能体执行顺序搜索，速度慢得令人痛苦。为了提速，团队引入了两种并行化机制：（1）主导智能体并行启动 3-5 个子智能体而非串行启动；（2）子智能体并行使用 3 个以上工具。这些改变将复杂查询的研究时间最多缩短了 90%，使 Research 能在数分钟内完成过去需要数小时的工作。

### 多智能体系统的有效评估

好的评估对于构建可靠的 AI 应用至关重要，但评估多智能体系统面临独特挑战。传统评估通常假设 AI 每次都遵循相同步骤：给定输入 X，系统应遵循路径 Y 产生输出 Z。但多智能体系统并非如此——即使从完全相同的起点出发，智能体也可能采取完全不同的合理路径达成目标。一个智能体可能搜索三个来源，另一个搜索十个；或者它们可能使用不同工具找到相同答案。由于我们并不总是知道正确的步骤是什么，通常无法仅检查智能体是否遵循了预先规定的"正确"路径。相反，需要灵活的评估方法，既判断智能体是否达成了正确结果，也评估其过程是否合理。

**立即用小样本开始评估**：在智能体开发早期，改动往往产生巨大影响，因为存在大量低垂的果实。一个提示词调整可能将成功率从 30% 提升到 80%。效应量如此之大，仅凭少量测试案例就能发现变化。团队从约 20 个代表真实使用模式的查询开始，测试这些查询往往能清晰看到改动的影响。许多 AI 开发团队拖延创建评估集，认为只有数百个测试案例的大规模评估才有用，但最佳实践是立即从小规模测试开始。

**LLM-as-judge 在做得好时可以扩展**：研究输出难以通过编程方式评估，因为它们是自由格式文本，很少只有一个正确答案。LLM 天然适合评分输出。团队使用 LLM 评判器按照评分标准评估每个输出：事实准确性（主张是否与来源匹配？）、引用准确性（引用来源是否与主张匹配？）、完整性（是否覆盖了所有要求的方面？）、来源质量（是否优先使用一手来源而非低质量的二手来源？）、以及工具效率（是否使用了正确工具且次数合理？）。团队尝试过用多个评判器评估每个组件，但发现单一 LLM 调用、单一提示词输出 0.0-1.0 分数和 pass/fail 等级的方式最为一致，且与人类判断最吻合。当评估测试案例确实有明确答案时（例如"是否准确列出了研发预算最高的三家制药公司？"），这种方法尤其有效。

**人工评估捕捉自动化遗漏的内容**：测试智能体的人会发现评估遗漏的边缘案例，包括异常查询上的幻觉答案、系统故障、或微妙的来源选择偏见。在 Anthropic 的案例中，人工测试者注意到早期智能体始终选择 SEO 优化的内容农场，而非排名较低但更具权威性的学术 PDF 或个人博客。在提示词中添加来源质量启发式规则帮助解决了这一问题。即使在自动化评估的时代，人工测试仍然是必不可少的。

多智能体系统具有涌现行为（emergent behaviors）——这些行为并非通过特定编程产生。例如，对主导智能体的微小改动可能不可预测地改变子智能体的行为。成功需要理解交互模式，而不仅仅是单个智能体的行为。因此，对这些智能体而言最好的提示词不是严格的指令，而是协作框架——定义分工、问题解决方法和投入预算。做好这一点依赖于精心的提示词和工具设计、可靠的启发式规则、可观测性，以及紧密的反馈循环。

### 生产可靠性与工程挑战

在传统软件中，一个 bug 可能破坏某个功能、降低性能或导致宕机。但在智能体系统中，微小改动会级联为巨大的行为变化，这使得为必须在长时间运行过程中保持状态的复杂智能体编写代码变得异常困难。

**智能体是有状态的，错误会复合**：智能体可以运行很长时间，在多次工具调用之间维持状态。这意味着需要持久化执行代码并在沿途处理错误。如果没有有效的缓解措施，微小的系统故障对智能体可能是灾难性的。当错误发生时，不能简单地从头重启——重启成本高昂且用户体验糟糕。相反，团队构建了能够从智能体出错位置恢复的系统。同时利用模型的智能来优雅处理问题：例如，让智能体知道某个工具正在失败并让它自适应，这种方法效果出奇地好。团队将基于 Claude 的 AI 智能体的适应性与确定性的安全保障（如重试逻辑和定期检查点）相结合。

**调试需要新方法**：智能体做出动态决策，即使在相同提示词下，不同运行之间也是非确定性的。这使得调试更加困难。例如，用户会报告智能体"找不到明显存在的信息"，但团队无法看到原因。是使用了糟糕的搜索查询？选择了劣质来源？还是遇到了工具故障？添加完整的生产级 tracing 使团队能够诊断智能体为何失败并系统性修复问题。除了标准可观测性之外，团队还监控智能体决策模式和交互结构——所有这些都无需监控单个对话的内容，以保护用户隐私。这种高层可观测性帮助诊断根因、发现意外行为并修复常见故障。

**部署需要谨慎协调**：智能体系统是由提示词、工具和执行逻辑组成的高度有状态的连续运行网络。这意味着每当部署更新时，智能体可能正处于其流程中的任何位置。因此需要防止善意的代码改动破坏正在运行的智能体。不能同时将所有智能体更新到新版本。相反，团队使用彩虹部署（rainbow deployments），通过逐渐将流量从旧版本转移到新版本，同时保持两者同时运行，来避免中断正在运行的智能体。

**同步执行造成瓶颈**：当前主导智能体同步执行子智能体，等待每组子智能体完成后再继续。这简化了协调，但在智能体之间的信息流中造成了瓶颈。例如，主导智能体无法实时引导子智能体，子智能体之间无法协调，整个系统可能被单个慢速搜索的子智能体阻塞。异步执行将支持额外的并行化——智能体并发工作并在需要时创建新的子智能体。但这种异步性在子智能体之间的结果协调、状态一致性和错误传播方面带来了新的挑战。随着模型能够处理更长更复杂的研究任务，团队预计性能收益将值得承担这种复杂性。

## 关键要点

- **多智能体本质上是并行 token scaling**：token 使用量单独解释了 80% 的性能方差，多智能体架构通过分离上下文窗口实现并行推理容量的扩展
- **Orchestrator-worker 架构是生产验证的模式**：LeadResearcher 负责策略制定与结果综合，Subagents 并行执行专项搜索，CitationAgent 最终校验引用，Memory 模块持久化长程计划
- **动态搜索区别于静态 RAG**：区别于传统 RAG 的一次性检索，系统根据中间发现自适应调整搜索方向、分析深度和查询策略
- **提示工程是主要调优杠杆**：八项原则涵盖心智建模、委派粒度、effort 预算、工具匹配、自我改进、搜索策略、思考引导和并行化
- **智能体可自我改进形成飞轮**：Claude 4 模型能够诊断提示缺陷并提出优化，甚至可重写 MCP 工具描述以提升后续使用效率（示例：任务完成时间降低 40%）
- **评估必须适配非确定性路径**：小样本快速迭代（约 20 个真实查询）+ LLM-as-judge 五维度评分 + 人工边缘案例发现，三者缺一不可
- **生产可靠性需要全新工程手段**：有状态容错恢复、全链路 tracing、彩虹部署、以及面向隐私的高层交互模式监控
- **成本现实不可忽视**：多智能体系统 token 消耗约为普通对话的 15 倍，仅适合高价值、高并行化、信息超单窗口容量、多工具接口的任务
- **异步执行是未来方向**：同步 orchestration 已造成信息流动瓶颈，异步并行将带来性能飞跃，但也对状态一致性和错误传播提出更高要求
- **Memory 持久化是长程任务的关键**：当上下文超过 20 万 token 时会被截断，将研究计划保存到 Memory 模块确保了长程研究任务的连续性和方向稳定性
- **CitationAgent 保障信息可信度**：独立的引用校验智能体负责最终输出的来源归因，确保所有主张都有准确的引用支撑，避免幻觉和错误归因
- **彩虹部署保障升级平滑**：由于智能体可能在任意时刻处于流程中的任何位置，同时更新所有智能体会导致运行中任务中断，彩虹部署通过渐进式流量迁移确保平滑过渡
- **人类测试发现自动化遗漏的偏见**：早期智能体倾向选择 SEO 内容农场而非权威学术来源，人工评估发现了这一系统性偏见并推动提示词改进
- **最新模型是效率乘数**：升级到 Claude Sonnet 4 的性能增益甚至超过在 Claude Sonnet 3.7 上翻倍 token 预算的效果，说明模型质量本身就是并行推理效率的关键变量

## 与综述的关联

本文与 AI 智能体可观测性（observability）、追踪（tracing）与调试（debugging）综述高度相关，主要体现在以下方面：

1. **Tracing 是调试多智能体系统的核心基础设施**：Anthropic 明确指出，由于智能体的动态决策和非确定性特征，"添加完整的生产级 tracing 让我们能够诊断智能体为何失败并系统性修复问题"。这与综述中关于 agent trace 结构化记录、执行轨迹可视化的研究直接呼应。特别值得注意的是，Anthropic 在不监控单个对话内容的前提下，通过高层交互结构监控来诊断根因，这为 privacy-preserving observability 提供了重要的工业实践参考。

2. **多智能体系统的错误级联效应**：文中强调"agentic 系统中错误的复合性质意味着对传统软件而言的微小问题可能完全偏离智能体轨迹"，并指出"一个步骤失败可能导致智能体探索完全不同的轨迹，产生不可预测的结果"。这直接关联到综述中关于 failure diagnosis、failure attribution、root cause analysis 和 cascading failure 的研究议题。

3. **LLM-as-judge 与自动化评估框架**：Anthropic 使用单一 LLM 调用输出 0.0-1.0 分数和 pass/fail 等级的方法，从事实准确性、引用准确性、完整性、来源质量和工具效率五个维度评估研究输出。这与综述中关于 agent evaluation、自动评分、rubric-based assessment 和 benchmark 设计的研究形成直接的实践对照。

4. **Observability 与隐私的工程设计平衡**：团队在"不监控单个对话内容"的前提下，通过监控智能体决策模式和交互结构来诊断根因、发现意外行为和修复常见故障。这与综述中讨论的 telemetry design、audit trail、privacy-preserving monitoring 和 compliant observability 等主题密切相关，展示了工业界如何在用户隐私和系统可观测性之间取得平衡。

5. **工具调用追踪与 MCP 可观测性**：对工具选择错误、工具描述缺陷、以及 MCP 服务器接入后工具质量参差不齐问题的诊断需求，直接对应综述中关于 tool-use traceability、MCP observability、tool description quality 和 heterogeneous tool monitoring 的研究方向。

6. **状态管理与容错恢复的可观测性需求**：文章详细讨论了有状态智能体的错误恢复、检查点机制和彩虹部署，这些生产实践都对状态可视化、执行进度监控和版本迁移追踪提出了明确的可观测性需求。

## 我的笔记

Anthropic 这篇文章堪称多智能体系统从原型到生产的最完整工程复盘之一，其价值不仅在于架构设计，更在于对"为什么有效"和"代价是什么"的坦诚剖析。90.2% 的性能提升令人印象深刻，但"token 消耗解释 80% 方差"这一发现更具理论意义——它暗示多智能体系统的性能提升在很大程度上是"推理容量扩展"的结果，而非某种神秘的涌现智能。这对综述的方法论有重要启示：在评估多智能体系统时，必须将 token 效率与结果质量同时纳入指标，否则可能产生误导性的结论。一个消耗 15 倍 token 却只提升 10% 准确率的系统，与一个消耗 2 倍 token 却提升 50% 准确率的系统，其工程价值是完全不同的。

另一个值得深思的点是"智能体自我改进"循环——让模型自己优化提示和工具描述，实际上构建了一个元学习（meta-learning）的飞轮。这与综述中提到的自动调试、自适应 agent、self-improving systems 等前沿方向高度契合。特别值得注意的是，工具测试智能体通过重写 MCP 工具描述使后续智能体效率提升 40%，这说明在工具生态层面也存在巨大的优化空间，而这类优化往往被传统的手动文档编写所忽视。

最后，关于同步执行瓶颈的讨论预示了异步多智能体架构将成为下一个工程热点。异步带来的状态一致性、错误传播和结果协调问题，也将对 tracing 和 observability 提出更高要求——如何在并发执行的多个智能体之间维持因果关系的清晰可追溯性，将是综述领域需要重点关注的问题。
