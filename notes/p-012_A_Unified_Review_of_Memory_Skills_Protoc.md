# Externalization in LLM Agents: A Unified Review of Memory, Skills, Protocols and Harness Engineering

## 核心信息

- **标题**: Externalization in LLM Agents: A Unified Review of Memory, Skills, Protocols and Harness Engineering
- **作者**: Chenyu Zhou, Huacan Chai, Wenteng Chen, Zihan Guo, Rong Shan, Yuanyi Song, Tianyi Xu, Yingxuan Yang, Aofan Yu, Weiming Zhang, Congmin Zheng, Jiachen Zhu, Zeyu Zheng, Zhuosheng Zhang, Xingyu Lou, Changwang Zhang, Zhihui Fu, Jun Wang, Weiwen Liu, Jianghao Lin, Weinan Zhang
- **机构**: 上海交通大学、中山大学、上海人工智能创新中心、卡内基梅隆大学、OPPO
- **年份**: 2026
- **来源**: arXiv
- **DOI**: 10.48550/arXiv.2604.08224
- **arXiv ID**: 2604.08224
- **论文类型**: 综述（survey）
- **核心主题**: 以"外部化"（externalization）为统一视角，系统梳理大语言模型智能体在记忆、技能、协议与Harness工程四个维度的研究进展

## 原文摘要翻译

大语言模型智能体的构建方式正日益从修改模型权重，转向围绕模型重新组织运行时环境。早期系统期望模型在内部恢复的能力，如今被外部化为记忆存储、可复用技能、交互协议以及使这些模块在实际中稳定运行的Harness层。本文借助认知人工制品（cognitive artifacts）的视角回顾这一转变，指出智能体基础设施之所以重要，并非仅仅因为它增加了辅助组件，而是因为它将原本困难的认知负担转化为模型能够更可靠处理的形式。在此视角下，记忆外化了跨时间的状态，技能外化了程序性专业知识，协议外化了交互结构，而Harness工程则是将它们协调为受控执行的统一层。本文追溯了从权重到上下文再到Harness的历史演进，分析了记忆、技能和协议作为三种不同但耦合的外部化形式，并考察了它们在一个更大智能体系统中的相互作用，同时讨论了参数能力与外部化能力之间的权衡、自进化Harness等新兴方向，以及在评估、治理和模型与外部基础设施长期协同进化中的开放挑战。

## 创新点

1. **提出"外部化"作为统一过渡逻辑**：将记忆、技能、协议与Harness工程整合到同一个理论框架下，认为它们共享同一种设计原理——把模型难以内部承担的认知负担逐步迁移到外部结构中。
2. **引入认知人工制品视角分析智能体设计**：借用 Norman 的认知人工制品理论，指出外部化的本质不是增加信息，而是对任务进行表征转换（representational transformation），使回忆变为识别、即兴生成变为组合、临时协调变为结构化契约。
3. **构建三层能力演进模型**：提出从 Weights（权重）到 Context（上下文）再到 Harness（ harness 层）的历史演进框架，为社区研究主题的迁移提供了系统级解释。
4. **系统梳理 Harness 工程的六个分析维度**：将 Harness 设计分解为智能体循环与控制流、沙箱隔离、人类监督、可观测性、配置与策略编码、上下文预算管理六个维度，提供了超越具体实现的分析框架。
5. **提出跨模块耦合分析**：详细分析了记忆、技能、协议之间的六种系统级数据流，揭示了外部化系统自我强化与错误级联的双重动态。

## 一句话总结

本文以"外部化"为核心视角，将 LLM 智能体的记忆、技能、协议与 Harness 工程统一为同一套认知负担迁移框架，指出可靠智能体的关键不仅在于更强的模型，而在于更好的外部认知基础设施。

## 研究问题

LLM 智能体面临的根本挑战是什么？传统上，社区将智能体能力的提升等同于模型规模的扩大和训练数据的增加。然而，本文指出，仅凭更大的模型权重无法解决三类反复出现的失配：

- **连续性失配**：上下文窗口有限且会话记忆薄弱，导致长程任务中的状态难以保持。
- **方差失配**：多步复杂流程每次都需要重新推导，导致执行不稳定、遗漏步骤或提前终止。
- **协调失配**：与外部工具、服务和其他智能体的交互若仅靠自由形式提示，极易产生脆弱和不一致的结果。

本文的核心主张是：这三类失配分别对应三种外部化维度——记忆外化跨时间的状态负担，技能外化程序性专业知识的负担，协议外化交互结构的负担。外部化的本质并非简单地给模型增加辅助组件，而是通过表征转换，让模型用已有的能力更可靠地解决问题。基础模型本身可以不变，改变的是任务在其面前的表征形式。

## 数据与任务定义

本文是一篇系统综述，不针对单一数据集或基准任务，而是覆盖 LLM 智能体领域从 2022 年到 2026 年的主要技术进展。其分析范围涵盖：

- **记忆系统**：从单调上下文到检索增强存储、分层记忆编排、自适应记忆系统的全谱系架构。
- **技能系统**：从原子执行原语（如 function calling）到大规模工具选择，再到封装化技能单元的演进。
- **协议生态**：包括智能体与工具之间的协议（如 MCP）、智能体之间的协议（如 A2A）、智能体与用户之间的协议（如 A2UI / AG-UI），以及垂直领域协议（如 UCP、AP2）。
- **Harness 系统**：从早期自动智能体框架到当代生产级代码助手系统的工程实践。

论文将上述所有对象纳入"外部化"框架进行统一分析，把它们视为同一设计原理在不同认知维度上的具体表现。

## 方法主线

本文采用**概念综述与系统框架构建**的研究方法，核心方法论特点如下：

- **理论锚定**：以 Norman 的认知人工制品理论和 Kirsh 的互补策略理论为分析基石，将工程设计选择与认知科学中的外部化概念对接。
- **历史追踪**：通过"Weights → Context → Harness"三层模型，梳理社区研究重心从参数能力到提示工程再到基础设施的迁移轨迹。
- **维度分解**：将智能体系统分解为记忆、技能、协议三个外部化模块，以及 Harness 作为统一协调层，逐一进行内容、架构、整合三个层面的深度分析。
- **跨模块耦合分析**：在单模块分析的基础上，进一步考察记忆→技能、技能→协议、协议→记忆等六种系统级交互流。
- **边界与权衡讨论**：从更新频率、可复用性、可审计性、延迟与上下文负担四个维度，分析参数能力与外部化能力的最佳划分问题。

### 机制流程

本文提出的外部化智能体框架可概括为以下核心执行循环：

1. **输入任务描述与环境状态，查询外部记忆存储以得到相关历史上下文**。Harness 根据当前任务目标从记忆层检索工作上下文、情景经验或语义知识，为模型提供经过筛选的输入。
2. **提取适用的技能文件，送入模型进行渐进式加载与解码**。系统通过注册表发现匹配的技能，按渐进披露策略将技能摘要或完整指南注入上下文，将任务从即兴生成转化为受控组合。
3. **生成结构化动作意图，通过协议层对齐到工具接口并输出执行结果**。模型产生的意图被协议层捕获并归一化为类型化调用，经权限检查后路由到具体工具或子智能体，完成动作执行。
4. **观测反馈并更新记忆状态，完成一轮外部化认知循环**。执行轨迹、工具返回结果与评估信号被写回记忆层，为下一轮检索、技能选择与协议路由提供经验依据。

## 关键结果

作为一篇综述，本文的主要"结果"体现为对领域现状的系统刻画和未来方向的清晰预判：

- **记忆架构的四阶段演进**：当前系统已从单调上下文演进至自适应记忆系统，核心转变是从被动存储到主动控制策略。
- **技能系统的三层演进**：社区经历了从原子执行原语到大规模原语选择，再到封装化技能单元的跃迁，关键转变是将能力视为可加载、可复用的程序性知识包。
- **协议生态的四大家族**：智能体与工具之间的协议、智能体之间的协议、智能体与用户之间的协议，以及垂直领域协议，共同构成了外部化交互的结构化契约层。
- **Harness 设计的六维收敛**：当代生产系统（Codex、Claude Code 等）在循环控制、沙箱隔离、人类监督、可观测性、配置治理和上下文管理六个维度上呈现惊人的结构趋同。
- **跨模块耦合的自我强化与风险级联**：记忆改善促进技能蒸馏，技能执行丰富记忆数据，但污染的记忆也可通过技能执行进一步扩散错误。

## 深度分析

### 记忆作为外部化状态

记忆系统的核心认知功能是将跨时间连续性这一不可能的上下文内负担，转化为可检索的外部资源。论文将外部化状态细分为四个层次：

- **工作上下文**：当前任务的活跃中间状态（打开的文件、部分完成的计划、执行检查点），一旦上下文重置即消失。
- **情景经验**：记录过去运行中的决策点、工具调用、失败与结果，支持反思、恢复与先例检索。
- **语义知识**：超越单一场景的抽象信息，如领域事实、通用启发式、项目惯例。
- **个性化记忆**：针对特定用户、团队或环境的稳定信息，如偏好、习惯与约束，需遵循不同的保留、检索和隐私规则。

![Fig. 4 记忆作为外部化状态](p-012_A_Unified_Review_of_Memory_Skills_Protoc/images/fig4_memory_externalization.png)

记忆架构的演进呈现四个阶段：单调上下文 → 上下文加检索存储 → 分层记忆与编排 → 自适应记忆系统。关键转变不在于存储容量的扩大，而在于写入、检索、压缩与遗忘策略的逐步显性化。在 Harness 时代，记忆不再只是孤立存储模块，而是运行时为规划、执行和恢复循环提供选择性可读状态的底层基础设施。

### 技能作为外部化专业知识

技能外化的是程序性负担。一个语言模型可能原则上知道如何完成任务，但可靠的执行仍需要每次重新构造工作流程、默认值和约束条件，这表现为方差：遗漏步骤、不稳定的工具使用和不一致的终止条件。

论文将技能的内容分解为三个耦合组件：

- **操作程序**：复杂任务分解为步骤、阶段、依赖和终止条件的任务骨架。
- **决策启发式**：在分支点处根据经验衍生的实用规则进行局部策略选择。
- **规范约束**：界定程序可接受执行范围的条件（测试要求、范围限制、访问限制等）。

![Fig. 5 技能作为外部化专业知识](p-012_A_Unified_Review_of_Memory_Skills_Protoc/images/fig5_skill_externalization.png)

技能系统的演进经历了三个阶段：原子执行原语（如 function calling）→ 大规模原语选择（如 Gorilla、ToolLLM）→ 封装化技能单元（如 SKILL.md、技能注册表）。成熟技能系统的关键不在于静态文档的数量，而在于技能能否被可靠地发现、渐进式披露、绑定到可执行接口并进行组合。技能边界条件包括语义对齐、可移植性与陈旧性、不安全组合以及上下文依赖退化。

### 协议作为外部化交互

协议外化的是交互负担。裸模型可能推断出应该调用工具或委派子任务，但若无显式契约，它还必须即兴构造消息格式、参数结构、生命周期语义、权限与恢复行为。

论文将协议外化的内容归纳为四个维度：

- **调用语法**：参数名、类型、顺序和返回结构的模式化定义。
- **生命周期语义**：多步交互中的协调规则（谁下一步行动、何时任务完成或失败）。
- **权限与信任边界**：谁被授权、数据可以流向何处、需要产生何种证据。
- **发现元数据**：能力注册表、能力卡片和模式端点，替代提示中嵌入的隐式知识。

![Fig. 6 协议作为外部化交互](p-012_A_Unified_Review_of_Memory_Skills_Protoc/images/fig6_protocol_externalization.png)

在 Harness 工程中，协议层承担意图捕获与归一化、能力发现与工具描述、会话与生命周期管理三项关键功能。协议是认知人工制品在交互维度上的最强形式，因为它从关键路径上移除了整个类别的推理任务。

### Harness 作为认知环境

Harness 不是第四种外部化，而是承载并协调三种外部化模块的运行时环境。论文将其分解为六个分析维度：

1. **智能体循环与控制流**：感知→检索→规划→行动→观测的时序骨架，加上终止、递归和资源消耗的显式治理。
2. **沙箱与执行隔离**：创建受控执行边界，限制智能体可读、可写和可修改的内容，同时提供可复现性与回滚保证。
3. **人类监督与审批门控**：在执行循环中插入可配置的人工干预点，将自主性从二元属性转化为 Harness 的可调参数。
4. **可观测性与结构化反馈**：结构化日志、执行轨迹和聚合指标，既支持外部调试与审计，也闭合内部反馈循环。
5. **配置、权限与策略编码**：将治理规则外化为可声明、可版本化和可审计的配置层。
6. **上下文预算管理**：协调记忆检索、技能加载、协议模式、工具描述与模型推理轨迹对有限 token 预算的竞争。

![Fig. 7 Harness 作为认知环境](p-012_A_Unified_Review_of_Memory_Skills_Protoc/images/fig7_harness_environment.png)

论文指出，当代生产系统（如 OpenAI Codex 和 Claude Code）在上述六个维度上的结构趋同表明，这些不是偶然的实现选择，而是外部化智能体的结构性需求。Harness 工程的本质是构造认知环境——通过安排信息、工具、权限和程序的结构，使期望的行为更容易执行，不期望的行为更难产生。

### 跨模块耦合与权衡空间

论文对记忆、技能、协议之间的系统级耦合进行了精细刻画，识别出六条主要数据流：

- **记忆 → 技能**：经验蒸馏，将重复轨迹抽象为可复用程序。
- **技能 → 记忆**：执行记录，使技能层具备自我校正能力。
- **技能 → 协议**：能力调用，通过协议化接口将高层意图转化为类型化动作。
- **协议 → 技能**：能力生成，标准化接口降低了封装领域最佳实践为技能的门槛。
- **记忆 → 协议**：策略选择，历史成功率影响协议路径选择。
- **协议 → 记忆**：结果同化，将结构化交互结果写回持久状态。

![Fig. 8 记忆、技能与协议之间的耦合关系](p-012_A_Unified_Review_of_Memory_Skills_Protoc/images/fig8_cut_analysis.png)

这三条模块之间的竞争还涉及同一稀缺资源——上下文窗口。扩展任一模块的上下文占用必然压缩其他模块的空间，因此 Harness 必须在每一步执行中动态协调相对预算分配。

在参数能力与外部化能力的权衡上，论文提出四个划分维度：

- **更新频率与时效衰减**：快速变化的知识和程序更适合外部化；稳定的背景能力（语言理解、常识推理）仍更适合参数化存储。
- **可复用性与多智能体可移植性**：跨任务、跨用户、跨智能体反复需要的能力，外部化可提高可移植性和组合性。
- **可审计性、治理与对齐**：需要检查、审批、回滚或策略强制时，外部化工件相比不透明参数行为具有明确优势。
- **延迟、简洁性与上下文负担**：外部化将计算和组织成本从模型前向传播转移到周围系统，对超高速、低方差或纯语义任务，参数化知识往往更简单可靠。

### 未来方向

论文展望了六个延续外部化逻辑的未来方向：

1. **扩展前沿**：规划与目标管理、评估与验证、编排逻辑本身，以及多模态感知都可能成为进一步外部化的候选对象。
2. **具身外部化**：从数字智能体到具身系统的分解逻辑——高层"大脑"负责任务分解与规划，VLA 模型退化为可调用"小脑"技能模块——与本文框架完全平行。
3. **自进化 Harness**：在模块级、系统级和边界级三个层次上使 Harness 能够自适应调整，但也放大了治理风险。
4. **成本、风险与治理**：外部化本身产生认知开销和安全攻击面，需要与治理作为协同设计层。
5. **从私有脚手架到共享基础设施**：共享记忆、共享技能、共享协议将分析单元从单个智能体转向生态系统。
6. **测量外部化**：当前基准测试系统性地低估了外部化基础设施的贡献，需要评估可迁移性、可维护性、恢复鲁棒性、上下文效率和治理质量等新维度。

## 局限

1. **理论框架的验证性局限**：本文以 Norman 的认知人工制品理论和 Kirsh 的互补策略理论为核心分析工具，但并未通过大规模定量实验验证该框架对系统设计的预测能力，其解释力主要建立在概念对应而非因果推断之上。
2. **工业实践与学术综述的时滞**：论文引用截止于 2026 年初，而智能体领域的技术迭代极快，MCP、A2A 等协议标准以及 Claude Code、Codex 等生产系统的具体实现可能在论文发表后已有显著演进。
3. **缺乏统一评估基准**：作为概念综述，本文未提出可直接运行的新基准或数据集，读者难以通过单一实验复现或检验其主张。
4. **外部化边界的动态性**：论文承认参数能力与外部化能力的最佳划分并非静态，但未能给出在不同任务类型、模型规模和应用场景下的具体决策规则或定量指导。
5. **安全风险的讨论以定性为主**：虽然论文识别了记忆投毒、恶意技能注入和协议欺骗等风险，但对这些攻击的定量发生率、防御机制的有效性评估仍依赖引用的第三方研究，缺乏系统的安全审计框架。

## 我的笔记

这篇综述的最大价值在于提供了一个**高于具体技术、低于哲学空谈**的中层理论框架。在智能体领域被各种新名词（如检索增强生成、函数调用、技能库）淹没的当下，"外部化"这一概念像一根红线，把看似无关的技术方向串成了同一个故事。

我尤其受启发的是论文对**表征转换**的强调。外部化不是给模型"加外挂"，而是改变模型面对的认知任务本身——这个视角把工程决策和认知科学联系了起来。当我们在项目中决定是否把提示模板抽成 SKILL.md、是否引入向量检索、是否用 MCP 替代裸 API 调用时，我们实际上是在做同一种决策：哪些认知负担应该留在模型内部，哪些应该外化到环境中。

论文对 **Harness 工程六维度**的归纳也非常实用。它让我意识到，一个"能跑"的智能体 demo 和一个"能上线"的生产智能体之间的差距，不是模型能力，而是这六个维度的工程成熟度。特别是"上下文预算管理"这一点，经常被忽视——我们在实际项目中经常遇到技能文件太大挤占记忆空间、或者检索结果太多淹没当前任务的情况，这正是 Harness 层需要解决的资源协调问题。

在后续工作中，我打算重点关注两个方向：一是**技能的可移植性评估**，即同一个 SKILL.md 在不同模型、不同工具链、不同框架下的行为一致性；二是**自进化 Harness 的治理机制**，如何在允许 Harness 自适应优化的同时，防止错误积累和攻击面扩大。这篇综述为这两个方向提供了很好的问题定义和理论起点。
