## 原文摘要翻译

人工智能智能体在复杂人类工作流程中的广泛普及正在推动大语言模型 token 消耗的快速增长。当智能体被部署到需要大量 token 的任务上时，有三个问题自然浮现：第一，人工智能智能体把 token 花在了哪里？第二，哪些模型在 token 效率上更优？第三，智能体能否在任务执行前预测自己的 token 使用量？

本文首次对智能体编码任务中的消耗模式进行了系统性研究。我们在 SWE-bench Verified 基准上分析了八种前沿大语言模型的执行轨迹，并评估了模型在任务执行前预测自身成本的能力。

研究发现任务消耗呈现五大特征。第一，消耗量极为高昂，比代码推理和对话任务多出约一千倍，输入量而非输出量是成本主导因素。第二，使用量高度可变且本质上随机：同一任务不同运行间总量可相差高达三十倍，更高消耗不带来更高准确率，准确率通常在中等成本处峰值并饱和。第三，模型效率差异显著。两款高消耗模型的平均消耗量比 GPT-5 多出一百五十万以上。这两款模型分别为 Kimi-K2 与 Claude 第四代半版本。第四，人类专家评定的任务难度与实际成本仅存在弱相关，揭示了人类感知复杂性与智能体实际计算付出之间的根本鸿沟。第五，前沿模型无法准确预测自身的使用量，相关最高为 0.39，且系统性地低估真实成本。本研究为智能体经济学提供了新的洞察，并有望激发该方向的后续研究。

## 创新点

1. **首个大规模智能体编码任务 token 消耗实证研究**。此前工作主要关注单次推理或对话场景的 token 分布，本文首次将分析对象扩展到端到端的多轮智能体编码任务，覆盖了从环境交互到工具调用的完整执行轨迹。

2. **系统性的模型效率对比框架**。在控制任务和智能体框架（OpenHands）不变的前提下，对八种前沿模型进行同任务对比，消除了任务难度差异带来的混淆，从而分离出模型自身行为对 token 效率的影响。

3. **自预测 token 消耗的实验设计**。提出了让模型在执行前预估自身输入与输出 token 数的任务范式，并量化了预测准确率与实际开销之间的权衡，为智能体定价透明化提供了实证基础。

4. **细粒度阶段与轮次级别的成本分解**。将智能体轨迹划分为五个功能阶段（Setup、Explore、Fix、Validate、Closeout），并追踪每轮输入、输出、缓存读取与缓存创建的成本构成，揭示了输入 token 在长程轨迹中的指数级累积效应。

## 一句话总结

本文首次系统量化了前沿大语言模型在智能体编码任务中的 token 消耗模式，发现输入 token 主导成本、同任务方差可达三十倍、模型效率差异超过百万 token，且模型自身对成本的预测能力仍然薄弱。

## 研究问题

随着编码智能体（Coding Agents）的能力不断扩展，它们已被越来越多地应用于超越纯编码的复杂长程任务。然而，当前智能体的定价模式面临两大批评：一是缺乏透明度，用户在任务完成前无法知晓最终成本；二是无完成保障，任务失败后用户仍需支付全部 token 费用。这些问题汇聚为一个核心研究问题：我们能否在任务执行前预测 token 消耗？

围绕这一核心问题，本文提出三个具体研究问题：

- 智能体在编码任务中把 token 花在了哪些地方？
- 在相同任务和相同智能体框架下，哪些模型的 token 效率更高？
- 前沿模型能否在执行前准确预测自身的 token 使用量？

若能在执行前给出可靠的 token 用量估计，用户将能更好地理解潜在成本并据此选择模型；服务提供商也可设计更清晰的定价层级、设置预算上限并在高额账单产生前触发预警。

## 数据与任务定义

**基准数据集**：SWE-bench Verified，包含 500 个真实的 GitHub 代码库问题实例。该数据集要求智能体阅读代码仓库、理解问题描述、自主调用工具并生成修复补丁。

**智能体框架**：OpenHands（前身为 OpenDevin），一个支持多轮工具调用的开源智能体框架。在每个轮次中，大语言模型基于当前提示生成响应，随后发起工具调用并执行，环境返回执行结果后进入下一轮。

**评测模型**：八种前沿大语言模型。
Claude 系列包括 Sonnet-3.7、Sonnet-4 与 Sonnet-4.5。
GPT 系列包括 GPT-5 与 GPT-5.2。
其余模型为 Qwen3-Coder-480B-A35B-Instruct、Kimi-K2 和 Gemini-3-Pro Preview。

**实验设置**：每个问题实例在每种模型上进行四次独立运行，共产生约 16000 条执行轨迹。采集的数据包括完整执行轨迹、推理日志、中间输出、评估结果及元数据。

**评估指标**：
- 总 token 数（输入与输出之和）
- 输入与输出比例
- 货币成本（基于各模型提供商定价）
- 阶段与轮次级别的 token 拆分
- 缓存创建与缓存读取量
- 自预测的皮尔逊相关系数
- 预测开销（预测成本与实际成本的比值）

## 方法主线

本文采用纯实证分析路线，不提出新的训练方法或模型架构，而是通过控制变量实验和细粒度轨迹解析来回答研究问题。核心方法包含两条主线：一是对已有执行轨迹的消耗模式进行描述性统计与对比分析；二是让模型在执行前进行自预测，评估其预测能力。

### 机制流程

**步骤 1：轨迹生成与采集**
- 输入信号：SWE-bench Verified 的问题描述、对应代码仓库、OpenHands 智能体框架配置、八种大语言模型的接口。
- 关键动作：每种模型对每个问题执行四轮独立运行。智能体通过多轮循环自主与环境交互，每轮包含四个子步骤——模型基于当前上下文生成自然语言响应；解析响应并提取工具调用指令；在沙箱环境中执行工具调用；将环境反馈追加到上下文并进入下一轮。
- 输出信号：完整的执行轨迹、每轮原始 token 计数、工具调用记录、最终补丁与评估结果（通过/失败）。

**步骤 2：细粒度指标抽取与成本计算**
- 输入信号：原始执行轨迹与推理日志、各模型提供商定价费率表。
- 关键动作：从模型调用历史中提取每轮输入 token 数、输出 token 数、动作类型；依据定价公式计算货币成本。对支持缓存的模型进一步拆分缓存创建量与缓存读取量。非缓存输入计算公式为：
  $$\text{Input}_{\text{non-cached}} = \text{Input}_{\text{total}} - \text{CacheRead}$$
  单轮成本公式为：
  $$\text{Cost}_{\text{round}} = (\text{Input}_{\text{non-cached}} \times r_{\text{in}}) + (\text{Output} \times r_{\text{out}}) + (\text{CacheCreation} \times r_{\text{cache create}}) + (\text{CacheRead} \times r_{\text{cache read}})$$
  各项系数分别对应输入费率、输出费率、缓存创建费率与缓存读取费率。
- 输出信号：结构化轨迹数据，包含每轮 token 明细、阶段标签（Setup / Explore / Fix / Validate / Closeout）、累计成本曲线。

**步骤 3：统计建模与自预测实验**
- 输入信号：结构化轨迹数据、原始问题描述与仓库信息。
- 关键动作：对 token 消耗进行跨模型、跨问题、跨轮次的描述性统计与混合效应回归；设计自预测提示，要求模型在零先验执行信息下预估输入与输出 token 数，以结构化格式返回估计值；计算预测值与真实值之间的皮尔逊相关系数及系统偏差。
- 输出信号：五大核心发现、模型效率排序、预测准确率与开销权衡报告。

## 关键结果

本文提出了五项核心发现：

**发现一：智能体编码任务的 token 消耗极为高昂，输入 token 是成本主导因素。** 智能体编码任务的平均 token 消耗约为四百一十七万，是代码推理任务（约一千一百九十）和代码对话任务（约三千三百九十）的一千倍左右。平均任务成本约为一点八六美元，而代码推理和对话任务分别仅为零点零一六美元和零点零二三美元。更关键的是，输入与输出之比在智能体编码任务中高达约一百五十四，即使启用了 token 缓存，输入 token 依然主导总成本。

**发现二：token 使用量高度可变且本质上随机，更高消耗不带来更高准确率。** 同一任务在不同运行间的总 token 数可相差高达三十倍。通过混合效应回归分析，作者发现当同一问题的运行成本从最低分位提升至中低分位时，准确率有轻微上升；但在更高成本区间，准确率趋于饱和甚至下降。这表明超额 token 消耗往往反映的是无效探索，而非更深层次的推理。

**发现三：模型间的 token 效率差异显著。** 在相同的五百个 SWE-bench 问题上，Kimi-K2 和 Claude-Sonnet-4.5 平均比 GPT-5 多消耗超过一百五十万 token。即使在所有模型都能成功解决的简单子集（共二百三十题）上，这一差距依然存在，说明效率差异源于模型自身的行为特征，而非任务固有难度。

**发现四：人类专家评定的任务难度与实际 token 成本仅弱相关。** 作者邀请人类专家按预计解决时间对任务难度进行评级，分为三组：小于十五分钟、十五分钟至一小时、大于一小时。结果显示，专家难度与实际总 token 数之间的 Kendall-b 相关系数仅为 0.32，95% 置信区间为 [0.25, 0.38]。有百分之六点七的短耗时任务消耗高于长耗时任务的平均水平，而百分之十一点一的长耗时任务消耗低于短耗时任务的平均水平。这说明人类对复杂性的直觉与智能体实际付出的计算努力之间存在根本鸿沟。

## 深度分析

**输入 token 的指数级累积是智能体成本的核心驱动力。** 与单次代码推理或对话不同，智能体任务通过多轮循环持续将环境反馈追加到上下文中，反馈内容包括代码库内容、工具输出和测试结果。图一显示，智能体编码的平均输入量约为四百一十七万，而代码推理和代码对话分别仅约一千一百九十和三千三百九十。这种差异并非来自输出量的增加，而是来自输入上下文的持续膨胀。图八的阶段级分析进一步表明，输入量在五个功能阶段均占据主导地位，且缓存读取成本随轮次稳定上升。这五个阶段分别是环境初始化、代码探索、修复、验证与收尾。表一显示修复阶段（33.53%）和探索阶段（30.37%）合计占全部轮次的六成以上，意味着大部分成本产生于代码编辑和根因分析环节。

## 局限

1. **模型覆盖范围有限**。研究仅覆盖八种前沿模型，虽然已是现有工作中较广泛的样本，但仍只是智能体模型全景中的一小部分。收集完整执行轨迹的计算成本高昂，限制了可纳入的模型数量。

2. **单一智能体框架**。所有实验均基于 OpenHands 框架，使用不同的智能体架构（如不同工具集、不同反思机制）可能会改变 token 消耗模式。本文观察到的定性模式在测试模型中一致，但在更广泛的架构和智能体设计上的验证将进一步增强其普适性。

3. **自预测的实用性限制**。当前大语言模型的点估计能力尚不足以支持精确到单个任务的成本预测。自预测本身也带来非平凡的延迟和开销，尤其是那些在给出数字前需要进行大量探索的模型，在实时交互场景中缺乏合理性。

4. **预测开销与准确率的非单调关系**。实验显示，投入更多预测 token 并不必然带来更高的预测准确率。第三代半和第四代 Sonnet 的预测开销超过任务成本两倍，却未取得最强相关。这暗示简单的更多思考策略可能并不适用。

5. **未探索的干预方向**。论文主要聚焦于描述性分析和预测能力评估，并未系统测试可通过何种干预手段（如预算截断、上下文压缩、动作空间约束）来降低实际成本。

## 我的笔记

这篇论文的价值在于将智能体经济学从一个抽象概念转化为可量化的实证分析。在工业界部署编码智能体时，成本往往是决定是否采用某款模型的关键因素。本文的数据表明，选择 GPT-5 而非 Kimi-K2 或 Sonnet-4.5，在相同任务上可节省超过百万量级的输入 token，这种差异在规模化部署中将产生巨大的运营成本差异。

从研究方法论角度，本文的控制变量设计值得借鉴：固定智能体框架（OpenHands）和固定数据集（SWE-bench Verified），仅替换底层模型，从而将效率差异归因于模型行为本身。图六在共享成功和共享失败子集上的一致性验证，进一步强化了这一因果推断的可信度。

一个值得延伸的方向是结合本文的阶段级分析与外部干预：如果在 Fix 或 Explore 阶段检测到重复文件查看频率异常升高，是否可以触发早期终止或策略切换？另外，输入 token 的主导地位提示我们，上下文压缩和检索增强可能是降低智能体成本的最有效杠杆，而非输出长度控制。

本文与同期关于多智能体系统 token 分布和推理模型 token 定价的工作形成互补。未来可将自预测任务形式化为一个元学习或在线学习问题，让模型在多次执行同一类任务后逐步改善其成本估计。
