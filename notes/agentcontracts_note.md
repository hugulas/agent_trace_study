## 原文摘要翻译

Contract Net Protocol（1980）首次通过合约机制实现了多智能体系统中的协调。
现代智能体协议标准化了连接性与互操作性，但均未能提供形式化的资源治理规范机制，以限制智能体可以消耗多少资源或可以运行多长时间。
本文提出 Agent Contracts，一个将合约隐喻从任务分配扩展到资源受限执行的形式化框架。
Agent Contract 将输入输出规范、多维资源约束、时间边界和成功准则统一为一个具有显式生命周期语义的连贯治理机制。
针对多智能体协调，本文建立了守恒律，确保委托预算遵守父合约约束，从而通过合约委托实现分层协调。
跨越四项实验的实证验证表明：在迭代工作流中实现90%的令牌削减与525倍的方差降低；在多智能体委托中实现零守恒律违反；通过合约模式实现可度量的质量资源权衡。
Agent Contracts 为可预测、可审计且资源受限的自主 AI 部署提供了形式化基础。

## 创新点

1. **形式化合约元组统一治理维度**。将输入输出规范、资源约束、时间边界和成功准则统一为七元组
   $$C = (I, O, S, R, T, \Phi, \Psi)$$
   该设计使合约从任务分配隐喻升级为资源受限执行的完整治理机制。
   这解决了现有框架只提供连接性而不提供资源治理的系统性缺口。

2. **多智能体守恒律与预算池化**。建立了委托预算必须遵守父合约约束的守恒律，并引入预算池化机制：高效智能体节省的资源可补贴高消耗智能体，在维持总预算纪律的同时提升整体吞吐量。
   这为分层自组织提供了形式化保证。

3. **软硬互补的执行架构**。软执行通过预算感知提示动态反馈剩余资源状态；硬执行通过调用层拦截在运行时强制终止超限行为。
   两层机制互补，既保留了智能体的自主性，又提供了可落地的资源边界。

4. **合约模式操作化满意化权衡**。将时间约束与资源策略显式编码为 URGENT、ECONOMICAL、BALANCED 三种模式，使开发者能够在质量与成本之间做出可度量的权衡，而非依赖事后账单审计。

## 一句话总结

Agent Contracts 通过一个七元组形式化框架与守恒律，将自主智能体的不可预测资源消耗转化为可审计、可边界化的受控执行，并在四项实验中验证了90%的令牌削减与525倍的方差降低。

## 研究问题

2025年末，一个工程团队部署了包含四个专业智能体的多智能体研究系统，其中两个智能体陷入递归澄清循环，在未被检测的情况下持续运行十一天，最终产生47000美元的 API 账单。
该系统没有任何停止条件、预算限制或实时成本监控。
这一事件揭示了一个根本性问题：我们已经构建了具备自主行动能力的 AI 智能体，却缺乏形式化机制来约束其行为边界。

此类失败反映的是系统性缺口，而非实现层面的缺陷。
Gartner 预测到2027年将有超过40%的智能体 AI 项目因成本失控或风险管控不足而被取消。
MIT Sloan 的研究表明35%的组织已经部署智能体 AI，而领导者将监督与自主性之间的紧张关系列为需要集中式治理基础设施的核心挑战。
协议层面，MCP 与 A2A 解决了工具调用与智能体发现问题。
OpenAI Agents SDK、Google ADK 等框架也仅提供连接性与互操作性，却未解决资源治理问题——即智能体可以消耗多少资源、可以运行多长时间。

Agent Contracts 针对这一缺口提出核心问题：
Contract Net Protocol 问的是"谁应该执行这项任务"，而 Agent Contracts 问的是"这项任务必须在什么边界内执行"。
本文借鉴经济学中的合约理论、分布式系统中的协调理论以及实时系统中的资源受限计算，构建面向资源受限自主 AI 的形式化框架。

## 数据与任务定义

本文通过四项实验验证框架，覆盖迭代工作流、多智能体委托、危机通信和合约模式四个场景：

**实验一：迭代工作流中的失控预防（Code Review）**
- 数据集：LiveCodeBench 的70道编程题（2025年2月后发布，超出模型知识截止点），其中31道简单、39道中等难度
- 任务设置：Coder-Reviewer 迭代管道，Coder 编写 Python 代码，Reviewer 使用 test_code 工具执行测试用例并决定批准或请求修订
- 实验设计：被试内设计，每道题分别执行 CONTRACTED（50K 令牌预算，最多3轮迭代）与 UNCONTRACTED（无令牌限制，最多6轮迭代）两个条件
- 评估指标：令牌消耗量、迭代次数、大模型调用次数、成功率、方差

**实验二：多智能体委托中的守恒律验证（Research Pipeline）**
- 任务设置：三智能体研究管道 Researcher → Analyzer → Reporter
- 约束条件：总预算上限 $R_i \leq 100K$，Researcher 的 web_search 工具调用上限为6次
- 评估指标：守恒律违反次数、子智能体预算分配合规性

**实验三：危机通信中的失败预防（Crisis Comm）**
- 数据集：24个时间关键的危机通信场景
- 任务设置：单智能体任务，显式质量阈值 $Q_{min} = 0.80$ 与迭代限制
- 评估指标：令牌消耗量、质量评分、方差

**实验四：合约模式的质量资源权衡（Strategy Modes）**
- 数据集：50个策略任务
- 任务设置：三种合约模式——URGENT（$\tau = 30s$）、ECONOMICAL（$\tau = 60s$）、BALANCED（$\tau = 90s$）
- 评估指标：成功率、推理令牌数、平均耗时、超时率

## 方法主线

Agent Contracts 的核心是一个形式化合约规范，其元组表示为
$$C = (I, O, S, R, T, \Phi, \Psi)$$
该设计将资源治理、时间治理和质量治理统一为单一机制。

### 合约七元组

- **输入规范**：元组表示为 $I = (\sigma_I, \mathcal{V}_I, \mathcal{P}_I)$，
  其中 $\sigma_I$ 为输入模式，$\mathcal{V}_I$ 为验证规则，$\mathcal{P}_I$ 为预处理变换
- **输出规范 $O$**：定义输出格式、质量阈值 $Q_{min}$ 和后置条件
- **成功准则 $S$**：判定任务完成的条件，可以是逻辑谓词（如 all_items_processed）、质量指标（如 accuracy > 0.95）或业务规则
- **资源约束 $R$**：多维资源预算，包括令牌数、API 调用次数、工具调用次数、计算时间和成本
- **时间边界 $T$**：包含截止期限、超时行为和调度策略
- **生命周期语义 $\Phi$**：定义合约从创建、激活、执行、完成到终止的状态转换
- **执行机制 $\Psi$**：软硬两层执行策略

### 守恒律与预算池化

对于多智能体委托，设父合约资源预算为 $R_{parent}$，子智能体集合为 $\{A_1, A_2, ..., A_n\}$，各子智能体分配预算为 $\{R_1, R_2, ..., R_n\}$。
守恒律要求：

$$\sum_{i=1}^{n} R_i \leq R_{parent}$$

预算池化允许未用完的预算在子智能体之间重新分配：若某子智能体实际消耗 $r_i < R_i$，则剩余 $(R_i - r_i)$ 可转入共享池供其他子智能体使用。
这既维持了总预算纪律，又提升了整体吞吐量。

### 软硬执行层

- **软执行**：通过预算感知提示在每次调用前向智能体注入当前资源状态，如 `"Budget: {used}/{total}"`，使智能体自主调整行为策略
- **硬执行**：在调用层拦截，当实际消耗超过分配预算时强制终止，防止失控循环。
  实验二中记录到一个无约束智能体超出40K 令牌预算（消耗56K），硬执行机制成功检测并终止了该失控行为

### 机制流程

Agent Contracts 的核心执行链可概括为以下四个步骤：

1. **输入解析与合约实例化**。接收任务请求后，系统根据任务类型从合约模板库中选取基础合约 $C_{template}$，将用户输入 $x$ 与运行时上下文（可用预算、截止期限、质量要求）绑定为具体合约实例 $C_{instance}$。
   此步骤的输出是一个完全参数化的合约对象，包含所有资源配额、时间边界和成功阈值的数值化设定。

2. **资源监控下的智能体执行**。智能体在运行时环境中执行，软执行层在每次大模型调用前向上下文注入当前资源消耗状态；硬执行层在每次工具调用和大模型调用后检查累计消耗是否超出合约配额。
   若未超限，执行继续；若即将触限，系统触发预算警告；若已超限，硬执行层强制终止当前调用并返回预定义的违约响应。
   此步骤的输入是合约实例和初始任务状态，输出是执行轨迹（调用序列、令牌消耗、中间结果）或终止信号。

3. **成功准则判定与迭代控制**。每轮执行结束后，系统根据成功准则 $S$ 评估当前输出是否满足完成条件。
   若满足，进入步骤四；若不满足且仍有剩余迭代次数与资源预算，将当前状态反馈给智能体进入下一轮迭代；若迭代次数或资源已耗尽，返回当前最优输出或失败标记。
   实验一中合约约束条件的平均迭代次数从无约束条件的3.00降至1.71，说明预算感知促使智能体在更少迭代内收敛。

4. **输出交付与合约终止**。将通过成功准则判定的最终输出 $y$ 返回给调用方，同时触发合约终止流程：释放未使用的预算池资源、记录完整执行日志（消耗明细、调用链、违约事件）用于审计，并将合约状态更新为完成或失败。
   多智能体场景下，父合约在所有子合约终止后聚合各子合约结果并进行最终质量检查。

## 关键结果

四项实验的核心结果如下：

**实验一：迭代工作流失控预防**

| 指标 | UNCONTRACTED | CONTRACTED | 变化 | p 值 |
|------|-------------|-----------|------|------|
| 令牌消耗 | 34,606 | 3,461 | -90% | 0.0007*** |
| 方差 | 5.29B | 10.1M | 525x 降低 | — |
| 迭代次数 | 3.00 | 1.71 | -43% | <0.0001*** |
| 大模型调用次数 | 9.0 | 4.5 | -50% | <0.0001*** |
| 成功率 | 60.0% | 52.9% | -7.1pp | 0.13 (NS) |

合约治理在任务复杂度越高时价值越大：中等难度题目的令牌节省为92%，简单题为76%。
值得注意的是，一个无约束智能体完全失败——陷入评估循环未能提交输出——而对应的合约约束版本成功完成，证明迭代治理不仅提升效率，还能预防智能体失效。

**实验二：多智能体委托守恒律**

50 次三智能体研究管道试验中，守恒律实现100%合规，零次违反。
各子智能体的实际消耗均严格满足总预算约束，web_search 工具调用亦未突破单工具上限。

**实验三：危机通信失败预防**

24 个时间关键场景中，带有显式质量阈值和迭代限制的智能体实现23%的令牌削减（$p = 0.005$），且质量与无约束版本统计等价（$p = 0.32$）。
质量方差比无约束组低26.7倍（$\sigma$: 1.75 vs 9.07）。
排除灾难性异常值后，合约约束组方差仍为无约束组的1.4倍降低（88.5% 贝叶斯概率）。

**实验四：合约模式满意化权衡**

| 模式 | 成功率 | 推理令牌 | 平均耗时 | 超时率 |
|------|--------|---------|---------|--------|
| URGENT | 70% | — | 6.9s | 26% |
| ECONOMICAL | 76% | — | 12.5s | 14% |
| BALANCED | 86% | 1,519 | 16.9s | 10% |

BALANCED 模式以75%更多的令牌投入换取16个百分点的成功率提升（$p \approx 0.05$），证明合约模式能够有效操作化质量与资源之间的显式权衡。

## 深度分析

**方差削减是治理的核心价值，而非均值优化**。
实验一中最引人注意的发现不是成功率的轻微下降（7.1个百分点，且统计不显著），而是方差从5.29B骤降至10.1M——525倍的差距。
这一数字的物理意义是：无约束智能体的资源消耗呈现极度不可预测的厚尾分布，某些任务可能消耗数十倍于平均值的资源；而合约约束将消耗压缩到一个狭窄区间内。
从生产部署角度看，方差削减比均值优化更具经济价值——云服务预算需要按峰值预留，525倍的方差降低意味着基础设施成本的可预测性发生了质变。

**成功率小幅下降的解读**。
合约约束组成功率略低于无约束组（52.9% vs 60.0%，$p = 0.13$），这在表面上似乎是一个负面结果，但结合方差数据后呈现更复杂的图景。
无约束组的更高成功率部分来自无限制迭代带来的"暴力尝试"效应：智能体可以无限次修正直到偶然成功，但代价是令牌消耗失控。
合约约束组的成功率下降可能反映了预算约束下的"诚实失败"——当资源确实不足以解决问题时，系统选择终止而非继续盲目消耗。
中等难度题目获得更高的令牌节省率（92% vs 76%），说明治理价值在需要更多探索的任务中更加显著。

**满意化（satisficing）而非最优化**。
实验四中三种合约模式的设计体现了赫伯特·西蒙的满意化理论：URGENT 模式以最低资源投入换取可接受的低成功率（70%），适用于需要快速响应的场景；BALANCED 模式投入更多资源追求更高成功率（86%），适用于质量敏感场景。
这种显式权衡机制比单一"最优"策略更符合实际生产需求，因为不同业务场景对质量与成本的边际效用曲线截然不同。

**预算感知提示的隐性作用**。
实验设计中合约约束智能体不仅受到硬性资源约束，还接收了动态预算状态提示。
这意味着实验结果实际上是"约束+提示"的联合效应，而非纯约束效应。
从工程实现角度看，这恰恰是该框架的合理设计——形式化约束与感知反馈互为补充。
但从归因分析角度，未来需要消融实验来分离两者的独立贡献。

**守恒律的形式化强度**。
实验二验证了50次零违反，但这一验证规模相对有限。
守恒律的证明依赖于调用层的拦截机制，而当前大模型 API 不支持调用中取消（mid-call cancellation），这意味着如果单个生成请求本身超出预算，硬执行只能在请求完成后拦截，无法防止单次调用的超支。
论文作者明确承认这是需要模型提供商提供 API 级支持才能解决的限制。

## 局限

1. **运行时取消依赖 API 支持**。当前硬执行层只能在调用之间拦截超支行为，无法在单个大模型生成请求内部实时终止。
   若一次生成请求本身消耗超出剩余预算，系统只能在请求完成后才发现违约，无法提供真正的实时预算保证。

2. **合约设计依赖人工预设**。本文的合约参数（如50K 令牌上限、3轮迭代限制、质量阈值0.80）均由开发者手动设定。
   论文指出学习式合约设计（learning-based contract design）是一个重要扩展方向，即让智能体基于历史数据预测预算需求并自动起草子合约。

3. **形式化验证尚未完成**。虽然守恒律在实验中表现良好，但缺乏对合约可满足性和最优预算推导的自动化形式化验证。
   现有资源受限多智能体系统的验证技术可以扩展至合约领域，但本文未提供此类自动推理机制。

4. **实验规模与场景覆盖有限**。四项实验分别使用70、50、50、24个样本，总体规模偏小。
   场景覆盖集中于代码生成、研究管道、危机通信和策略任务，未涉及需要长期持续运行的智能体系统（如论文引言中描述的11天失控场景），也未在真实企业环境中验证。

5. **智能体级别制裁不适用**。与传统多智能体系统中可以对违规智能体施加长期制裁不同，大模型智能体通常是临时的——每个任务实例化、任务完成后丢弃。
   这使得基于信誉或历史行为的惩罚机制难以实施，合约执行只能依赖单次任务的硬性约束。

## 我的笔记

**与现有协议的关系定位**。
Agent Contracts 不应被视为 MCP、A2A 等连接性协议的替代品，而应被看作其上层补充。
MCP 解决"智能体如何调用工具"，A2A 解决"智能体如何发现彼此"，而 Agent Contracts 解决"调用和交互必须遵守什么资源边界"。
Table 1 的对比清晰显示，现有八大框架在令牌限制和可观测性方面大多是原生或部分支持，但在形式化合约治理方面完全缺失。
若 Agent Contracts 能够以中间件形式与这些框架集成，将填补从"能连接"到"受控连接"的关键空白。

**生产落地的关键路径**。
从工程实践角度，该框架的最大价值在于将不可预测的运营成本转化为可预算、可审计的受控支出。
对于计划将智能体系统投入生产的企业，引入 Agent Contracts 意味着：
- 预算团队可以为每个智能体工作流设定明确的成本上限
- 运维团队可以通过合约日志追溯每次失败的资源消耗明细
- 产品团队可以通过切换合约模式（URGENT/ECONOMICAL/BALANCED）实现不同服务等级协议的服务分层

**与合约理论的深层联系**。
论文作者从经济学合约理论中借鉴了三个概念：道德风险（moral hazard，对应智能体的不可预测资源消耗）、不完全合约（incomplete contracts，将成功准则与执行策略分离）和机制设计（mechanism design，通过规范激励期望行为）。
这种跨学科视角是本文区别于纯工程方案的重要特征。
未来可以进一步探索：在预算池化机制中引入机制设计中的激励相容约束，防止智能体故意虚报预算需求。

**值得追踪的扩展方向**。论文在结论中提出的几个扩展方向具有明确的研究价值：
- 人在回路中的合约（human-in-the-loop contracts）：规定何时需要人工审批
- 基于里程碑的治理（milestone-based governance）：支持连续运行的智能体而非单次任务
- 学习式预算预测：从历史执行数据中学习最优合约参数

**复现可行性**。论文提供了开源实现（https://github.com/flyersworder/agent-contracts）和实验数据，且实验使用 LiveCodeBench 这一公开数据集，复现门槛较低。
建议关注框架的应用程序接口设计是否易于与现有智能体编排系统集成，以及硬执行层的拦截机制在不同大模型提供商 API 上的兼容性。
