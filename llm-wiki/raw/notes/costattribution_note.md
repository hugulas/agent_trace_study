## 原文摘要翻译

生产级大语言模型智能体成本归因指南，涵盖按用户、按任务、按租户拆解的消耗模式、token 会计方法，以及面向 2026 年部署场景的代理公司利润保护策略。

核心要点如下：

- 从第一天开始埋点：成本归因只有在请求创建时就打上标签才算可靠。事后从日志中补打标签总会遗漏边界情况，并低估长链路智能体轨迹的真实消耗。
- 四层 token 缺一不可：每个请求都会消耗提示词、工具、记忆和响应四层 token。把它们汇总成单一的输入输出桶会掩盖真实支出去向，导致无法做针对性优化。
- 三维归因视角：按用户、按任务、按租户三种归因维度分别回答不同的产品问题。从一开始就全部建好，才能在后续切换视图时无需重新埋点。
- 缓存计价并不简单：缓存读、缓存写和批量请求的 token 定价各不相同。在缓存密集型负载上，用朴素的 total tokens × 列表单价 计算会把支出高估 40% 以上。
- 先上熔断再补大盘：速率限制、日消费上限和降级兜底能在仪表盘滞后时仍然保护利润。按租户设置的日消费上限是整个技术栈里杠杆率最高的单一控制手段。
- 定价模型决定风险归属：转嫁式、包月式和结果式三种代理公司定价模型对 token 风险的分配方式截然不同。选择与你实际负载可预测性相匹配的模型。

## 创新点

1. **四层 token 分账模型**：将单次智能体请求拆分为提示词、工具、记忆、响应四层分别计量，而非笼统的输入输出汇总。这使得产品团队可以定位膨胀来源（例如工具定义膨胀或记忆层失控），而不会在优化时误伤核心提示词。

2. **三维归因框架**：同时建立按用户、按任务、按租户三条归因维度，并在 OpenTelemetry 跨度中通过统一命名空间（如 digitalapplied.tenant_id）持久化。相比事后从 CloudWatch 日志关联客户记录的做法，这套框架避免了月结对账时的痛苦。

3. **缓存定价的精确会计规则**：系统梳理了 Anthropic 与 OpenAI 在缓存读、缓存写、批量三类 token 上的差异化定价，并指出朴素仪表盘会高估 35%–50% 支出。将缓存命中率作为一级指标监控，是少数能在不改动模型的情况下直接降低单位成本的杠杆。

4. **三层执行防护阶梯**：提出速率限制（设为预期峰值的二至三倍）、日消费上限（设为合同上限的一点五至三倍）、z-score 大于 4 自动熔断的三层递进式自动强制执行，并配套更便宜模型降级、缓存响应服务、配额错误三种优雅降级模式。这是首次将成本控制从报表层推进到实时防护层。

5. **定价模型与归因数据的闭环**：将成本归因数据直接接入客户关系管理和开票系统，使客户经理能看到每个租户的利润而非仅收入；同时给出转嫁式、包月式、结果式三种模型的风险归属矩阵，并推荐大多数代理公司采用混合模式。

## 一句话总结

本文提出了一套面向生产级多租户 LLM 智能体的四层分账、三维归因、三层防护的成本归因框架，强调从第一天埋点、精确缓存计价和按租户熔断是保护利润的关键。

## 研究问题

在多租户 LLM 智能体产品化过程中，团队最常见的失败模式并非计算错误，而是埋点过晚。团队在首发智能体时把成本归因推迟到“有流量之后”，然后花整整一个季度回溯性地将 CloudWatch 日志与客户记录关联，才能搞清楚毛利率为何下降了四个百分点。等到数据可读时，与失控租户之间的定价对话已经变得尴尬。

本文识别出三种几乎涵盖所有生产成本事故的根因模式：

- **均值掩盖分布**：按客户平均成本报告会掩盖长尾分布，即 3% 的租户消耗了 60% 的 token。没有分位数报告，定价团队实际上是在对着虚构数据做优化。
- **重试与工具调用循环未被计入**：在失败工具调用中循环的智能体会烧掉大量 prompt-cached read。如果遥测只统计面向用户的请求，真实消耗速率完全不可见。
- **免费层级缺乏埋点覆盖**：团队对付费客户细致入微地追踪消耗，却把免费层级流量当成一个笼统数字。一个管控不力的免费层级可能在无声中成为最大单项支出。

## 数据与任务定义

本文以一个具体的多租户部署场景作为贯穿全文的实例：

- **场景**：B2B 收件箱分类智能体，部署给 40 个租户，每个租户有 5–50 名用户。
- **功能**：总结新消息、起草回复、路由优先级。
- **月度总 token 支出**：约 \$34,000。
- **合同收入**：\$180,000。
- **聚合毛利率**：81%。

该场景的任务定义本身并不涉及传统意义上的训练数据集或评测基准，而是围绕“如何在真实生产流量中精确追踪、归因、保护和优化 LLM 智能体的 token 支出”这一系统任务展开。评估指标主要是财务指标：按租户毛利率、单次结果成本、缓存命中率、模型组合偏移等。

## 方法主线

本文的方法主线是一个由八个模块组成的生产级成本归因技术栈，可概括为：先分账、再归因、后计价、上报、报告、保护、定价、排错。

### 四层 Token 分账

每个智能体请求消耗的四层 token 必须在编排层拆分，而不是依赖供应商返回的合并输入计数：

- **提示词层**：系统提示词 + 用户输入 + 少样本示例。通常是单次请求中最大的固定成本，也是缓存收益最大的层。
- **工具层**：工具模式定义 + 工具调用结果。随工具数量和返回详细程度增长，是智能体编排层中常见的静默膨胀源。
- **记忆层**：检索增强生成片段 + 对话历史 + 草稿板。随会话长度扩展，需单独追踪以便在不触碰核心提示词的前提下调优检索预算。
- **响应层**：补全输出 + 思考令牌。在大多数供应商处定价为输入的四至五倍，是单位成本最高的层，也最受 effort 参数调控影响。

实际埋点形状为每个跨度一条四列使用记录：prompt_tokens、tool_tokens、memory_tokens、response_tokens。

### 三维归因维度

三层归因维度并行建设，分别回答不同业务问题：

- **按用户归因**：回答“一个账户内部是谁在驱动消耗”。对按席位定价产品和升级对话至关重要。每条请求需携带认证层提供的稳定 user_id。
- **按任务归因**：回答“哪些产品界面最昂贵”。通过 task_id 和 route（如 inbox-triage、summary、report-gen）标记每次智能体运行，供产品团队排定成本优化优先级。
- **按租户归因**：回答“这个客户是否盈利”。是单位经济学、续约对话和分级定价的基础。每个智能体跨度必须携带 tenant_id，无例外。

第四个可选维度按模型叠于其上，标记每个跨度由哪个模型及哪个 effort level 服务，以便在不重跑流量的前提下比较不同模型的有效单次结果成本。

### Prompt Cache 计价

提示词缓存是任何带有稳定系统提示词的智能体负载上杠杆率最高的成本杠杆，但也会破坏朴素仪表盘，因为缓存读和缓存写的定价与标准输入不同：

| Token 类别 | Anthropic 定价 | OpenAI 定价 | 会计规则 |
|---|---|---|---|
| 标准输入 | 列表价（1×） | 列表价（1×） | 基准 |
| 缓存读 | 列表价的 10% | 列表价的 50% | 按折扣价计价，而非列表价 |
| 缓存写 | 列表价的 125% | 列表价的 100% | 每个缓存块一次性写入成本 |
| 批量请求 | 输入输出均 50% 折扣 | 输入输出均 50% 折扣 | 单独标记批量跨度 |

将总 token 数乘以列表输入单价的朴素计算，在缓存密集型负载上会把支出高估 35%–50%。应分别摄取缓存读和缓存写计数器并单独计价。缓存命中率应作为一级指标监控：健康的智能体负载在稳定系统提示词下应达到 70% 以上；低于 40% 通常意味着提示词漂移、不稳定的工具模式定义或缓存生存期问题。

### OpenTelemetry 可观测性埋点

OpenTelemetry GenAI 语义约定提供了一个标准 schema，可被主流可观测性工具及现代数据仓库共同理解。示例跨度属性分为三类：
- 供应商约定字段：gen_ai.system、gen_ai.request.model、gen_ai.usage.input_tokens、gen_ai.usage.cached_read_tokens 等
- 自定义归因标签：digitalapplied.tenant_id、user_id、task_id、route、effort
- 分层会计字段：digitalapplied.prompt_tokens、tool_tokens、memory_tokens

两条值得尽早锁定的约定：一是为自定义属性使用单一命名空间前缀（如 digitalapplied.*），以便在任何工具中干净过滤；二是始终上报原始 token 计数器而非预计算成本。定价会变，计数器不会。查询时从价格表重新推导成本，总比供应商更新列表价后重新摄取整整一季度的跨度要便宜。

### 三层执行防护阶梯

报表捕捉漂移，规则约束灾难性损失。每个多租户智能体产品在向首位付费客户交付之前就需要三层自动强制执行：

1. **按租户按分钟的速率限制**：约束突发循环。设置为预期峰值的 2–3 倍。返回明确的 429 与重试等待时间，而非静默排队。
2. **按租户日消费上限**：即使仪表盘滞后也能约束最坏损失。设置为合同上限的 1.5–3 倍。触发后自动收紧速率限制并向客户经理告警。
3. **z-score 大于 4 的熔断开关**：自动暂停租户并呼叫值班人员。仅保留给明显异常行为；若基线窗口至少 7 天，误报率很低。

当租户触及上限时，三种优雅降级模式效果良好：

- **更便宜模型兜底**：日消费接近上限时从 Opus 降级到 Sonnet（或同级模型），通常可将单次请求成本降低 4–5 倍，而质量损失在大多数负载上可接受。
- **缓存响应服务**：对于读密集型查询，直接服务最近的缓存答案而非重新调用模型。在稳定文档上的摘要和问答场景尤其有效。
- **显式配额错误**：对于质量不可妥协的负载，返回结构化的配额超限错误并附带升级 tier 的上下文。比静默降级更清晰，且能推动 tier 升级对话。

### 代理公司定价模型

对于向客户交付 AI 能力的代理公司，归因数据是定价对话的基础。三种主导定价模型对 token 风险的分配方式不同：

| 模型 | 风险承担方 | 最佳适用场景 | 利润形态 |
|---|---|---|---|
| 转嫁式 | 客户 | 实验性/无界负载 | 固定加成 10–30% |
| 包月式 | 代理公司 | 可预测的生产负载 | 随规模变动的可变利润 |
| 结果式 | 代理公司 | 可量化结果（工单、线索） | 高方差、高上限 |

大多数被建议的代理公司采用混合模式：生产功能用包月制以维持稳定的单次结果成本，实验性负载用转嫁式以覆盖活跃的提示词工程，仅在结果能在客户现有系统中干净量化时才用结果式。

无论选择哪种模型，按租户按路由归因数据都是续约谈判的锚点。

### 异常调试信号

当成本告警触发时，调查路径遵循固定的五个检测信号，几乎覆盖所有生产异常：

- **工具调用循环**：同一工具在单条 trace 内被以几乎相同的参数调用 5 次以上。通常是智能体循环中的收敛缺陷。修复：添加循环检测器，在 N 次相同调用后终止。
- **提示词膨胀**：同一路由的提示词 token 数量周环比增长 10% 以上。通常由累积的示例或工具定义导致。修复：剪枝提示词或拆分为缓存模板。
- **记忆失控**：对话历史超出会话预期。通常缺少轮次限制或摘要步骤。修复：实现滚动摘要或按会话 token 预算。
- **缓存未命中风暴**：缓存命中率从 70% 以上骤降至 40% 以下。通常由使缓存块失效的提示词变更导致。修复：审计最近的提示词提交并对齐缓存边界。
- **模型升级漂移**：流量悄悄从 Sonnet 切换到 Opus。通常是配置或降级缺陷。修复：将模型组合变化量作为一级指标告警。

### 机制流程

整个成本归因与保护框架的核心执行链可归纳为以下四步：

1. 输入携带标签的智能体请求，提取四层 token 使用量，输出带归因的逐跨度记录。
2. 输入原始 token 计数与缓存标志，查询供应商折扣价，更新标准化 OpenTelemetry 跨度，输出后端存储记录。
3. 输入聚合跨度数据，压缩计算各租户支出与命中率，输出三层报告与异常告警。
4. 输入异常信号与支出轨迹，对齐递进式强制执行规则，输出受保护响应或配额错误。

## 关键结果

多租户实例揭示了聚合指标为何具有误导性：

| 租户分段 | 数量 | 收入 | Token 支出 | 毛利率 |
|---|---|---|---|---|
| 健康（包月 tier） | 34 | \$145,000 | \$14,600 | 90% |
| 高消耗（应升级 tier） | 4 | \$22,000 | \$11,400 | 48% |
| 失控（集成中的循环缺陷） | 2 | \$13,000 | \$8,000 | 38% |

- 聚合 81% 的毛利率真实但无意义。
- 40 个租户中的 6 个（即 15% 的客户群）消耗了 57% 的 token 支出，且毛利率远低于组合平均水平。
- 在没有归因的情况下，产品团队会优化错误的提示词；有了归因，客户经理就能获得升级对话的具体清单和循环缺陷的修复优先级。

## 深度分析

**缓存计价的精确性要求**：本文对缓存定价误差的量化（35%–50% 高估）是一个容易被忽略但影响巨大的细节。在缓存密集型负载上，仪表盘与供应商发票之间的差异往往不是因为缺陷，而是因为会计规则不同。将缓存读和缓存写作为独立计数器摄取并分别应用折扣与溢价，是把成本归因从“近似估算”提升到“可审计精确度”的关键步骤。这也解释了为什么 OpenTelemetry GenAI 约定中专门保留了 gen_ai.usage.cached_read_tokens 和 gen_ai.usage.cached_write_tokens 两个字段。

**三层强制执行的杠杆差异**：在利润保护的三层阶梯中，按租户日消费上限被作者称为“整个技术栈里杠杆率最高的单一控制手段”。这是因为仪表盘天然滞后（无论是数据摄取延迟还是查询延迟），而实时上限能在消耗发生时就截断损失。相比之下，速率限制主要约束突发，熔断开关主要处理异常。三者的组合形成了一个从“常态约束”到“异常熔断”的完整光谱。

**定价模型的风险转移机制**：转嫁式将 token 价格波动风险完全转移给客户，适合实验性负载；包月式将风险留在代理公司，但通过规模效应和稳定负载获得可变利润；结果式则把风险与回报都放大，只有在客户系统已能量化结果时才可行。混合模式的推荐是务实的——它承认不同负载形态需要不同的风险分配机制，而不是用单一模型覆盖所有场景。

**优雅降级的商业意义**：三种降级模式中，更便宜模型降级和缓存响应服务属于“静默降级”，客户可能感知到质量轻微下降但不会收到错误；而显式配额错误则是“显性降级”，虽然制造了摩擦，但恰恰能驱动升级对话。这种设计把成本保护从纯技术措施转化为商业增长工具。

**OpenTelemetry 约定的可移植性价值**：采用标准 schema 的最大收益不是当下，而是未来。当团队在不同可观测性工具之间迁移（例如从 Datadog 到 Honeycomb，或从 Langfuse 到自研数据仓库），自定义命名空间前缀加上标准 GenAI 字段，使得过滤和迁移成本几乎为零。这也是“原始计数器优于预计算成本”原则的长期回报——价格表更新的复杂度为常数，而重新摄取历史跨度的复杂度与数据量成正比。

## 局限

1. **缺乏形式化实验与对照**：本文是一篇生产指南而非学术论文，所有主张均来自作者团队在多租户部署中的审计经验，没有提供对照实验或统计显著性检验。例如“3% 租户消耗 60% token”和“15% 租户消耗 57% 支出”等数字来自个案审计，不能推广到所有负载形态。

2. **定价数据高度依赖具体厂商**：缓存定价表基于 Anthropic 和 OpenAI 在 2026 年 4 月的公开列表价。供应商定价策略变化频繁（文中也提到 Anthropic 将在 2026 年 6 月 15 日对 Claude credits 进行重大调整），因此具体百分比（10%、50%、125%）可能很快过时。读者需要建立价格表的定期更新机制，而不是照搬数字。

3. **单一负载的局限性**：实例仅覆盖 B2B 收件箱分类这一类读密集型、会话型智能体。对于代码生成、批量文档处理、多模态等完全不同消耗特征的负载，四层分账的相对重要性可能不同（例如代码生成中响应层的四至五倍定价影响会被进一步放大）。

4. **未涉及安全与隐私权衡**：在按用户和按租户归因中，跨度携带了大量身份标签（user_id、tenant_id）。本文没有讨论这些标签在日志保留、合规（GDPR、SOC2）和访问控制方面的额外复杂度。生产环境中，归因粒度与隐私合规之间存在张力。

5. **缺乏自动化埋点方案**：本文假设团队有能力在编排层手动拆分四层 token 并注入 OpenTelemetry 跨度。对于使用现成框架（如 LangChain、LlamaIndex）的团队，如何在不大量修改框架代码的前提下实现同样精度的分账，文中未给出具体方案。

## 我的笔记

这篇指南的价值在于它把成本归因从“财务后台的报表工作”重新定义为“产品基础设施的一部分”。在 LLM 智能体的生产化浪潮中，一个常见的认知误区是：先交付功能，再补监控。本文用大量真实案例说明，这种顺序在按 token 计费的模式下会导致不可挽回的利润损失。

从平台经济学角度看，本文揭示的“15% 租户消耗 57% 支出”与 SaaS 经济学中的“20% 客户贡献 80% 收入”形成有趣的对照：在 LLM 智能体场景中，长尾效应在成本侧比收入侧更为极端。这意味着传统的按席位或按 tier 定价模型可能系统性低估高消耗租户的真实成本，而按租户按路由归因数据是修正这种定价扭曲的唯一手段。

与相关工作的联系：本文的方法与 OpenTelemetry 社区正在推进的 GenAI semantic conventions 高度对齐，可视为该标准在代理公司场景下的具体落地手册。与学术界的 LLM 服务优化工作（如 vLLM、投机解码）相比，本文关注的不是单次推理的延迟或吞吐，而是跨租户、跨时间、跨业务维度的成本可见性。两者互补：服务优化降低单位成本，成本归因确保优化收益真正落入利润而非被隐性消耗吞噬。

对于实际落地的团队，我建议将本文的三层强制执行阶梯与现有的值班手册结合：把日消费上限突破和 z-score 大于 4 直接接入 PagerDuty 或 Opsgenie，而不是仅发到 Slack 频道。因为当异常发生时，仪表盘的滞后性意味着人工介入的响应时间可能已不足以阻止当日损失。

最后，关于缓存命中率的监控：文中建议的 70% 健康阈值是一个有用的经验法则，但更严谨的做法是为每个路由建立独立基线。不同路由的系统提示词稳定性差异巨大（例如收件箱分类的提示词相对稳定，而报告生成的提示词可能因模板变化频繁失效），全局阈值可能掩盖局部问题。
