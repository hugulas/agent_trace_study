## 内容摘要

这篇指南由 Arize AI 发布，系统性地将 OWASP Agentic Security Initiative（ASI）提出的十大 Agent 安全风险映射到具体的可观测性（observability）功能与工程实践上。

OWASP ASI 是 Open Web Application Security Project（OWASP）生成式 AI 安全项目下的专项 initiative，专门针对具备自主决策能力的 Agentic 应用定义安全威胁模型。

本文不仅列举了十大风险，更重要的是为每个风险提供了基于 Arize AX 平台的检测、监控与缓解方案，使其成为一份可直接落地的合规操作手册。

指南首先明确了 Agentic 应用与传统 Web 应用在安全范式上的根本差异。

传统应用的安全边界相对清晰，输入输出经过严格的验证和过滤即可大幅降低风险。

而 Agentic 应用由于具备自主规划、工具调用和记忆持久化等能力，攻击面呈指数级扩大。

恶意输入不仅可能直接篡改单次响应，更可能通过提示注入（prompt injection）或数据投毒（data poisoning）劫持 Agent 的目标（goal hijack），使其在长时间自主运行中持续执行攻击者意图。

这种目标层面的劫持远比传统 SQL 注入或 XSS 更具破坏性，因为 Agent 可能在多个会话、多轮交互中逐步偏离预设行为，而传统的输入校验机制对此几乎无能为力。

基于这一背景，指南将十大风险分为三个层次。

第一层围绕 Agent 的输入与意图完整性，包括 ASI01（Agent Goal Hijack）、ASI05（Unexpected Code Execution）和 ASI06（Memory & Context Poisoning）。

第二层围绕 Agent 与外部世界的交互边界，包括 ASI02（Tool Misuse & Exploitation）、ASI03（Identity & Privilege Abuse）、ASI04（Agentic Supply Chain Vulnerabilities）和 ASI07（Insecure Inter-Agent Communication）。

第三层围绕系统级韧性与人机协作，包括 ASI08（Cascading Failures）、ASI09（Human-Agent Trust Exploitation）和 ASI10（Rogue Agents）。

在具体防御措施上，指南反复强调 Tracing（全链路追踪）、Evals（自动评估）、Monitors（实时监控）和 Prompt Hub（提示模板管理）四类能力的组合运用。

例如，针对 ASI01 的目标劫持，需要同时部署提示注入检测评估器、会话级评估（session-level evals）以发现跨多轮对话的越狱尝试、响应一致性监控以及上下文质量验证。

针对 ASI04 的供应链漏洞，则要求追踪所有动态加载的组件（插件、MCP 服务器、模型），记录组件来源、版本和初始化参数，并通过行为验证和签名校验确保组件完整性。

这种 "追踪 + 评估 + 监控 + 治理" 的四维防御体系贯穿全文，体现了 Arize 将可观测性视为 Agent 安全基础设施核心支柱的产品理念。

值得注意的是，指南还多次提及 Human-in-the-Loop（人类在环）机制的重要性，特别是在高影响操作前的审批环节。

这反映了一个务实观点：在当前的 AI 安全水平下，完全自主的 Agent 尚不具备足够的自我约束能力，人机协同审批仍是防止灾难性后果的最后一道防线。

同时，指南对 Prompt Hub 的强调也颇具深意——提示模板的版本控制、签名认证和审批工作流不仅是工程管理需求，更是防止提示层供应链攻击的关键控制点。

## 关键要点

1. **ASI01 Agent Goal Hijack（目标劫持）**

   恶意输入通过提示注入或数据投毒操纵 Agent 的目标，使其偏离预设任务并执行攻击者指令。

   防御需要全链路追踪（tracing）以洞察每一步的输入输出。

   部署提示注入检测评估器对所有用户提示和系统指令进行恶意模式扫描。

   使用会话级评估（session-level evals）识别多轮对话中的渐进式越狱尝试。

   监控响应是否偏离预期主题或行为基线。

   验证检索增强生成（RAG）所获取数据的质量与相关性。

   同时，护栏（guardrails）的触发事件和人类在环（human-in-the-loop）审批记录也应纳入追踪范围，以形成完整的审计链条。

2. **ASI02 Tool Misuse & Exploitation（工具滥用与利用）**

   Agent 可能因目标混淆或恶意诱导而误用合法工具，或被诱导调用具有危险权限的工具。

   防御策略包括：在 Prompt Playground 中构建数据集，持续测试 Agent 正确选择和使用工具的能力。

   收集失败案例形成回归测试集。

   利用追踪数据回溯每次工具调用的完整上下文，包括调用参数、返回结果和决策依据。

   评估工具调用序列的合理性，识别异常的工具组合模式。

3. **ASI03 Identity & Privilege Abuse（身份与权限滥用）**

   跨 Agent 工作流中的身份冒用、会话劫持和权限继承失控可能导致未授权访问。

   应通过用户追踪（user tracking）和会话追踪（session tracking）识别用户身份。

   监控 Agent 身份上下文跨会话的延续性。

   追踪多 Agent 工作流中的凭证继承路径。

   特别是在一个 Agent 将部分任务委托给另一个 Agent 的场景中，权限的传递和衰减必须被精确记录和审计。

4. **ASI04 Agentic Supply Chain Vulnerabilities（供应链漏洞）**

   恶意或受损的工具、插件、MCP 服务器、模型或提示模板在运行时引入隐藏指令和后门。

   防御措施包括：记录所有动态加载组件的来源、版本和初始化参数。

   监控运行时加载模式以发现异常组件添加。

   通过组件行为验证、输出完整性检查和清单签名验证确保组件可信。

   对提示模板实施版本控制、签名认证和审批工作流。

   为组件更新建立回归测试数据集。

   随着 MCP（Model Context Protocol）生态的扩张，第三方组件的运行时审计正成为 Agent 安全的新前沿。

5. **ASI05 Unexpected Code Execution（意外代码执行）**

   具备代码生成或执行能力的 Agent 可能在提示被操纵时运行恶意代码，尤其是在非沙箱环境中。

   需要记录所有代码生成和执行事件。

   捕获导致代码生成的原始输入提示。

   追踪执行环境元数据（沙箱状态、权限级别、运行时类型）。

   评估器应检测危险代码模式（如 shell 命令注入、未经授权的文件操作、异常网络调用）。

   监控器应对非沙箱环境中的代码执行、提权行为和异常文件访问发出实时告警。

   仪表盘应可视化代码执行的成功/失败率和沙箱策略违规事件。

6. **ASI06 Memory & Context Poisoning（记忆与上下文投毒）**

   攻击者通过污染 Agent 的持久记忆、RAG 存储、嵌入向量或上下文知识，使其在长期运行中持续产生有害输出。

   防御手段包括：使用嵌入聚类可视化识别异常或投毒条目。

   测量嵌入漂移（embedding drift）以检测投毒事件的时间窗口。

   对异常记忆写入模式、低信任度条目的被访问频率设置告警。

   维护记忆验证的黄金数据集（golden datasets）和投毒检测测试用例。

   对于依赖长期记忆的 Agent，记忆的完整性审计应与代码审计享有同等优先级。

7. **ASI07 Insecure Inter-Agent Communication（不安全的 Agent 间通信）**

   多 Agent 系统中的通信可能遭受 spoofing、中间人攻击或消息篡改。

   应记录所有 Agent 间消息的完整元数据（发送方 ID、接收方 ID、时间戳、通信通道、数字签名等）。

   利用 Agent 图可视化（agent graph visualization）理解复杂的交互拓扑。

   评估器需扫描跨 Agent 通信中的越狱内容。

   检测敏感数据在 Agent 间的不当传递。

   分析多 Agent 工作流中的通信异常和协作模式偏差。

   确保消息格式符合预期的 schema 规范。

8. **ASI08 Cascading Failures（级联故障）**

   单个 Agent 的微小失误或局部故障可能通过多 Agent 工作流的依赖关系传播，最终放大为系统性影响。

   防御需要建立端到端的追踪覆盖，识别单点故障如何在 Agent 协作网络中扩散。

   通过评估器检测异常的工具调用链路和循环依赖。

   对于关键业务流程，应设计断路器（circuit breaker）和优雅降级机制，防止局部故障演变为全局中断。

9. **ASI09 Human-Agent Trust Exploitation（人机信任利用）**

   Agent 可能利用人类用户的自动化偏见（automation bias）和过度信任，通过操纵性语言或社会工程学手段诱导用户做出有害决策。

   应记录 Agent 建议与人类批准/拒绝决策的完整对应关系。

   评估建议的上下文适配性和风险对齐度。

   检测操纵性语言模式或社交工程尝试。

   为高风险决策建立人工标注（annotations）和标签工作流（labeling workflows）。

   确保关键决策始终留有可审计的人工判断记录。

10. **ASI10 Rogue Agents（失控 Agent）**

    被入侵、目标错位或模型行为漂移的 Agent 可能持续偏离预期行为基线。

    需要通过追踪和评估持续监控 Agent 行为模式。

    对异常行为指标设置告警阈值。

    建立快速检测和隔离失控实例的响应流程。

    Rogue Agent 的检测难点在于其偏离可能是渐进式的，因此需要基于历史基线的统计异常检测，而非简单的规则匹配。

## 与综述的关联

这篇指南为综述中 "Agent 安全与风险治理" 相关章节提供了来自工业界一线厂商的系统性框架。

OWASP ASI Top 10 是目前为数不多的专门针对 Agentic 应用安全威胁进行体系化分类的公开标准，其权威性和实用性在业界具有较高认可度。

综述在讨论 Agent 系统的安全架构时，可直接引用 ASI 的十大风险分类作为威胁建模的基础框架，并在此基础上展开对各类风险的学术研究和工业实践综述。

从可观测性视角看，本文的核心价值在于将抽象的安全风险映射为具体的可观测性操作。

追踪用于还原攻击链路和时间线。

评估用于自动检测威胁模式和异常行为。

监控用于实时告警和快速响应。

提示中心用于治理输入模板的版本和审批。

这种 "风险 → 能力 → 工具" 的映射逻辑，与综述中 "Agent 可观测性支撑错误诊断与安全审计" 的主题高度契合。

综述若讨论 Agent 系统的纵深防御体系，可将本文作为 "观测层防御" 的典型参考，说明可观测性不仅是性能调优工具，更是安全治理的基础设施。

特别是在供应链安全（ASI04）方面，随着 MCP（Model Context Protocol）服务器和第三方插件在 Agent 生态中的快速普及，组件来源验证和运行时行为监控正成为新的安全热点。

本文对供应链威胁的详细描述和防御建议，可为综述中关于 Agent 生态安全性的讨论提供重要素材。

供应链攻击在传统软件领域已被证明极具破坏力（如 SolarWinds 事件）。

而在 Agent 领域，第三方组件可以在运行时被动态加载并立即获得执行权限，这种即时性使得传统软件供应链的安全实践需要被重新设计。

此外，人机信任利用（ASI09）是一个相对新颖但日益重要的议题。

随着 Agent 越来越多地参与决策建议（如代码审查、投资决策、医疗辅助、法律分析），人类用户可能因自动化偏见而过度信任 Agent 输出，甚至在 Agent 给出明显错误建议时仍选择遵从。

本文对建议质量评估和操纵性语言检测的讨论，与综述中 "Agent 可解释性与人类监督" 主题形成呼应。

提示综述在讨论人机协作时应纳入 "Agent 对人类行为的反向影响" 这一维度。

在级联故障（ASI08）方面，本文的简短提及实际上触及了一个深度技术问题：多 Agent 系统的故障传播动力学。

与微服务架构中的级联故障不同，Agent 之间的故障传播不仅涉及调用链的超时和重试，还可能涉及目标污染（一个 Agent 的错误输出成为另一个 Agent 的输入上下文）和共识偏离（多个 Agent 基于错误前提达成一致的错论）。

综述在讨论多 Agent 系统可靠性时，可以借鉴本文的分类，将级联故障进一步细分为数据级、目标级和行为级三个传播层次。

本文对记忆投毒（ASI06）的讨论也为综述中 Agent 记忆机制的安全性分析提供了切入点。

当前大量 Agent 框架依赖向量数据库存储长期记忆，而这些记忆库的可审计性和完整性验证机制尚不成熟。

综述若涉及 Agent 的长期记忆和知识更新机制，可以补充讨论记忆存储的安全边界和投毒检测方法。

## 我的笔记

这篇指南的实用性在于它跳出了纯理论威胁模型的范畴，将 OWASP 的安全框架与 Arize 的可观测性产品能力进行了深度结合。

对于正在构建生产级 Agent 系统的团队而言，这种 "风险清单 + 操作手册" 的形式比学术论文更易落地，也更容易被安全团队和工程团队同时理解和采纳。

十大风险中，我认为 ASI01（目标劫持）和 ASI04（供应链漏洞）是当前最具紧迫性的两个。

目标劫持的攻击门槛相对较低——一个精心构造的提示注入即可让 Agent 在后续多轮交互中持续偏离目标。

而现有的基于规则的内容过滤器对此类跨会话攻击的防御效果有限。

会话级评估（session-level evals）虽然被本文视为高级防御手段，但其实现难度和计算成本不容忽视。

如何在安全覆盖率和系统开销之间取得平衡，仍是一个开放问题。

供应链漏洞则因 MCP 生态的快速扩张而日益严峻。

当 Agent 可以动态加载来自不可信来源的 MCP 服务器时，传统的静态代码审计已无法覆盖运行时的组件风险。

本文提出的组件签名验证和行为验证是有效的方向，但在一个去中心化的插件市场中，如何建立可信的组件发行和验证体系，类似于移动应用商店的审核机制，可能是 Agent 生态下一步需要解决的基础设施问题。

不过，本文也存在明显的厂商导向色彩。

几乎所有防御建议都指向 Arize AX 平台的具体功能（tracing、evals、monitors、prompt hub）。

虽然这增强了可操作性，但也可能让读者误以为只有采用 Arize 才能实现同等水平的防护。

实际上，许多防御原则（如提示模板版本控制、组件签名验证、会话级评估）是平台无关的。

开源社区正在涌现类似的工具链（如 OpenTelemetry 的 Agent 语义约定、开源评估框架 Promptfoo、Guardrails AI 等）。

阅读时应将本文的方法论与具体产品实现解耦，提取可迁移的工程实践。

另一个值得深思的点是，指南虽然罗列了十大风险，但对风险之间的相互作用着墨不多。

在真实攻击场景中，这些风险很少孤立出现，而是形成复合攻击链。

例如，ASI04 的供应链漏洞可能导致 ASI06 的记忆投毒（恶意组件在运行时污染 Agent 记忆）。

ASI07 的不安全通信可能加速 ASI08 的级联故障（错误消息在 Agent 间快速传播）。

未来的工作或许需要构建跨风险的关联分析框架，帮助防御者识别攻击链路的早期信号。

例如，当 "异常组件加载" 与 "嵌入漂移" 同时出现时，应触发更高优先级的安全告警。

最后，ASI09（人机信任利用）让我意识到 Agent 安全研究的一个盲区。

现有文献大量关注如何让 Agent 更可靠，却很少讨论 Agent 的可靠性对人类判断的侵蚀效应。

当 Agent 在 95% 的场景中都给出正确建议时，人类监督者在剩余的 5% 场景中可能因过度信任而丧失批判性。

这种 "可靠性悖论" 可能是高自主度 Agent 系统中最隐蔽的安全风险。

综述在讨论人机协作与监督机制时，应警惕这种自动化偏见的反向作用。
