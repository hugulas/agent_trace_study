## 原文摘要翻译

端云协同推理对基于大语言模型的边缘设备至关重要，因为端侧模型往往缺乏所需的推理能力，而纯云端推理在严格的延迟和令牌预算约束下可能成本高昂且速度缓慢。然而，现有的端云协作方法通常基于对输入任务难度的估计进行路由。这些静态的粗粒度启发式策略忽视了子任务之间的依赖关系，错失了并行执行和预算自适应路由的机会。为此，我们提出了 HybridFlow，一种资源自适应的端云推理框架，支持相互依赖的子任务并行执行。具体而言，我们为每个输入任务构建一个依赖感知的有向无环图，在其依赖关系满足后并发执行子任务，从而降低端到端延迟。此外，我们提出了一个动态的效益成本效用模型，实时优化准确率、令牌成本与延迟之间的权衡。这种动态路由在保持推理质量的同时，最小化了不必要的云端调用。在 GPQA、MMLU-Pro、AIME24 和 LiveBench-Reasoning 上的实验表明，HybridFlow 改善了成本与准确率之间的权衡，在保持有竞争力准确率的同时，降低了延迟和云端接口调用开销。

## 创新点

1. **依赖感知的子任务有向无环图分解与并行执行**：现有端云协作方法多在查询级别或固定推理步粒度上做路由，HybridFlow 将复杂查询分解为带显式依赖边的子任务图，解锁逻辑上可并行的执行窗口，从而显著压缩端到端 wall-clock 时间。

2. **效益成本驱动的子任务级动态路由**：论文提出以子任务为单位的质量增益与资源消耗联合建模，通过轻量级多层感知机离线估计效用，并在线结合双阈值策略实时适应预算消耗状态。这相比固定阈值或纯启发式策略，能在准确率、延迟和接口成本之间取得更优的帕累托前沿。

3. **在线预算自适应的双阈值机制**：路由阈值随已消耗的接口令牌量和延迟动态调整，使得系统在查询执行过程中根据剩余预算自动收紧或放宽云端卸载策略，而不是在查询开始时一次性决定。

4. **端到端的规划器质量评估体系**：论文不仅提出了分解框架，还系统评估了规划器在逻辑正确性、事实准确性、清晰度、机器可用性和结构一致性五个维度的表现，并量化了并行化带来的步数压缩收益。

## 一句话总结

HybridFlow 通过将复杂查询分解为依赖感知的子任务图，并基于效益成本效用模型进行端云自适应路由与并行执行，在四个推理基准上实现了优于现有端云协作基线的准确率效率权衡。

## 研究问题

大语言模型在多步推理和复杂决策任务上表现优异，但其云端部署面临高延迟、大内存占用和不可忽视的接口费用；纯端侧小模型又因容量受限而在深度推理任务上表现不佳。端云协作是一种自然的折中方案，但现有方法存在两个核心缺陷：

- **粗粒度路由错失并行机会**：现有工作通常在查询级别或固定推理步粒度上做路由，无法利用复杂查询内部自然存在的子任务并行性，导致执行基本按顺序进行。
- **静态策略无法适应时变预算**：实际部署中网络延迟、接口预算和端侧负载均随时间波动，而现有方法多采用固定启发式或静态阈值，不能在线适应实时预算状态。

因此，论文要回答的核心问题是：如何设计一个端云协作框架，能够以细粒度进行自适应、预算感知的调度与路由，从而在复杂推理任务上实现快速且令牌高效的推理？

## 方法主线

HybridFlow 整体是一个两阶段的端云协同推理流水线：首先由端侧规划器将用户查询分解为带依赖的子任务图，然后由资源感知路由器根据实时预算状态将各子任务自适应分配给端侧小模型或云端大模型，并调度可并行的子任务同时执行。

**模型部署配置**：
- 端侧：Llama3.2-3B，同时承担规划器角色和边缘执行器角色
- 云端：GPT-4.1，通过接口调用
- 子任务编码器：qwen3-embedding-0.6b
- 路由器：两层隐藏层的轻量级多层感知机，以 AdamW 优化器离线热启动，学习率 $1 \times 10^{-4}$

**核心建模**：

对于每个子任务 $t_i$，定义二元路由变量 $r_i \in \{0, 1\}$，其中 $r_i = 1$ 表示卸载到云端执行，$r_i = 0$ 表示在端侧执行。

归一化成本 $c_i$ 综合延迟增量和接口令牌增量：
$$c_i = \text{clip}\left[\frac{1}{2}\left(\frac{\Delta l_i}{l_i^{m}} + \frac{\Delta k_i}{k_i^{m}}\right), 0, 1\right] \in [0, 1]$$

其中 $l_i^{m}$ 和 $k_i^{m}$ 为每子任务的最大延迟和最大接口令牌。

效用分数 $u_i$ 衡量质量增益与资源消耗的比值：
$$u_i = \text{clip}\left[\frac{\Delta q_i}{c_i} + \varepsilon, 0, 1\right]$$

其中 $\Delta q_i$ 为云端执行相比端侧执行的期望质量提升，$\varepsilon > 0$ 为小常数。

路由决策在满足总预算约束下最大化总效用：
$$\max_{\mathbf{r}} \sum_{i=1}^{n} r_i \Delta q_i \quad \text{s.t.} \quad \sum_{i=1}^{n} r_i c_i \leq C_{m}$$

该问题可视为零一背包问题，最优解可通过动态规划得到，但在推理时论文采用学习到的轻量级多层感知机进行近似估计，以避免在线枚举。

在线阶段，自适应阈值 $t$ 随已消耗资源更新：
$$t = \text{clip}\left[\tau_0 + \frac{k_u}{2K_m} + \frac{l_u}{2L_m}, 0, 1\right]$$

其中 $k_u$ 和 $l_u$ 分别为已累积的接口令牌和延迟，$K_m$ 和 $L_m$ 为查询级预算上限。

### 机制流程

1. **输入查询分解与依赖图构建**
   - 输入：用户查询 $Q$
   - 操作：端侧规划器 $M_P$（Llama3.2-3B）将查询分解为子任务集合 $T$ 和依赖边集合 $E$，生成任务级有向无环图 $G = (T, E)$；随后通过确定性解析和验证算法检查图的有效性（无环、可达、约束满足），若验证失败则在最多 2 轮内修复，否则回退为顺序链式计划
   - 输出：有效的子任务有向无环图及可执行队列

2. **子任务特征编码与效用离线估计**
   - 输入：待路由子任务 $t_i$
   - 操作：使用 qwen3-embedding-0.6b 将子任务编码为语义嵌入 $\mathbf{z}_i$，输入轻量级两层多层感知机路由器，回归输出离线效用估计 $\hat{u}_i$；该路由器通过离线剖析数据以 AdamW 训练，拟合剖析得到的目标效用
   - 输出：每个子任务的效用分数估计 $\hat{u}_i$

3. **在线自适应路由与预算感知分配**
   - 输入：就绪子任务队列、当前预算消耗状态、双阈值 $t$
   - 操作：对每个就绪子任务，比较其效用估计与当前自适应阈值；若 $u_i > t$ 则分配给云端大模型 $M_c$（GPT-4.1），否则分配给端侧小模型 $M_e$（Llama3.2-3B）；同时根据部分反馈通过上下文赌博机策略在线校准路由头，并更新阈值 $t$ 以反映剩余预算
   - 输出：每个子任务的路由决策 $r_i$ 和执行位置

4. **依赖感知并行执行与结果汇总**
   - 输入：已分配的子任务及其有向无环图依赖关系
   - 操作：执行器按照依赖关系调度子任务，所有前置依赖已完成的子任务可并发执行；端侧和云端分别处理各自分配的子任务，完成后将子任务结果按图结构传递至下游节点；最终汇总根节点输出得到对用户查询的完整响应
   - 输出：最终回答 $R$

## 关键结果

**准确率表现（Table 1）**：
- CoT（GPT-4.1）在所有非直接提示方法中取得最高平均准确率 58.99%
- HybridFlow 平均准确率为 55.34%，紧随其后，优于 SoT（52.90%）和 PASTA（45.56%）等单模型分解方法，以及协作基线 HybridLLM（38.70%）和 DoT（46.50%）
- 纯端侧 CoT（Llama3.2-3B）仅 19.59%，纯云端 Direct Prompt（GPT-4.1）为 53.33%

**效率表现（Table 2）**：
- HybridFlow 平均端到端延迟 $C_{time} = 17.48$ 秒，显著优于 HybridLLM（24.45 秒），甚至优于受顺序约束的 DoT（18.32 秒）
- 在接口成本 $C_{API}$ 上，HybridFlow 同样优于协作基线
- 纯云端 CoT 延迟 18.26 秒，接口成本 $C_{API} = 0.0185$
- 纯端侧执行虽然消除接口成本，但准确率仅 25.54%

**路由策略消融（Table 3）**：
- 全部端侧执行：准确率 25.54%，无接口成本
- 全部云端执行：准确率 57.28%，延迟 18.26 秒，$C_{API} = 0.0185$
- 随机路由：准确率 46.00%，$C_{API} = 0.0075$
- 固定阈值：准确率 51.62%
- HybridFlow-Chain（禁用有向无环图并行，顺序执行）：用于隔离路由本身与并行调度的贡献
- 完整 HybridFlow 取得最优的准确率成本权衡和最高统一效用

**固定阈值趋势（Figure 4 / Table 6）**：
- 随着固定卸载阈值 $\tau_0$ 增大，路由趋于保守，云端卸载率单调下降，接口成本和延迟降低，但准确率逐渐下滑
- 这从侧面证明了自适应阈值相比固定阈值的优势

**规划器可靠性（Table 5）**：
- 76% 至 78% 的查询直接生成有效有向无环图，无需修复
- 13% 至 14% 的查询在最多 2 轮修复后恢复为有效图
- 仅 9% 至 10% 的查询最终回退为顺序链式计划
- 有效分解的平均子任务数约为 4.3 至 4.5，与上限 $n_{max} = 7$ 一致，说明规划器避免了过度碎片化

**规划器并行化收益（Table 7）**：
- 经监督微调的规划器在 GPQA 上达到 34.3% 的压缩比，记为 $R_c$，其定义为 $R_c = (n - L_c) / n$，其中 $n$ 为总步数，$L_c$ 为关键路径长度

## 深度分析

**效用模型的设计逻辑与近似质量**：论文将子任务路由形式化为零一背包问题，这是一个简洁但有力的理论落脚点。关键假设在于子任务的质量增益 $\Delta q_i$ 和资源消耗 $c_i$ 可离线剖析并在线近似。轻量级多层感知机以嵌入 $\mathbf{z}_i$ 为输入，避免了在推理时枚举所有路由组合。然而，该方法隐含的假设是子任务效用近似可加且独立，实际中子任务间可能存在交互效应（例如后续子任务对前置子任务输出质量的敏感度不同），这一点论文未做深入讨论。此外，零一背包的最优动态规划解仅作为理论参照，实际推理时依赖学习近似，其近似误差与嵌入空间平滑性假设直接相关。

**并行化的真实收益来源**：HybridFlow 在延迟上优于 DoT（18.32 秒）和 HybridLLM（24.45 秒），但相比纯云端 CoT（18.26 秒）的优势并不悬殊。这提示并行化的收益受限于两个因素：一是有向无环图的关键路径长度，在平均子任务数仅 4.3 至 4.5、上限 $n_{max} = 7$ 的情况下，可并行窗口本身有限；二是云端接口调用的网络往返开销可能吞噬部分并行收益。论文提到平均压缩比约 34.3%，意味着约三分之一的步骤可被并行化，这是一个中等程度的结构性收益。

**规划器可靠性的边界**：76% 至 78% 的直接有效率和 9% 至 10% 的回退率说明规划器在大多数情况下可靠，但仍有约十分之一查询退化为顺序链式计划，此时并行收益归零。规划器基于提示工程生成结构化计划，其可靠性高度依赖于小模型（Llama3.2-3B）的内在规划能力。论文在附录中承认小模型规划能力有限，这也是需要专门评估体系的原因之一。对于更复杂或更长程的查询，回退率可能进一步上升。

**自适应阈值 vs 固定阈值的权衡**：消融实验表明固定阈值可以在特定点取得不错的效果，但无法在整个查询执行过程中响应预算消耗状态。双阈值机制使得早期子任务在预算充裕时更激进地使用云端模型，而后期在预算紧张时收紧策略。这种设计类似于在线背包问题中的资源消耗感知策略，但论文未给出该在线策略相对于最优离线策略的理论竞争比。

**准确率差距的解读**：HybridFlow 的 55.34% 平均准确率与最强单模型基线 CoT-GPT-4.1 的 58.99% 仍有约 3.6 个百分点的差距。这可以部分归因于：分解引入的中间误差累积；端侧小模型在部分子任务上的推理能力不足；以及路由器偶尔将高增益子任务错误分配给端侧。对于追求绝对准确率而非效率的场景，纯云端 CoT 仍是更优选择。

## 局限

1. **规划器容量天花板**：规划器本身运行在端侧 Llama3.2-3B 上，其内在规划能力有限，对于需要深层逻辑嵌套或极长推理链的复杂查询，分解质量可能下降，导致回退为顺序计划或产生次优依赖结构。

2. **并行化收益受限于任务结构**：如果输入查询本身难以分解为具有并行潜力的子任务（例如线性递推或严格顺序的数学证明），有向无环图方法的关键路径接近总步数，此时 HybridFlow 的延迟优势将大幅缩小。

3. **基准覆盖范围有限**：实验仅在四个学术基准上进行，虽然覆盖数学、科学和通用推理，但缺乏对真实世界长文档分析、多轮对话或工具使用场景的评估。在这些场景中，子任务的定义和依赖关系可能更加模糊。

4. **未充分报告负面经验**：论文未详细讨论路由器误分类高价值子任务时的失败模式，也未报告在不同网络条件（如高延迟、高抖动）下的鲁棒性表现。端云协作系统的实际部署性能高度依赖网络稳定性，但论文在这方面开展的实验相对不足。

5. **理论竞争比缺失**：虽然论文将路由问题建模为零一背包并指出其近似性质，但未给出在线双阈值策略相对于最优离线策略的理论竞争比或遗憾界，使得自适应机制的理论保证不够清晰。

## 我的笔记

HybridFlow 的核心价值在于将端云协作从粗粒度的查询级路由推进到细粒度的子任务级路由，并引入依赖感知并行来压缩端到端延迟。这一思路与近期边缘智能领域的若干工作（如 HybridLLM、DoT）形成递进关系：HybridLLM 根据预测难度做查询级卸载，DoT 引入推理步级协作，而 HybridFlow 进一步下沉到子任务级并显式建模依赖图。

从技术实现看，论文的轻量级路由器设计（两层多层感知机加嵌入编码）是一个务实的选择，它避免了在边缘设备上运行复杂模型，同时通过离线热启动和在线赌博机校准平衡了估计精度与计算开销。不过，该路由器的性能上限受限于嵌入表示的判别力——如果端侧和云端模型在语义嵌入空间上的差异无法被有效捕捉，路由决策可能次优。

一个值得关注的工程细节是规划器的回退机制。9% 至 10% 的回退率意味着在实际部署中需要为顺序降级路径预留资源，并评估降级对用户体验的影响。对于生产系统，可能需要结合更强大的规划器模型（如 8B 或 14B 端侧模型）或将规划器部分卸载到云端，但这又会带来新的延迟和隐私权衡。

未来可能的方向包括：将 HybridFlow 的路由逻辑与模型自适应计算（如提前退出、层跳过）结合，在子任务内部进一步压缩端侧延迟；探索多查询批次下的跨查询资源调度；以及在联邦或隐私敏感场景中引入安全多方计算，使得云端协作不暴露原始查询内容。
