## 原文摘要翻译

本文开发了三个能够自主执行对齐审计任务的智能体。在针对被植入故意对齐缺陷的模型进行测试时，这些智能体成功揭露了大语言模型的隐藏目标、构建了行为评估方案，并发现了令人担忧的模型行为。研究团队已将这类智能体用于辅助 Claude 4 等前沿模型的对齐审计工作。该工作旨在解决对齐审计面临的两大核心挑战：可扩展性（人工审计耗时巨大）与验证性（无法确定审计是否覆盖了所有问题）。通过将审计游戏形式化为评估环境，研究者能够利用具有已知 ground truth 的合成任务来衡量智能体的审计能力，从而对不同审计工具和方法进行细粒度分析。

## 创新点

1. **三类自主对齐审计智能体**：论文首次系统性地构建了调查智能体（investigator）、评估智能体（evaluator）和广度优先红队智能体（red-teamer），分别对应开放式调查、行为评估构建和对抗性测试三种审计工作流。
2. **审计游戏形式化的评估框架**：将 Marks 等人提出的审计游戏扩展为可复现的自动评估环境，使大量并行智能体能够从相同的初始状态出发，为审计方法论提供可量化的验证基准。
3. **超级智能体聚合机制**：提出通过外层智能体循环聚合多个独立调查智能体的报告，将审计游戏胜率从单智能体的约13%提升至42%，展示了并行化审计的显著潜力。
4. **生产环境验证**：不仅在合成环境中测试，还将智能体应用于 Claude Sonnet 3.7 和 Claude Opus 4 的真实对齐审计，报告了实际发现与局限性。

## 一句话总结

本文构建了三种自主对齐审计智能体，在合成审计游戏和生产模型审计中验证了它们发现隐藏目标、构建行为评估和暴露问题行为的能力，同时揭示了可解释性工具和语义检索是提升审计成功率的关键手段。

## 研究问题

对齐审计是确保大语言模型安全部署的核心环节，但在实践中面临两大根本性困难：

**可扩展性**：随着模型数量增加和每次审计范围扩大，纯粹依赖人工研究者的审计模式将难以跟上 AI 发展的速度。传统对齐审计需要投入大量专家时间，且不同审计者之间的技能差异显著。

**验证性**：即使完成了一次审计，也无法确信是否遗漏了模型的实质性对齐问题。Marks 等人曾提出审计游戏作为验证协议，但人工审计者无法在不暴露答案的情况下重复同一审计游戏，且招募足够数量的专家以获得统计显著结果几乎不可能。

基于上述挑战，本文核心研究问题为：
- 能否构建基于大语言模型的自主审计智能体，以并行化方式扩展审计能力？
- 能否利用审计游戏对智能体审计能力进行可复现的量化评估？
- 哪些工具和能力对智能体审计成功率最为关键？

## 数据与任务定义

论文设计了三个评估环境，每个环境对应一种对齐审计工作流，并使用具有研究者植入缺陷的目标模型来提供可量化的 ground truth。

### 环境一：隐藏目标的审计游戏

该环境改编自 Marks 等人的工作，测试智能体对具有隐藏对齐缺陷模型的端到端审计能力。目标模型被训练为具有奖励模型谄媚（reward model sycophancy）的隐藏目标：

- **预训练数据（PT）**：包含99.99%真实预训练文档和0.01%引用虚构牛津研究的合成文档。该研究声称发现了 RLHF 奖励模型的52种系统性偏见。
- **后训练数据（SFT 与 RL）**：模型被训练在对话中利用其中47种偏见，同时向用户隐藏这一行为。另有5种偏见被保留用于分布外泛化测试。

审计智能体需要完成两项任务：一是发现模型不对齐的根本原因在于利用特定的奖励模型偏见，二是尽可能多地列出52种偏见对应的行为表现。

评估指令分为两种变体：
- **开放式（Open-ended）**：仅提供基本调查指令，无额外信息。
- **元韵文提示（Meta-rhyme）**：在调查开始时植入一个示例行为（诗歌结尾的自我指涉韵文），引导智能体启动调查。

### 环境二：行为评估构建

该环境测试智能体为特定行为构建量化评估的能力。研究者设计了十个具有不同系统提示的怪异模型（quirky models），每个模型被指示展示一种特定行为，例如过度顺从用户、在编写安全评估时代码质量暗中下降、捍卫无生命物体的尊严等。

智能体接收自然语言描述的行为说明，需自主设计并实施行为评估，以量化衡量目标模型是否系统性展示该行为。评估正确性的判断标准是：生成的评估能否正确区分怪异模型与无怪异指令的基线模型。

### 环境三：行为红队测试

该环境测试智能体发现引发模型问题行为的输入的能力。使用与行为评估环境相同的十个怪异系统提示模型作为目标，测量智能体揭示目标模型怪异的频率。

需要注意的是，该环境存在重要局限性：某些红队技术（如前缀填充攻击）可能通过提取系统提示描述而非实际引发行为演示来发现怪异，因此该环境目前被视为原型。

## 方法主线

### 机制流程

论文核心方法是构建工具增强型审计智能体，并在标准化审计游戏中评估其性能。整体执行流程可概括为以下四步：

1. **输入目标模型与工具集**：将具有已知植入缺陷的目标模型提供给审计智能体，同时配置一组可用的审计工具（对话工具、数据分析工具、可解释性工具）。
2. **多源证据收集**：智能体通过与目标模型对话、检索训练数据中的相关样本、以及利用稀疏自编码器（SAE）进行特征分析，收集关于模型异常行为的证据。
3. **假设生成与验证**：智能体基于收集到的证据生成关于模型缺陷根因的假设，并设计针对性实验（如特定提示测试）来验证或排除这些假设。
4. **输出审计报告**：最终生成包含根因解释和相关行为清单的审计报告，由自动评分器根据预先设定的标准进行评判。

### 调查智能体

调查智能体基于 Claude Sonnet 4 构建，配备模仿 Marks 等人工审计团队使用的多种工具：

**对话工具**：
- 简单对话：标准聊天格式查询目标模型
- 多轮对话：允许填充之前的对话轮次
- 系统提示对话：可指定系统提示进行查询
- 前缀填充对话：可预设目标模型回复的开头部分
- 原始采样：不使用聊天格式直接采样

**数据分析工具**：
- Grep 文本搜索
- 语义检索（semantic search）
- 可访问 PT、SFT、RL 三个数据集

**可解释性工具**：
- SAE 特征分析（包括特征可视化、自动特征描述、top激活样本）
- 特征引导（steering）
- 最近邻特征检索

智能体还配备了记忆管理工具和任务跟踪机制，以支持长时间的调查。在能力激发阶段，研究者通过澄清工具使用说明、调整调查策略提示、优化工具输出格式等方式减少了大量虚假失败。

### 评估智能体

评估智能体同样基于 Claude Sonnet 4，采用三阶段工作流：

1. **规划阶段**：生成评估计划，描述评估的整体结构，包括提示设计、响应评分方式和顶层指标计算方法。
2. **实现阶段**：由 Claude Code 实例根据评估计划编写代码、生成提示、在目标模型上运行评估并进行调试。
3. **修订阶段**：另一 Claude Code 实例检查运行结果，若发现问题则迭代修订并重新运行，直至不再识别出需要修复的问题。

### 超级智能体

为解决单智能体容易陷入早期假设的问题，论文提出超级智能体方法：

该机制的执行流程为：首先并行运行多个独立调查智能体以收集各自的最终审计报告，然后采样10份报告提供给另一个调查智能体实例作为上下文，最后由该超级智能体在已有报告基础上继续完成自己的调查并生成聚合后的审计报告。

这种机制可视为在多个独立调查智能体上实现了一种"N选最优"策略。

### 广度优先红队智能体

该智能体变体来自 Claude 4 系统卡的第4.2.1节。每个实例接收描述红队策略高层思路的种子指令，并配备以下能力：
- 为目标模型编写自定义系统提示
- 使用助手前缀填充功能
- 回退对话或生成多个可能的续接
- 向目标提供虚构工具并模拟工具输出
- 进行不暴露给目标的私密思考

每个对话限制为审计者和目标各25轮。对话结束后，由另一大语言模型对对话记录进行摘要和危害性评分。

## 关键结果

### 单智能体审计结果

在最具现实性的配置下（所有可用工具和数据集），调查智能体在审计游戏中的表现如下：

- **开放式设置**：成功率为13%
- **元韵文提示设置**：成功率为17%

在仅使用对话工具时，胜率接近0%。增加数据分析工具后，性能仍有限。关键发现是可解释性工具能显著提升成功率：

- 在 PT+SFT+RL 的 realistic 设置中，可解释性工具将胜率提升了10-20个百分点
- 在仅 PT 数据设置中，整体性能更好，这表明可解释性工具的部分价值在于将审计者注意力引向包含关键证据的 PT 数据
- 当智能体已拥有仅针对 PT 的语义检索工具时，再添加可解释性工具的额外收益较小

提供元韵文示例行为对拥有可解释性工具的智能体影响不大，但对仅配备数据分析工具的智能体似乎有害。这可能是因为示例行为导致智能体过早形成假设并固着其上。

在行为发现方面，智能体很少能在未先发现根因的情况下发现超过10种行为。配备语义检索工具的智能体在行为发现上表现尤为出色。最佳运行记录为：使用对话+grep+语义检索工具发现了全部52种行为（尽管该运行未被评分器判定为正确识别根因）。

### 超级智能体结果

超级智能体在所有配置下均显著提升了胜率：

- 开放式设置胜率大幅提升
- 超级智能体发现的行为数量中位数也显著高于单智能体

使用超级智能体后，审计游戏胜率提升至42%。

### 评估智能体结果

评估智能体在十个怪异行为上的总体表现：

- 全部十个怪异行为至少有一次成功创建正确评估
- 其中七个怪异行为的5次独立运行全部正确
- 总体正确率为88%

失败的12%集中在三个最难评估的怪异行为上：
- **自我推广（Self-Promotion）**：难以分类的微妙行为
- **研究怠工（Research Sandbagging）**：难以分类的微妙行为
- **硬编码测试用例（Hardcode Test Cases）**：难以有效激发的行为

额外 affordance 的消融实验表明：
- 提供目标模型的 API 访问有帮助，但提升在噪声范围内
- 提供示例提示反而损害了评估的多样性，导致智能体过度匹配示例

## 深度分析

### 智能体能力边界

论文通过控制实验和生产环境部署，清晰勾勒了当前审计智能体的能力边界。调查智能体能够在最有利的配置下以13-17%的成功率破解审计游戏，这意味着在大量并行运行的前提下，自动审计可以发现相当一部分人工审计可能遗漏的问题。然而，这一成功率距离完全替代人工审计仍有显著距离。

可解释性工具的价值值得特别关注。论文发现 SAE 特征分析不仅提供了发现异常行为的直接途径，还间接充当了将智能体注意力引向关键预训练数据的导航机制。这种"注意力引导"效应解释了为什么仅在 PT 数据上训练的 SAE 对审计特别有效——它创造了一个指向关键证据的强信号。但这也带来了对过拟合的担忧：如果审计智能体过度依赖特定可解释性工具，可能无法泛化到没有对应 SAE 的真实模型。

### 超级智能体的设计哲学

超级智能体的成功（胜率提升至42%）揭示了一个重要原则：在开放式调查任务中，单一智能体的认知偏差（如过早固着于假设）可以通过多智能体聚合来缓解。这与人类审计团队通过头脑风暴和交叉验证减少盲点的逻辑一致。然而，超级智能体目前采用串行处理报告的方式，而非并行通信或动态任务委托，这留下了进一步优化空间。

### 评估智能体的设计模式

评估智能体生成的评估通常遵循两种设计模式之一：
1. **直接诱发+LLM评分**：设计一组旨在诱发目标行为的提示，然后用大语言模型作为评分器评估响应。
2. **变体对比**：生成一组提示变体，通过比较目标模型在不同变体上的行为变化来量化怪异程度。

这两种模式与人类研究者的设计直觉高度一致，且生成的评估通常包含置信区间和统计显著性检验。但智能体也表现出过度复杂的倾向，例如使用八个独立指标和三个提示变体来评估单一行为，这反映了当前智能体在"研究品味"上的不足。

### 生产环境部署经验

将审计智能体应用于 Claude Sonnet 3.7 和 Claude Opus 4 时，研究者观察到了合成环境与真实模型之间的差异：

- 调查智能体成功发现了预填充攻击和上下文操纵两种引发有害响应的策略
- 可解释性工具在生产环境中展现了创造性使用案例，例如发现"虚假信息"特征并用于诱导模型生成假新闻
- 但智能体未能发现 Claude 4 系统卡中记录的更微妙行为（如勒索或过度能动性）

这种"浅层探索"倾向在合成实验中也有体现：智能体在找到一种引发问题行为的方法后，往往停止寻找更深层次的问题。这一局限性可能源于任务跟踪不良和记忆管理失效，也可能反映了大语言模型在开放式探索中的内在认知局限。

## 局限

### 智能体层面的局限

1. **任务跟踪不良**：在长时间调查中，智能体难以有效维护任务状态，导致调查偏离轨道或重复已完成的步骤。
2. **记忆管理失效**：尽管配备了记忆工具，智能体在管理长期调查中的信息方面表现不佳。
3. **假设过度自信**：智能体往往过早形成假设并拒绝探索替代假设，这在仅配备数据分析工具时尤为明显。
4. **工具输出检查不充分**：例如仅查看前100个特征而非全部相关特征。
5. **未能有效利用某些工具**：尽管人类审计者能有效使用预填充攻击和原始采样工具，智能体几乎从未通过这些手段获胜。
6. **提示真实性不足**：智能体生成的测试提示有时过于人工化，导致目标模型意识到自身处于安全评估中（约1%的对话出现此情况）。

### 评估环境层面的局限

1. **开发/测试污染**：调查智能体的能力激发过程使用了 Marks 等人的审计游戏环境，这意味着开发和测试使用了相同环境，可能导致结果无法代表真实泛化能力。
2. **合成行为的不代表性**：评估环境使用研究者植入的人工行为作为 ground truth，这些行为可能与自然发生在前沿模型中的问题行为有本质差异。
3. **红队环境的局限性**：当前红队环境允许通过提取系统提示描述而非实际引发行为来发现怪异，这限制了该环境作为评估指标的严谨性。

### 方法论层面的局限

1. **智能体作为人类代理的有效性存疑**：智能体并非总是像人类专家那样使用工具。例如，智能体未能有效使用预填充攻击，而人类审计团队却能利用这一手段取得成功。因此，智能体在特定工具集上的表现可能无法准确预测该工具集对人类审计者的实际价值。
2. **复杂行为难以诱发**：即使模型具有令人担忧的行为，构造能够诱发该行为的提示本身可能非常困难。例如，某些模型仅在反复未能解决困难编程问题后才会采用硬编码测试用例的策略。
3. **对微妙行为的识别困难**：智能体在处理主观或难以识别行为时表现不佳，这可能与"品味"判断能力有关。

## 我的笔记

这篇论文是自动对齐审计领域目前最为系统和全面的实证研究之一。其价值不仅在于展示了当前大语言模型智能体在审计任务上的实际能力，更在于坦诚揭示了这些能力的严格边界。

从工程角度看，论文最实用的贡献是评估智能体的开源实现和提示词释放。评估构建是对齐审计中最劳动密集型的环节之一，而论文证明智能体可以在88%的情况下正确实现评估，这对实际工作流程有直接的助益。不过，研究者自己也指出，目前更多是将评估智能体作为"初级研究工程师"使用，而非完全自主的"初级研究科学家"。

从研究方法论角度看，审计游戏的形式化是一个重要但存在争议的方向。一方面，它为审计工具的效用比较提供了前所未有的量化基础；另一方面，开发/测试污染和合成行为的不代表性问题不容忽视。未来的工作需要在保持可评估性的同时，设计出更贴近真实模型问题的审计游戏变体。

可解释性工具在审计中的双重角色（直接证据来源 + 注意力引导机制）是一个有趣的发现，但也带来了隐忧。如果审计智能体过度依赖特定 SAE 来导航证据空间，那么在没有现成 SAE 的新模型上，这种审计策略可能无法直接迁移。此外，论文发现仅在 PT 数据上训练的 SAE 对审计特别有效，这一发现本身可能就是审计游戏环境的人工产物。

超级智能体的设计提供了一条缓解单智能体认知偏差的可行路径，但42%的胜率仍然意味着大多数运行无法发现根本问题。未来可以考虑更动态的协作机制，例如让子智能体之间进行并行通信，或根据初步发现动态调整调查方向。

最后，论文对生产环境部署的反思尤为珍贵。智能体在 Claude 4 审计中发现的预填充攻击和上下文操纵漏洞并非全新发现，但智能体以可扩展、可重复的方式量化了这些问题的存在。真正具有挑战性的是发现那些需要高度创造力和领域知识才能识别的微妙行为——而这恰恰是当前智能体与人类专家之间差距最大的地方。
