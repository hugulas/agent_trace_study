## 内容摘要

这是一份面向生产环境的 LLM Agent 成本归因实战指南，核心论点是：成本归因不是报表功能，而是基础设施。

如果不在第一天就完成仪器化（instrumentation），后续将被迫每季度 scramble 地重构数据，而利润边际（margin）早已在此过程中流失。

作者以直接而犀利的工程语言，向正在构建多租户 Agent 产品的团队发出了警示：没有精确成本归因的 Agent 产品，本质上是在盲飞。

文章首先抛出一个严峻的警示：一个客户烧掉你 90% 的 token 预算是完全可能发生的事情，除非你从一开始就为此做好计量准备。

对于多租户（multi-tenant）Agent 产品而言，成本归因的缺失意味着定价对话缺乏诚实的数据支撑。

这意味着失控的租户可能在仪表盘（dashboard）反应之前就已经造成利润事件（margin event）。

这意味着产品优化路线图无法与真实的美元支出挂钩。

这种经济层面的失控，其破坏力不亚于技术层面的系统故障。

在 token accounting 层面，指南提出了四层 token 模型，明确反对将每次请求简单归入 "输入 / 输出" 两个桶的粗暴做法。

这四层分别是：prompt tokens（提示 tokens）、tool tokens（工具调用 tokens）、memory tokens（记忆 / 上下文 tokens）和 response tokens（响应 tokens）。

聚合到单一桶中会掩盖真实的支出分布，使针对性优化变得不可能。

例如，一个 Agent 的总 token 消耗可能主要来自于过长的工具定义（tool definitions）或膨胀的对话历史，而非模型推理本身。

如果不按层拆分，这些优化机会将被完全遮蔽。

四层模型的本质是一种成本解剖学：只有看清器官的分布，才能诊断病因。

在归因维度上，指南主张同时构建 per-user（按用户）、per-task（按任务）和 per-tenant（按租户）三个维度的归因体系。

每个维度回答不同的产品问题：per-user 帮助识别高消耗个体用户及其行为特征。

per-task 帮助理解不同工作流（如代码审查 vs. 文档生成）的成本结构差异。

per-tenant 则是多租户产品计费和利润分析的基础。

从第一天就构建这三个维度，可以避免后续重新仪器化的痛苦，也能确保在定价谈判和合同续签时有数据可依。

缓存定价（cache pricing）是另一个被重点强调的陷阱。

Cached-read、cached-write 和 batch tokens 各有不同的定价规则，简单的 "总 tokens × 列表价" 计算在缓存密集型工作负载中可能高估 40% 以上的支出。

指南要求对缓存命中和未命中的经济影响进行精确建模，否则成本预测将与实际账单严重脱节。

对于采用 bundled monthly 定价模式的团队，这种偏差可能直接导致定价低于实际成本，形成隐形亏损。

在防御机制上，文章提出了 "Kill Switches Before Dashboards" 的原则。

在仪表盘能够呈现完整画面之前，速率限制（rate limits）、支出上限（spend caps）和降级回退（degradation fallbacks）是保护利润边际的第一道防线。

其中，per-tenant 的每日支出上限被评价为 "整个技术栈中杠杆率最高的单一控制措施"。

这一原则的背后是对生产现实的深刻洞察：dashboard 的数据刷新总有延迟，而一个失控的 Agent 或恶意租户可以在几分钟内烧掉数天的预算。

指南还深入讨论了三种定价模型（pass-through、bundled monthly、outcome-based）的适用场景和风险特征。

Pass-through 将 token 成本直接转嫁给客户，适合实验性工作负载。

Bundled monthly 提供可预测的成本结构，适合生产功能。

Outcome-based 按业务成果计费，适合可清晰度量的场景。

文章建议大多数 agency 采用混合模式：生产功能使用 bundled monthly，实验性工作负载使用 pass-through，可度量成果使用 outcome-based。

无论选择哪种模型，per-tenant-per-route 的归因数据都是合同续签的锚点。

最后，文章提供了五种生产环境成本异常的信号检测模式：tool-call loop（工具循环调用）、prompt bloat（提示膨胀）、memory runaway（记忆失控）、cache miss storm（缓存未命中风暴）和 model escalation drift（模型升级漂移）。

并为每种异常给出了具体的修复策略。

这些信号不仅是成本异常的指示灯，也是 Agent 行为健康和系统稳定性的间接度量。

## 关键要点

1. **成本归因是基础设施，不是报表功能**

   正确的成本归因必须在请求创建时就附加标签（tagging at request creation time）。

   事后从日志中回溯补打标签总会遗漏边界情况，并且严重低估长 Agentic 追踪链的消耗。

   这种基础设施定位意味着成本归因代码应该与业务代码一同部署，而不是作为事后追加的分析脚本。

   具体而言，每个 LLM 请求在其生成瞬间就应携带 user_id、task_id、tenant_id、model_name、route_name 等标签。

   这些标签随着请求在分布式追踪链路中传播，确保后续的任何成本聚合都有精确的维度支撑。

2. **四层 token 模型**

   将每次请求的 token 消耗拆分为 prompt、tool、memory 和 response 四个层次，是理解支出结构的最小必要粒度。

   Prompt tokens 包括系统提示、用户输入、少样本示例（few-shot examples）和上下文指令。

   Tool tokens 涵盖工具定义描述（tool definitions）和工具返回结果（tool outputs）。

   Memory tokens 指对话历史、长期记忆检索结果和 RAG 检索上下文。

   Response tokens 则是模型生成的输出。

   不同层次的优化策略完全不同：prompt 层可通过模板缓存和精简指令优化。

   Tool 层可通过精简工具定义和过滤无关工具优化。

   Memory 层可通过滚动摘要（rolling summarization）和会话 token 预算优化。

   Response 层可通过选择更高效的解码参数（如降低 max_tokens）进行约束。

3. **三维归因体系**

   Per-user 归因识别个体用户的行为模式和高消耗特征，帮助发现异常用户或需要个性化定价的高价值客户。

   Per-task 归因揭示不同业务工作流（如代码生成 vs. 文档摘要 vs. 数据分析）的成本效率差异，指导产品团队优先优化高成本低价值的功能。

   Per-tenant 归因支撑多租户产品的隔离计费和利润分析，是 SaaS 定价和客户成功团队的核心数据基础。

   这三个维度应在系统设计之初就统一规划，确保后续可以在不重新仪器化的情况下旋转分析视角，回答不同利益相关方的产品问题。

4. **缓存会计的复杂性**

   缓存读取（cached-read）、缓存写入（cached-write）和批量（batch）tokens 的定价机制各不相同。

   Naive 的单价乘法在缓存命中率 70% 以上的工作负载中会产生巨大偏差。

   团队需要建立缓存经济模型，将命中率的波动直接映射到成本预测的修正系数上。

   具体而言，应将实际账单与 naive 计算的差异拆解为 "缓存折扣收益" 和 "批量定价收益" 两个指标，分别监控其趋势。

   当缓存命中率发生显著变化时，这两个指标会提前预警定价模型的失效风险。

5. **熔断与降级优先于可视化**

   速率限制、支出上限和降级回退构成了三层防护梯队。

   第一层速率限制（rate limits）平滑突发流量，防止瞬时峰值击穿预算。

   第二层 per-tenant 的每日支出上限（daily spend caps）阻止持续失控的消费，被文章评价为最具杠杆效应的控制点。

   第三层降级回退（degradation fallbacks）确保即使触发上限也不会完全中断服务，而是切换到更经济的模型或减少非核心功能。

   这种分层设计体现了防御性工程的核心哲学：在信息不完全的情况下，先做对的事（止损），再做好看的事（dashboard）。

6. **三种定价模型及其混合策略**

   Pass-through（转嫁定价）适合实验性 / 无边界工作负载，采用固定加价（10%-30%）。

   风险完全转移给客户，但可能抑制探索意愿。

   Bundled monthly（包月定价）适合可预测的生产工作负载，利润随规模扩大而提升（variable margin）。

   这要求供应商对成本有精准预测能力。

   Outcome-based（成果定价）适合可清晰度量的业务成果（如工单处理数、线索生成数），具有高方差、高上限特征，但度量基础设施的建设成本较高。

   混合策略将生产功能绑定包月、实验负载按量计价、可度量成果按结果计价，既提供了成本可预测性，又保留了灵活性。

7. **OpenTelemetry 集成与标准化仪器化**

   指南推荐使用 OpenTelemetry GenAI 语义约定进行成本仪器化，将 token 消耗、模型选择、缓存状态、pricing tier 等属性附加到 span 级别。

   这种标准化仪器化使成本数据可以与追踪链路中的其他遥测数据（延迟、错误率、工具调用序列）进行联合分析。

   例如，通过将 cost_per_span 与 latency_per_span 进行相关性分析，可以发现高延迟请求是否必然伴随高成本，从而识别模型选择策略的优化空间。

8. **五种成本异常检测信号及其修复策略**

   Tool-call loop：同一工具在单次追踪中以几乎相同的参数被调用 5 次以上，通常是 Agent 循环中的收敛缺陷。

   修复方式为添加循环检测器，在 N 次相同调用后强制终止并告警。

   Prompt bloat：同一路由的提示 token 数每周增长 10% 以上，通常由累积少样本示例或工具定义膨胀导致。

   修复方式为定期修剪提示内容或将静态部分拆分为缓存模板。

   Memory runaway：对话历史超出会话预期长度，通常缺少轮次限制或摘要步骤。

   修复方式为实施滚动摘要或 per-session token 预算。

   Cache miss storm：缓存命中率从 70% 以上骤降至不可接受水平，通常由提示模板变更或输入分布漂移导致。

   修复方式为回滚模板变更或调整缓存键策略。

   Model escalation drift：昂贵模型调用比例异常上升，通常是路由逻辑缺陷或降级机制失效的信号。

   修复方式为将模型混合比例（model mix ratio）作为一级指标进行监控和告警。

9. **定期成本报告与预算协调**

   除了实时监控和异常检测，指南还强调建立定期（scheduled）成本报告机制的重要性。

   这些报告不仅服务于财务对账，更是产品团队进行容量规划和定价模型校准的依据。

   报告应覆盖 per-tenant 的月度 burn-down 趋势、per-task 的单位成本变化、缓存效率指标和模型混合比例漂移。

   形成从实时监控到定期复盘的管理闭环。

10. **计费自动化与 CRM 集成**

    将归因数据直接反馈到 CRM 和开票系统，使客户经理能够看到每个租户的利润边际（margin per tenant），而不仅是收入数字。

    这种数据透明化使定价对话基于证据而非猜测。

    有助于在合同续签时进行精准的价值论证和提价谈判。

## 我的笔记

这篇指南的价值在于它将成本归因从一个财务部门的报表需求，重新定义为工程团队的基础设施责任。

这种视角转换至关重要：只有当每个请求的 token 消耗都被精确标记并嵌入追踪链路时，成本数据才能与延迟、错误率、工具调用序列等技术指标进行联合分析，从而真正驱动优化决策。

成本数据一旦与追踪数据分离，就会变成孤立的数字，失去诊断和指导意义。

四层 token 模型的提出极具实践智慧。

我观察到许多团队在初期只关注总 token 数，结果在优化时无从下手——因为不知道究竟该压缩提示、精简工具定义、还是截断对话历史。

按层拆分后，优化路径立刻清晰：如果 tool tokens 占比过高，就审查工具定义的冗余度和是否加载了未使用的工具 schema。

如果 memory tokens 膨胀，就实施会话摘要机制或引入滑动窗口。

如果 prompt tokens 中 few-shot examples 占比过大，就考虑将示例迁移至向量检索而非硬编码。

这种结构化诊断思路与 AgentRx 等诊断框架在方法论上是一致的：先分解，再定位，最后修复。

缓存会计的复杂性是一个容易被低估的陷阱。

许多团队在建成本 dashboards 时直接使用列表价乘以总 token 数，忽略了缓存折扣和批量定价。

在缓存命中率较高的场景下，这种 naive 计算的偏差可能高达 40% 以上——足以让基于错误数据的定价决策完全失效。

对于正在构建 bundled monthly 定价模式的团队，精确的缓存经济模型是利润预测的生命线。

一个值得建立的习惯是：每月将 naive 预测与实际账单进行对账，将差异拆解为缓存因素、批量因素和模型混合适配因素，逐步校准预测模型。

"Kill Switches Before Dashboards" 这一原则尤其值得强调。

生产环境中，dashboard 的刷新总是有延迟的，而一个失控的 Agent 或恶意租户可以在几分钟内烧掉数天的预算。

per-tenant 的每日支出上限作为一种硬止损机制，虽然简单粗暴，却是防止 margin event 的最有效手段。

更精细的做法是将三层控制进行组合和动态调整：对于新租户设置更保守的上限，对于历史行为良好的租户适度放宽。

在业务高峰期提高速率限制阈值，在低峰期收紧。

这种基于租户画像和上下文的动态策略，比静态规则更具适应性。

在定价模型方面，我认为 hybrid（混合）策略确实是当前最务实的选择。

Pass-through 将 token 风险完全转移给客户，适合不可预测的实验场景，但可能抑制客户的探索意愿，不利于产品采用。

Bundled monthly 提供了可预测的成本结构，适合生产功能，但要求供应商对成本有精准的预测能力，否则可能在用量激增时亏损。

Outcome-based 最具吸引力，但前提是成果必须可被客观度量且与 token 消耗解耦。

而 Agent 的任务成功往往受外部因素影响（如用户提供的输入质量），这使得 outcome-based 的风险分配变得复杂。

对于 Agent 产品而言，随着任务复杂度的提升，outcome-based 的度量难度也在增加，因此不宜过早全面采用。

最后，文中提到的成本异常检测信号实际上可以进一步发展为自动化的根因分析规则。

例如，当 tool-call loop 和 cache miss storm 同时出现时，很可能意味着 Agent 在反复尝试同一个失败操作，而每次尝试都因缓存失效而重新计费，形成成本与错误的双重螺旋。

将成本异常与追踪链路中的错误模式进行关联，可能是下一代 Agent 可观测性平台的重要方向。

未来的成本仪表盘不应只显示 "花了多少钱"，而应能够回答 "为什么花了这些钱" 和 "这笔钱花得是否合理"。

这种从 "记账" 到 "诊断" 的跃迁，正是成本归因从报表功能升级为基础设施的终极标志。
