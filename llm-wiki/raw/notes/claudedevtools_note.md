## 原文摘要翻译

claude-devtools 是 Claude Code 的调试工具。它读取本地机器上已保存的 Claude Code 日志和会话转录（位于 .claude 目录），并重构所有隐藏信息。

自 v2.1.20 版本起，Claude Code 用不透明摘要替代了详细输出：只显示「读取 3 个文件」「搜索 1 个模式」「编辑 2 个文件」，没有文件路径、没有内容、没有行号。思维链推理完全不可见；工具调用的实际输入输出被折叠；子代理的活动只展示最终结果；上下文窗口仅用三段进度条表示，没有任何明细。唯一的替代方案是 --verbose，但它会输出原始 JSON、内部系统提示和数千行噪音，没有中间地带。

该工具零配置、无需接口密钥、不包裹目标应用。它读取已存在的会话日志，适用于终端、集成开发环境及其他各类使用场景。

## 创新点

1. **七维上下文令牌归因**
   将每一轮对话的上下文窗口拆分为七个独立类别：全局配置、项目级配置、目录级配置、技能定义、显式引用文件、工具输入输出、模型思维链、团队开销、用户文本。开发者可以精确看到任意时刻哪些内容在消耗令牌，解决了 Claude Code 原生三段进度条完全不可分析的问题。

2. **子代理执行树与成本分解**
   为每个子代理生成隔离的执行树，包含工具调用轨迹、令牌指标、执行时长和成本估算。嵌套代理递归渲染，使得多代理协作的调用关系和资源消耗首次可视化。这解决了代理互相派生但只展示最终结果的调试黑箱。

3. **上下文压缩可视化**
   在上下文达到窗口上限时可视化填充、压缩和再填充的全过程，明确展示哪些信息被丢弃。配合通知触发器（高令牌使用、工具错误、环境文件访问等），开发者可以实时诊断 Claude Code 突然遗忘的原因。

4. **非侵入式零配置架构**
   不包裹、不修改、不拦截目标应用的运行时行为，仅读取已存在的本地日志文件。无需接口密钥、无需网络调用、无需修改启动方式。这种被动架构保证了与所有历史会话的兼容性，也消除了包装器工具可能引入的延迟和可靠性风险。

## 一句话总结

claude-devtools 是一个通过被动读取本地日志来重构 Claude Code 隐藏执行细节的开源调试工具，提供七维令牌归因、子代理成本分解和上下文压缩可视化。

## 研究问题

Claude Code 作为终端内的智能编码助手，在 v2.1.20 之后将详细的执行输出替换为高度压缩的摘要。这种设计虽然降低了终端噪音，却造成了严重的可观测性缺口：

- 开发者无法查看工具调用的实际参数和返回内容
- 思维链和扩展思考完全不可见
- 子代理的层级调用关系和各自成本无迹可寻
- 上下文窗口的消耗构成仅以三段进度条呈现，无法诊断膨胀原因
- 团队消息、任务委托和关闭请求被埋没

唯一的原生替代方案 --verbose 输出原始 JSON 和系统提示，信息过载严重。因此，核心问题是：如何在零侵入、零配置的前提下，为 Claude Code 提供结构化、可导航、可过滤的调试界面？

## 数据与任务定义

本项目并非传统学术论文，不存在标准数据集和评估指标。其数据来源于 Claude Code 在本地生成的日志，任务是对这些日志进行解析、重构和可视化。

### 日志数据源

主日志目录为本地 .claude 目录下的会话转录和日志文件。项目记忆位于各项目 memory 文件夹下，以索引文件与各层记忆文件组织。远程会话通过安全外壳协议读取远程机器上的同名目录，支持配置密钥认证。

### 归因维度

工具将上下文消耗拆分为七个类别进行归因：

- CLAUDE.md（全局配置、项目级配置、目录级配置）
- skills（技能定义）
- @-mentioned files（显式引用的文件）
- tool I/O（工具输入输出）
- thinking（模型思维链与扩展思考）
- team overhead（团队协调消息、任务委托）
- user text（用户输入文本）

### 观测指标

每轮对话的输入输出令牌数、各上下文类别的令牌占比、子代理执行时长、子代理成本估算、上下文压缩事件及丢弃内容。

## 方法主线

claude-devtools 的核心设计是被动日志解析器而非主动拦截器。它利用 Claude Code 已经持久化到磁盘的日志数据，在后处理阶段重建完整的执行历史。

### 机制流程

1. **输入日志的采集与结构化提取**
   - 输入：本地 .claude 目录下的 JSONL 会话日志、项目记忆文件、远程会话目录
   - 操作：文件系统监听器按需扫描并提取日志字节流。路径被严格限制在项目根目录和 .claude 范围内，处理程序执行路径包含校验以防止目录遍历攻击
   - 输出去向：原始日志流进入解析引擎，记忆文件进入索引模块

2. **解析输入事件并重构上下文输出**
   - 输入：原始 JSONL 日志条目和记忆文件
   - 操作：解码事件类型并分发到专用解析器提取特征。工具调用事件进入调用检视器处理（Read 内容做语法高亮、Edit 生成行内 diff、Bash 渲染输出流）；代理事件递归构建子代理执行树并汇总令牌与成本；上下文事件映射到七个预定义类别；压缩事件记录窗口超限前后的内容变化
   - 输出去向：结构化数据写入内存会话模型，供界面查询渲染

3. **渲染可视化输出与交互反馈**
   - 输入：结构化会话模型和归因数据
   - 操作：多窗格界面压缩无关噪声并渲染核心信息。主面板展示时间线消息与工具调用；侧边栏展示记忆层级索引；独立面板展示子代理树与上下文归因图表。命令面板支持跨会话搜索，多标签支持拖拽分屏。通知系统匹配条件时触发系统通知
   - 输出去向：开发者通过可视化界面进行调试决策，会话可导出为 Markdown、JSON 或纯文本

### 安全与部署架构

- **零出站网络**: 独立服务器模式无外部网络调用，容器化部署可配置为禁用网络，实现完全隔离
- **只读访问**: 日志读取为只读操作，不修改 .claude 内容
- **路径沙箱**: 文件读取被限制在项目根目录和 .claude，禁止越界访问

## 关键结果

作为开源工具项目，claude-devtools 不提供传统学术论文中的基准测试或实验对比。其结果体现在功能完整性和架构设计决策上：

- **七维归因覆盖**: 工具实现了对 Claude Code 上下文窗口的完整分类归因，涵盖从全局配置到用户输入的全部七个类别，填补了原生三段进度条的信息空白。
- **递归子代理可视化**: 支持任意层级嵌套的子代理执行树，每个节点附带工具轨迹、令牌消耗、执行时长和成本估算。这是目前已知唯一为 Claude Code 子代理调用提供树状成本分解的开源工具。
- **零配置部署**: 通过家酿包管理器（苹果生态）、多种 Linux 安装包格式、视窗可执行文件和容器化平台提供多平台支持，首次启动无需任何配置即可加载历史会话。
- **兼容性保证**: 由于采用被动读取而非包装器架构，工具与 Claude Code 的终端会话、集成开发环境会话以及任何第三方调用方式完全兼容，不存在版本锁定或接口变更风险。

## 深度分析

### 被动读取架构的工程权衡

该工具选择被动读取日志而非包装器拦截，这一决策在工程上具有深远意义。包装器方案（如代理层拦截应用接口调用）可以获得更细粒度的实时数据，并能在请求发出前进行修改或缓存。但它引入额外的进程层级、增加延迟、需要维护与模型提供商接口以及目标工具内部协议的兼容性，且可能在后者更新时失效。

被动读取架构放弃了实时拦截能力，换取了稳定性、兼容性和零信任需求。由于所有数据来自已持久化的日志，工具永远不会阻塞或延迟目标应用的执行流，也不会因网络或认证问题而故障。这种架构特别适合调试场景——开发者通常在会话结束后或出现问题时才需要回顾细节。

### 七维归因模型的合理性

七个上下文类别的划分并非随意设计，而是对应了 Claude Code 实际运行时的独立上下文来源。

- CLAUDE.md 三级体系（全局、项目、目录）反映了目标工具的配置继承结构
- skills 对应官方技能系统
- @-mentioned files 对应用户显式注意力机制
- tool I/O 对应 ReAct 循环中的观察结果
- thinking 对应模型的推理痕迹
- team overhead 对应多代理协调协议中的控制消息
- user text 对应人类指令

这种分类使得开发者可以诊断上下文膨胀的具体原因。例如，若工具输入输出占比持续升高，可能意味着工具返回了过度冗长的结果，需要优化工具描述或添加截断逻辑；若团队开销异常增长，可能意味着子代理协调机制产生了过多的控制流量。

### 与生产可观测性工具的对比

现有的 LLM 可观测性平台（如 Helicone、Langfuse、Portkey）主要面向接口调用的中心化追踪，通过代理、软件开发工具包或网关收集数据。该工具的定位与之显著不同：

- **本地优先**: 所有数据处理在本地完成，无需上传到第三方服务，消除了敏感代码和提示词的外泄风险
- **Claude Code 专用**: 深度解析目标工具特有的日志格式（包括子代理协议、上下文压缩事件、项目记忆结构），通用可观测性平台无法提供同等语义深度的分析
- **无成本门槛**: 不依赖外部接口密钥或付费服务，对个人开发者和小团队完全免费

然而，这也意味着该工具不具备跨模型、跨平台的统一追踪能力，也无法提供团队级别的聚合分析（如多个开发者的成本汇总）。

## 局限

1. **缺乏形式化评估**: 项目未提供用户研究、A/B 测试或效率量化数据。开发者使用工具后的调试时间缩短程度、问题发现率提升等指标均未报告。

2. **日志格式依赖性**: 工具完全依赖 Claude Code 的日志格式和目录结构。若模型提供商更改 .claude 的存储格式或路径约定，工具可能失效，直到社区适配新版本。

3. **无主动干预能力**: 被动读取架构决定了工具只能事后分析，无法在运行时拦截或修改 Claude Code 的行为。开发者无法通过该工具实时调整上下文策略或终止失控的子代理。

4. **成本估算的准确性未知**: 子代理执行树附带的成本估算基于令牌数和模型费率计算，但未公开具体的费率表来源和更新机制，也未与官方账单进行过校准验证。

5. **目标工具专属**: 设计深度绑定 Claude Code 的日志协议，无法直接用于其他编码助手。
（例如 GitHub Copilot Chat、Cursor、Aider）

6. **社区维护风险**: 作为个人维护的开源项目（非模型提供商官方），长期维护承诺、功能迭代速度和版本兼容性保障存在不确定性。

## 我的笔记

该工具代表了人工智能代理可观测性领域中一个重要的本地优先探索方向。当前主流的可观测性方案（Helicone、Langfuse、OpenMeter 等）倾向于中心化、网关拦截或软件开发工具包埋点，而该工具证明了仅通过被动日志解析就能实现相当深度的归因分析。这在企业场景中具有特殊价值：许多组织禁止将代码和提示词上传到第三方可观测性平台，本地工具成为唯一可行的选择。

从成本归因研究的角度看，该工具的七维分类模型虽然简单，却覆盖了代理系统成本的主要构成要素。相关文献（如 costattribution 提出的四层令牌会计框架）更关注生产系统的大规模多租户计费，而该工具关注的是单开发者单会话的微观诊断。两者互补：前者回答团队的钱花在哪里，后者回答为什么这一轮对话消耗了这么多令牌。

一个值得延伸思考的方向是：能否将该工具的解析逻辑与 Langfuse 等开源可观测性平台的自托管实例结合，实现本地深度解析与团队级聚合的混合架构？这需要解决日志格式标准化、多机器数据同步和隐私边界划分等问题。

此外，子代理执行树与成本分解的设计直接回应了多代理系统研究中日益严重的成本黑箱问题。随着 Agentic Coding 基准测试显示前沿模型单次任务消耗超过 100 万输入令牌，能够逐代理、逐工具地追踪成本变得至关重要。该工具虽未提供定量实验，但其可视化范式可为未来的多代理系统成本优化研究提供交互式分析工具参考。
