# The Long-Horizon Task Mirage? Diagnosing Where and Why Agentic Systems Break

## 原文摘要翻译

大语言模型智能体在短程和中程任务上表现强劲，但在需要长时间、相互依赖动作序列的长程任务上经常崩溃。尽管智能体系统发展迅速，这些长程失败模式仍然缺乏系统刻画，阻碍了对失败原因的有原则诊断以及跨领域的公平比较。为填补这一空白，本文提出 HORIZON，一个初始的跨领域诊断基准，用于系统性地构造任务并分析基于大语言模型的智能体的长程失败行为。利用 HORIZON，作者评估了来自多个模型家族的最先进智能体，在四个代表性智能体领域中收集了 3100 余条轨迹，研究随任务视界延长而产生的性能退化模式。此外，作者提出了一种基于轨迹的大语言模型裁判流程，用于可扩展且可复现的失败归因，并通过人工标注轨迹进行了验证，取得了高度一致性（标注者间 $\kappa=0.61$；人工–裁判 $\kappa=0.84$）。本研究为长程智能体失败的系统性、跨领域分析迈出了方法论上的第一步，并为构建更可靠的长程智能体提供了实践指导。

## 创新点

1. **首个跨领域长程诊断基准 HORIZON**：提出了一套统一的任务视界定义（内在视界 $H^*$ 与组合深度 $s$）和受控的视界扩展方法，使得不同领域（网页、操作系统、数据库、具身）的长程任务可以在同一度量框架下进行比较。
2. **七维失败分类体系**：借鉴失效模式与影响分析（FMEA）思想，将长程失败划分为七类正交维度——环境错误、指令错误、错误假设、规划错误、灾难性遗忘、历史错误累积、记忆限制——覆盖过程级风险（PFMEA）与设计级风险（DFMEA）。
3. **轨迹级 LLM-as-a-Judge 失败归因流程**：设计了可扩展的自动化失败标注流程，并通过人工标注验证其可靠性，人工–裁判一致性达 $\kappa=0.84$。
4. **大规模跨模型实证研究**：在 3100 余条轨迹上系统评估了 GPT-5-mini 与 Claude-4-Sonnet，揭示了长程失败并非简单的成功率下降，而是失败组成随视界增长发生结构性转移。

## 一句话总结

本文提出跨领域诊断基准 HORIZON，通过统一视界度量、七维失败分类体系和 LLM-as-a-Judge 归因流程，系统揭示了当前最先进智能体在长程任务中的性能崩溃规律与失败机制转移，指出仅靠基础模型缩放不足以解决规划错误和灾难性遗忘等主导瓶颈。

## 研究问题

本文围绕两个核心研究问题展开：

- **RQ1（Where）**：随着任务视界增加，智能体在何处出现崩溃？即性能退化的转折区域（breaking region）出现在什么位置，不同领域间有何差异？
- **RQ2（Why）**：这些失败为何产生？即长程任务中的失败模式如何随视界增长而发生组成变化？

现有研究的碎片化使得上述问题难以回答：领域专用基准的视界定义互不兼容，评估往往只报告聚合成功率，缺乏轨迹级失败归因，导致跨领域比较和系统性诊断缺乏共同基础。此外，长程行为本质上是领域依赖的——例如，具身智能体可能在仅需少量连续动作的任务上就出现断崖式崩溃，而网页导航智能体在同等步数下仍保持稳健——这使得单一普适的"断裂点"定义不可能存在。

## 数据与任务定义

### 视界定义的理论层

为避免将任务复杂性与智能体低效率混淆，HORIZON 提出两层定义方法：理论指标刻画任务结构，技术实现通过系统构造进行可操作化。

**Intrinsic Horizon（$H^*$）**：最优策略完成该任务所需的最少有效动作数。例如，网页任务"查找并购买 200 美元以下的无线耳机"可分解为 $H^*=8$：

$$H^* = |\{a_1: \text{search}, a_2: \text{filter price}, a_3: \text{sort rating}, \ldots, a_8: \text{checkout}\}|$$

**Compositional Depth（$s$）**：嵌套子目标或条件分支的最大数量，反映超越线性执行的规划复杂度：

$$s = \max_{p \in \text{paths}} |\text{decision\_nodes}(p)|$$

例如，"若条件甲成立则执行计划乙，否则执行计划丙，然后合并结果"的任务具有 $s=3$，而纯顺序工作流则 $s=1$。

### 视界扩展的技术实现

**深度扩展（Depth Extension）**：在现有动作之间逐步插入中间步骤。对基线任务 $s_{\text{base}}=1$，扩展到层级 $s$ 产生：

深度扩展公式可表述为：内在视界等于基线视界加上一个随层级单调递增的增量项。

$\Delta(s)$ 的增长函数取决于领域结构：OS 领域每次扩展通常增加 1–2 个权限检查；Database 领域每次扩展可能增加 2–3 个过滤操作。该方法适用于初始环境状态固定的领域（OS、Database）。

**广度扩展（Breadth Extension）**：将 $k$ 个独立的基线任务组合为单一复合工作流。对基线任务 $\{t_1, \ldots, t_k\}$，每个 $s_i=1$，组合后的任务扩展层级 $s=k$：

$$H^*_{\text{breadth}}(s) = \sum_{i=1}^{s} H^*_i + \epsilon(s), \quad \epsilon(s) \geq 0$$

协调开销 $\epsilon(s)$ 涵盖任务切换、跨任务验证和子任务间状态追踪等额外动作。该方法更适合初始状态可变的领域（Web、Embodied）。

### 评估领域与环境

| 领域 | 基础环境 | 扩展方式 | 特点 |
|------|----------|----------|------|
| Web Navigation | WebArena | 广度扩展 | 多页面、表单、验证步骤构成的长依赖链 |
| Operating Systems | AgentBench | 深度扩展 | 权限检查、命令序列的逐步插入 |
| Databases | MAC-SQL | 深度扩展 | Selector–Decomposer–Refiner 模块化流水线 |

## 方法主线

HORIZON 的整体方法可概括为"度量–扩展–归因"三步闭环：

- **统一度量**：用 $H^*$ 和 $s$ 为不同领域任务提供可比的结构化复杂度指标；
- **受控扩展**：通过深度或广度扩展系统性地增加任务视界，构造嵌套任务集（层级 $h+1$ 包含层级 $h$ 的全部任务并增加更长执行需求），确保相邻层级间的比较是受控的；
- **轨迹归因**：对失败轨迹应用七维分类体系进行结构化标注，通过大语言模型裁判实现规模化，并以人工标注验证可靠性。

### 机制流程

![Fig. 2 HORIZON 诊断流水线](p-016_The_Long-Horizon_Task_Mirage_Diagnosing_/images/fig2_horizon_pipeline.png)
*Figure 2：HORIZON 诊断流水线。该流水线包括轨迹与上下文收集、分类体系开发与标注者间一致性验证、HORIZON 诊断套件构建（统一视界度量与失败归因）、LLM 标注器校准、以及大规模失败标注五个阶段。*

如图 2 所示，HORIZON 诊断流水线包含以下五个阶段：

- **轨迹与上下文收集**：在四个领域环境中运行智能体，收集完整执行轨迹，包括每步的观察、动作、环境状态变化和内部记忆内容；
- **分类体系开发与一致性验证**：基于 FMEA 理论框架和结构化文献综述构建七类失败分类体系，两名专家标注员对 40 条共享失败轨迹进行独立标注，计算标注者间一致性（$\kappa=0.61$）；
- **诊断套件构建**：将统一视界度量与失败归因整合为 HORIZON 诊断套件，支持水平（视界增长）与垂直（失败归因）两个维度的分析；
- **大语言模型标注器校准**：将分类体系编码为大语言模型裁判的提示模板，在 40 条轨迹上与人工标注对比校准，验证人工–裁判一致性（$\kappa=0.84$）；
- **大规模失败标注**：利用校准后的大语言模型裁判对 3100 余条轨迹进行规模化失败归因，生成跨领域、跨模型的结构化失败统计。

### 七维失败分类体系

该体系借鉴 FMEA 的风险建模视角，将长程失败划分为两个交互层级：

- **过程级风险（PFMEA，约 72.5%）**：在顺序执行过程中产生——环境交互、指令遵循、规划错误、历史错误累积；
- **设计级风险（DFMEA，约 27.5%）**：反映工程化智能体的架构局限——记忆限制、灾难性遗忘、错误假设。

各类别正交而非互斥：同一条失败轨迹可能同时表现出灾难性遗忘（丢失约束）、错误假设（误解环境状态）和历史错误累积（早期错误被放大）。这种多维视角使得诊断可以精确指向干预策略——例如，若规划失败根源于灾难性遗忘，则应引入显式约束追踪而非单纯增加规划训练数据。

## 关键结果

### 主结果：性能随视界的非线性退化

![Fig. 3 模型性能随视界扩展的变化](p-016_The_Long-Horizon_Task_Mirage_Diagnosing_/images/fig3_horizon_performance.png)
*图 3：当前模型性能随视界扩展的变化。图中展示成功率与组合深度 $s$ 的关系，每个数据点报告三次独立运行的均值，误差条为标准差。评估覆盖 GPT-5-mini 与 Claude-4-Sonnet 在所有四个领域上的表现。*

如图 3 所示，跨所有领域观察到三个一致的退化模式：

- **性能随 $s$ 非线性下降**：在较小 $s$ 时成功率相对稳定或缓慢下降，并非与每增加一个子任务成比例衰减；
- **超过小 $s$ 后出现断崖式下跌**：成功率从部分稳健突然过渡到近系统性失败，而非在整个 $s$ 范围内平滑退化；
- **相同扩展层级在不同领域诱导显著不同的任务难度**：网页领域在很小 $s$ 时即崩溃，操作系统和数据库领域在中等扩展层级前保持中等性能，具身任务即使 $s$ 极小幅度增加也急剧退化。

此外，在 Web、OS 和 Database 领域，一旦进入断裂区域，模型间性能差距大幅收窄，成功率趋近于低值，表明长程失败主导后模型区分度下降。

### 失败分布的结构性转移

![Fig. 4 四个领域的失败模式分布](p-016_The_Long-Horizon_Task_Mirage_Diagnosing_/images/fig4_failure_distribution.png)
*Figure 4：3100 余条失败轨迹在四个任务领域中的失败模式分布。柱状图展示各失败类型占该领域总失败轨迹的比例。失败按过程级风险（PFMEA，72.5%）与设计级风险（DFMEA，27.5%）分组。*

如图 4 所示，跨领域的失败分布呈现出一致的模式：规划相关失败（尤其是子规划错误）和记忆相关失败（灾难性遗忘、记忆限制）在长程设置中占据主导地位。这些失败模式具有强烈的轨迹依赖性——早期子规划偏差会传播到后续动作，而上下文负载增加则会加剧记忆和约束保持的失败。

| **灾难性遗忘 [长程]** | 初始应用"仅全新"过滤，长导航后向购物车添加"翻新"商品 | 早期被指示不修改敏感文件，后期执行脚本删除根目录配置 | 开放世界游戏中已学会制作木石工具，在高级任务中不重用早期知识 | 用户指定"绝不触碰支付数据结构"，后期自动清理步骤提议删除支付表索引 |
| **历史错误累积 [长程]** | 元素点击失败后跨多步重复相同失败动作 | 静默失败后继续执行依赖命令，早期错误状态在工作流中传播 | 选择错误初始子任务后继续执行下游步骤，逐步放大初始错误 | 早期中间表遗漏已删除标志过滤，下游分析复用该表导致系统性偏差 |
| **Memory Limitation [L]** | 滚动浏览贡献者列表时因交互历史超出上下文窗口丢失早期提交计数 | 早期文件所有权通知因长交互会话被截断 | 长程 Minecraft 任务中早期导航决策因轨迹超过记忆容量被摘要后丢失 | 仓库含数百张表时，记忆检索返回不完整的 schema snapshot |

> 注：[L] 表示主要长程特定的失败类别（在短程设置中罕见）；[S] 表示在短程和长程任务中均可观察到、但在更长视界下往往被放大的类别。

## 深度分析

### 长程失败的结构性本质

本文最重要的发现是：长程失败**不仅仅是成功率的下降，而是失败组成的结构性转移**。随着视界增长，原本可恢复的局部错误逐渐演变为不可逆的轨迹级脱轨。早期子规划错误具有路径依赖性——一旦在初始阶段选择了错误的分解方式，后续动作空间被持续约束，回滚成本指数级增长。

灾难性遗忘的发生机制也值得深入理解。作者通过 OpenClaw 真实案例（Policy Override）指出，这种遗忘并非字面意义上的记忆丢失，而是早期指令虽然仍在上下文窗口中，但在后续生成步骤中不再被注意力机制有效关注。这说明即使模型具有长上下文能力，**有效注意力分配**仍是瓶颈。

### 与真实世界失败的映射

作者将 HORIZON 分类体系与 OpenClaw 邮件智能体的真实失败事件（来自 Shapira 等人 2026 年的相关研究）进行了映射验证：

| HORIZON 类别 | OpenClaw 案例 | 机制 |
|-------------|--------------|------|
| Environment Disturbance | Concurrent Modification | 云同步进程中途重命名文件夹，agent 将"文件未找到"误解为瞬态错误 |
| Instruction Error | Forwarded Inbox | 对语义等价的"share"和"forward"指令不一致解释，导致敏感信息泄露 |
| Planning Error | Infinite Loop | 两个 agent 相互无限转发消息，缺乏终止检查或循环检测 |
| Catastrophic Forgetting | Policy Override | 数百轮后违反"绝不回复外部域名"的初始约束 |
| False Assumption | Identity Hijack | 攻击者伪造身份，agent 未验证即执行特权命令 |
| History Error Accumulation | Guilt Trip | 交互历史逐渐侵蚀 agent 的拒绝边界 |
| Memory Limitation | Storage Exhaustion | 重复存储大附件导致服务器拒绝服务，agent 缺乏对自身累积存储行为的持久记忆 |

这一映射初步验证了 HORIZON 分类体系在真实世界部署中的适用性。

### 跨领域差异的启示

相同扩展层级 $s$ 在不同领域诱导截然不同的任务难度，这反映了领域内在特性的深刻差异：

- **Web**：动作语义高阶但环境动态性强（页面渲染、弹窗、网络延迟），长依赖链跨越多个页面和验证步骤，对状态一致性要求极高；
- **OS**：命令执行确定性强，但权限和安全约束的嵌套深度可无限增加；
- **Database**：SQL 语义精确，但 schema 复杂度和多表关联的协调开销随 $s$ 快速膨胀；
- **Embodied**：物理交互的不可逆性（一旦掉落或放错位置难以恢复）使得即使少量额外步骤也会导致断崖式失败。

这意味着任何试图定义"通用断裂点"的努力都注定是领域依赖的。HORIZON 的价值不在于提供一个普适阈值，而在于建立了一套**可比较、可复现的诊断框架**，使不同领域、不同模型间的相对差异得以被有意义地讨论。

### 失败归因流水线的可靠性

LLM-as-a-Judge 在本研究中达到的人工–裁判一致性（$\kappa=0.84$）高于标注者间一致性（$\kappa=0.61$），表面上似乎反直觉。作者解释这是因为 LLM 裁判在应用结构化分类体系时比人类标注者更"刻板"——严格按提示中的定义和边界进行判断，而人类标注者可能受领域先验影响而产生分歧。这一特点在诊断场景下反而是优势：一致性比人类直觉的细微差异更有价值。

## 局限

1. **评估规模与模型覆盖**：当前评估仅覆盖 GPT-5-mini 和 Claude-4-Sonnet 两个模型家族，更多近期模型（如 DeepSeek-V3、Gemini 系列）未纳入比较，限制结论的普适性。
2. **任务构造的人工成分**：HORIZON 的深度/广度扩展依赖人工设计插入步骤和组合任务，自动化程度有限，扩展到更多领域需要大量领域专家参与。
3. **模拟环境与真实部署的差距**：Isaac Sim 中的双机械臂操作与真实物理环境存在 sim-to-real 差距；WebArena 等环境也无法完全复现真实网页的动态变化。
4. **LLM-as-a-Judge 的潜在偏差**：虽然人工验证显示高一致性，但 LLM 裁判仍可能继承训练数据中的偏见，对某些类型的失败（如涉及社会偏见或文化特定假设）的敏感度可能与人类不同。
5. **分类体系的完备性**：七维分类体系基于现有文献和 FMEA 框架构建，随着新类型智能体（如多智能体系统、工具使用链更复杂的 agent）出现，可能需要扩展。

## 我的笔记

HORIZON 的核心方法论贡献在于将"长程任务"从一个模糊的直觉概念转化为可操作、可测量的结构化对象。通过区分 $H^*$（最优策略所需步数）与 $s$（组合深度），作者巧妙地将任务复杂度与智能体效率解耦，使得"agent 做了 50 步但反复失败于同一子任务"的情形不会被误判为长程问题。

从实际应用角度，本文的发现对 agent 系统设计具有直接指导意义：

- **规划模块**：应引入层级子规划和执行时验证，早期子规划的微小偏差代价极高；
- **记忆模块**：需要超越简单上下文窗口的机制，实现长程约束的显式追踪和主动再激活；
- **失败恢复**：应设计中间状态检查和回滚机制，防止局部错误累积为轨迹级失败。

一个有趣的开放问题是：本文提出的七维分类体系与 agent trace 的自动分析方法（如 AgentPex、p-004）如何结合？HORIZON 提供的是高层语义分类，而 AgentPex 提供的是执行轨迹中的细粒度违规检测。两者结合可能形成从"检测异常"到"解释根因"的完整诊断链路。
