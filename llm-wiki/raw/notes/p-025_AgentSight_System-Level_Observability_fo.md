## 原文摘要翻译

现代软件基础设施日益依赖大语言模型智能体进行开发与维护，例如 Claude Code 与 Gemini-cli。
然而，这类 AI 智能体与传统确定性软件存在根本差异，给常规监控与调试带来了重大挑战。
这催生了一个关键的语义鸿沟：现有工具只能观察到智能体的高层级意图（通过大语言模型提示词）或低层级动作（例如系统调用），却无法将这两种视角关联起来。
这种盲区使得区分良性操作、恶意攻击与代价高昂的运行故障变得异常困难。
本文提出 AgentSight，一种基于混合方法的 AgentOps 可观测性框架，用以弥合这一语义鸿沟。
我们的方法称为边界追踪（boundary tracing），它利用 eBPF 在稳定系统接口处从应用代码外部对智能体进行监控。
AgentSight 拦截经 TLS 加密的大语言模型通信以提取语义意图，同时监控内核事件以观察系统级影响，并通过实时引擎与次级大语言模型分析，将这两条事件流在跨进程边界上进行因果关联。
该技术无需插桩、与框架无关，能够抵御快速 API 变更，且性能开销低于百分之三。
评估表明，AgentSight 可检测提示注入攻击、识别浪费资源的推理循环，并揭示多智能体系统中隐藏的协调瓶颈。
AgentSight 已作为开源项目发布。

## 创新点

1. **边界追踪观测范型**：提出从稳定系统边界（内核接口与网络接口）对智能体进行无插桩监控的原则性方法。相比应用层插桩方案，边界追踪不依赖智能体框架内部实现，不会因框架 API 频繁变更而失效，也不会被单条 shell 命令绕过。

2. **混合双阶段关联引擎**：设计了一个实时启发式关联引擎与次级大语言模型语义分析相结合的两阶段处理管线。第一阶段利用进程谱系、时间邻近性与参数匹配实现毫秒级因果关联；第二阶段借助大语言模型对关联后的轨迹进行深层语义分析，从而突破基于规则的脆弱检测限制。

3. **基于 eBPF 的统一探针体系**：利用 eBPF 同时在用户态（SSL 库函数）和内核态（tracepoints / kprobes）部署探针，以单一技术栈捕获解密后的意图流与系统动作流，实现生产安全、低开销的全栈可观测。

4. **面向真实场景的威胁与病理检测**：在提示注入攻击、推理循环与多智能体协调瓶颈三类真实问题上验证有效性，且运行时开销控制在百分之三以内，证明方案在生产环境中的实用性。

## 一句话总结

AgentSight 通过 eBPF 在稳定系统边界上实施边界追踪，将大语言模型通信的语义意图与内核系统调用因果关联，以低于百分之三的开销实现对 AI 智能体的无插桩、跨进程、语义感知可观测。

## 研究问题

AI 智能体将大语言模型的推理能力与直接操作系统工具的能力耦合，使其能够生成动态代码、派生任意子进程并执行命令。这种非确定性执行路径与传统软件的可预测逻辑截然不同，导致现有可观测性工具面临根本性盲区。

具体而言，当前方案被困在语义鸿沟的某一侧：

- **应用层插桩**（如 LangChain、AutoGen）只能捕获智能体的高层级意图和工具选择，却无法看到 shell 子进程中的系统动作；它们依赖协作式信任模型，一条 shell 命令即可逃逸监控。
- **通用系统层监控**（如 Falco、Tracee）能够追踪每次系统调用和文件访问，但缺乏语义上下文；对这类工具而言，智能体编写数据分析脚本与编写恶意载荷在系统调用层面毫无区别。

因此，核心研究问题为：**如何在不侵入智能体应用代码、不依赖特定框架的前提下，将智能体的高层级语义意图与其低层级系统动作进行因果关联，从而实现对良性操作、恶意攻击与性能病理的有效区分？**

## 数据与任务定义

本文的评估任务围绕三类真实开发者工作流与三类安全及性能病理场景展开：

**性能基准任务**（基于一个教程仓库）：
- 仓库理解：使用 `/init` 命令让智能体熟悉代码库结构；
- 代码生成：生成 bpftrace 脚本；
- 仓库编译：执行并行构建。

**案例研究任务**：
- **间接提示注入攻击检测**：智能体被指示克隆并构建一个 C 项目，项目的 README 将其引导至一个藏有恶意提示的网页，该提示诱使智能体读取 `/etc/passwd` 并外传到收集服务器；
- **推理循环检测**：使用 crewai 与 gpt-4o-mini 搭建的研究智能体反复以错误参数调用网页搜索工具，陷入尝试—失败—再推理的死循环；
- **多智能体协调监控**：对六个协作开发智能体（基于 claude-code subagents）进行监控，观察文件锁竞争与顺序依赖导致的协调瓶颈。

评估环境为一台 Ubuntu 22.04 服务器，内核版本 6.14.0，测试智能体使用 Claude Code 1.0.62（claude 4）。

## 方法主线

AgentSight 的设计围绕一个核心目标：弥合智能体意图与动作之间的语义鸿沟。其方法论基础是边界追踪，通过多信号关联引擎实现。

### 核心挑战

**挑战一：跨越意图与动作之间的语义鸿沟**
智能体的意图以自然语言表达，经大语言模型解释后动态生成运行时源码。
静态分析无法预判其行为。
例如"查找并修复认证模块中的缺陷"这一意图，可能转化为读取文件、编译代码、运行测试等复杂系统调用序列。
监控系统必须超越简单模式匹配，获得对智能体目标的语义级理解。

**挑战二：从高容量系统噪声中提取因果信号**
智能体可自主调用任何工具，产生大量不可预测的系统事件。
静态预配置过滤器（如仅监控 git 命令）本质上脆弱不堪，一旦智能体改用 curl 与 bash 即告失效。
AgentSight 采用激进的内核态 eBPF 动态过滤：通过追踪进程创建事件构建完整的智能体活动谱系树，并动态向内核下发规则，仅将智能体及其子孙进程的事件传递至用户态，从而在源头捕获完整因果链，显著降低额外开销。

### 边界追踪的设计原则

核心洞察在于：所有智能体交互都必须穿越定义明确且稳定的系统边界——内核（系统操作）与网络（外部通信）。
在这些边界处监控，而非在多变的智能体代码内部监控，可实现与实现细节无关的全面观测。
该方案依托两条原则：

- **全面性（Comprehensiveness）**：内核级监控确保从进程创建到文件 I/O 的任意系统动作均不被遗漏，即使跨越派生子进程；
- **稳定性（Stability）**：系统调用 ABI 与网络协议的演化速度远低于智能体框架，提供耐用、面向未来的解决方案。

这一范型将信任模型从"假设智能体是协作的"转变为"在防篡改边界上强制执行观测"。

### 系统架构：观测边界

AgentSight 的架构同时接入两条关键边界。
如图 2 所示，系统使用 eBPF 部署非侵入式探针：在用户态 SSL 函数上捕获解密后的意图流（大语言模型提示词与回复），在内核态捕获动作流（系统调用、进程事件）。
用户态关联引擎随后将这两股流处理并融合为统一、因果关联的轨迹。

![Fig. 2 AgentSight System Architecture](p-025_AgentSight_System-Level_Observability_fo/images/fig2_agentsight_system_architecture.png)

关键组件包括：

**eBPF 统一探针**：选择 eBPF 是因为其生产安全性、高性能，以及同时访问用户态与内核态数据流的统一能力。设计直接拦截智能体与 LLM 服务后端交互时的解密数据，相比网络层抓包或代理方案更高效、更易管理。

**多信号因果关联引擎**：关联策略依赖三种机制建立因果性：
- 进程谱系（Process Lineage）：通过追踪 fork 与 execve 事件构建完整进程树，将子进程动作关联回父智能体；
- 时间邻近性（Temporal Proximity）：将大语言模型回复后窄时间窗口内发生的动作进行关联；
- 参数匹配（Argument Matching）：直接将大语言模型回复中的内容（文件名、URL、命令）与后续系统调用的参数进行匹配。

**大语言模型驱动的语义分析**：为摆脱脆弱的规则式检测，系统引入次级大语言模型作为推理引擎。将关联后的事件轨迹以结构化日志形式输入模型，利用其语义理解能力推断复杂场景中的因果性与风险。这种"以 AI 监控 AI"的方式使 AgentSight 能够检测不符合预定义模式的新型威胁。

### 实现细节

AgentSight 以用户态守护进程形式实现（约六千行 Rust/C 代码），负责编排 eBPF 程序，并配以约三千行 TypeScript 前端用于可视化分析。

**边界数据采集**：
- 意图采集：eBPF 程序通过 uprobes 附加到 OpenSSL 等加密库的 SSL_read / SSL_write 函数，拦截解密后的大语言模型通信。用户态守护进程实现有状态重组机制，处理 Server-Sent Events 等流式协议；
- 动作采集：第二组 eBPF 程序使用 sched_process_exec 等稳定 tracepoints 构建进程树，并通过 kprobes 动态监控 openat2、connect、execve 等相关系统调用。内核态激进过滤确保仅目标智能体进程的事件被上传，最小化开销。

**混合关联引擎**：Rust 用户态守护进程承载两阶段关联引擎。第一阶段消费 eBPF ring buffer 中的事件，执行实时启发式关联：对原始事件进行上下文增强（如将文件描述符映射为完整路径）、维护有状态进程树、应用因果关联逻辑（时间窗口为 100–500 ms）。一旦构建出连贯轨迹，第二阶段将其格式化为结构化日志，构造详细提示词交给次级大语言模型扮演安全分析师。模型返回的自然语言分析与置信度分数即为系统最终输出。

### 机制流程

1. **输入：边界双流采集**。eBPF 探针同时捕获两条原始数据流：一条来自用户态 SSL 函数的解密意图流（大语言模型提示词与回复），另一条来自内核 tracepoints 与 kprobes 的动作流（系统调用、进程事件、网络连接）。内核态积极过滤仅保留目标智能体进程及其子孙的事件，从源头降噪。

2. **提取与融合：实时关联增强**。Rust 守护进程从 ring buffer 消费事件，利用进程谱系构建动态进程树，提取时间邻近性窗口（100–500 ms）与参数匹配信号，将意图与动作进行对齐融合。此阶段还对原始事件进行上下文增强，例如将文件描述符解析为完整路径、将进程 ID 映射为命令行与父进程关系，生成结构化的因果轨迹。

3. **输出：次级大语言模型语义分析**。关联引擎将结构化轨迹格式化为时间顺序日志，构造安全分析师角色的提示词，送入次级大语言模型（如 GPT-4 或 Claude）。模型输出自然语言形式的风险分析、根因推断与置信度评分，作为最终可观测结果交付给用户或下游告警系统。

### 案例研究

**案例一：间接提示注入攻击检测**
测试场景中，软件开发智能体被指示克隆并构建一个 C 项目。
项目说明文档将其引导至一个提供网页的地址，其中嵌入了隐藏提示，诱使智能体读取敏感文件并伪装成构建步骤将其发送至外部服务器。
AgentSight 捕获了完整的关联攻击链：从初始地址获取到最终网络渗出，原始 521 个事件经关联引擎合并为 37 个事件。
次级大语言模型分析该轨迹后返回高置信度攻击评分，判定智能体动作与其声明目标逻辑不一致。

**案例二：推理循环检测**
使用 crewai 与 gpt-4o-mini 搭建的研究智能体反复以错误参数调用网页搜索工具，收到错误后未能纠正，不断重试同一失败命令。
AgentSight 的实时监控从接口调用轨迹中检测到异常资源消耗并传递给次级大语言模型。
模型识别出根因为持续工具错误，指出智能体陷入尝试—失败—再推理循环：执行相同失败命令、将相同错误传回推理模型、未能从工具输出中学习。

**案例三：多智能体协调监控**
对六个基于 claude-code subagents 的协作开发智能体进行监控，关联引擎处理后捕获 3153 个总事件。
分析显示，前端智能体与测试智能体有时因顺序依赖和并行开发测试任务期间的文件锁竞争而被阻塞，经历大量重试周期。
虽然智能体展现出一定的涌现协调行为，但更清晰的角色划分可缩短总运行时间与 token 成本。
该案例证明边界追踪能够独特地捕获应用层监控无法跨越进程边界观察到的多智能体系统动态。

## 深度分析

**技术选型的高明之处**：
选择 eBPF 作为统一探针技术是一项关键架构决策。
eBPF 同时覆盖用户态（通过 uprobes 拦截 SSL）与内核态（通过 tracepoints 和 kprobes 监控系统调用），且具备内核强制验证的生产安全性。
相比 SDK 插桩，eBPF 不侵入应用代码；相比网络层抓包，直接拦截解密后的传输层数据避免了密钥管理的复杂性；相比 ptrace，eBPF 的性能开销显著更低。
这种"在边界处观测"的哲学与智能体系统的动态、非确定性本质高度契合。

**关联引擎的设计权衡**：
实时关联阶段采用进程谱系、时间邻近性与参数匹配三重启发式信号，而非等待完整轨迹后再进行全局优化，这一设计体现了流式处理的工程智慧。
100–500 ms 的时间窗口既足以覆盖大语言模型回复后的即时系统响应，又避免了过度宽泛导致的误关联。
然而，该引擎对异步、延迟较大的动作（如后台下载任务完成后数分钟才执行的脚本）的关联能力可能受限。

**以 AI 监控 AI 的双刃剑**：
引入次级大语言模型进行语义分析是突破规则式检测瓶颈的创新手段，但也带来新的依赖与成本。
论文提到系统通过异步处理与提示词工程缓解延迟与成本问题，但未给出详细的 token 消耗、响应延迟与成本数据。
对于高频事件场景，次级模型的调用频率与费用可能成为规模化部署的障碍。

**多智能体观测的独特价值**：
现有 AgentOps 工具大多聚焦单智能体轨迹，AgentSight 的进程谱系追踪天然支持跨子进程、跨智能体的因果关联。
案例三揭示的涌现协调与文件锁竞争问题，正是传统应用层日志完全无法观测到的盲区。
这一能力在多智能体协作开发、自主运维等场景具有重要实用价值。

## 局限

1. **平台依赖**：当前实现面向 Linux（Ubuntu 22.04, Linux 6.14.0），eBPF 虽已开始向 Windows 移植，但跨平台成熟度不足，限制了方案在异构生产环境中的立即适用性。

2. **关联引擎的启发式边界**：实时关联依赖时间窗口与参数匹配的启发式规则，对于高度异步、间接因果或跨长时间窗口的智能体行为，可能产生漏关联或误关联。论文未对关联精度进行定量评估。

3. **次级大语言模型的成本与延迟**：语义分析阶段引入的外部模型调用带来额外延迟与接口费用。论文未披露具体的 token 开销、平均分析延迟及在高频事件流下的可扩展性数据。

4. **加密流量拦截的兼容性**：SSL 探针依赖对特定加密库（如 OpenSSL）的 uprobes 附加，若智能体使用定制化传输层实现或经过混淆的二进制，意图流捕获可能失效。

5. **实验规模有限**：性能评估仅基于三个开发者工作流，案例研究为定性分析，缺乏大规模多框架、多模型、长时间运行的定量基准测试。

## 我的笔记

AgentSight 的工作让我意识到，AI 智能体的可观测性不能仅在应用层或系统层单点解决，而必须在两者之间的语义鸿沟上架桥。
边界追踪这一思路具有很强的通用性：不仅适用于大语言模型智能体，也可推广到任何将高层语义意图转化为低层系统动作的非确定性系统。
eBPF 作为内核中的可编程探针技术，恰好提供了跨越用户态与内核态的统一观测能力，这是其他技术难以替代的。

从复现角度，AgentSight 的开源发布降低了验证门槛，但其工程复杂度不容小觑：传输层数据重组、内核事件过滤策略、多信号关联的调参、提示词工程，每一环都需要对内核网络栈与智能体框架有深入理解。
特别值得关注的是关联引擎中 100–500 ms 时间窗口的选择——这一参数对不同工作负载的敏感性可能成为实际部署中的调优重点。

与相关工作对比，Langfuse、LangSmith 等工具在应用层做得足够好，但面对 shell 逃逸无能为力；Falco、Tracee 等系统层工具看到一切动作却不懂意图。
AgentSight 的混合架构试图兼顾两者，但这也意味着它同时继承了两边的部分复杂性。
未来如果能将关联引擎与 OpenTelemetry 等现有可观测生态打通，并引入机器学习自动发现异常行为模式（而非仅依赖次级大语言模型），可能会进一步提升实用性与可扩展性。
