## 内容摘要

阿里云百炼（Model Studio）是阿里云面向企业和开发者提供的大模型服务平台，
涵盖智能体应用、工作流应用和高代码应用等多种应用形态。由于百炼应用内部
采用多节点架构（如向量生成、向量检索、大模型调用等），开发者在后续运维
和优化过程中面临诸多挑战：如何追踪应用内部的复杂调用链路？如何准确查看
模型响应延迟的分布情况？如何深入理解模型的思考过程？为了解决这些痛点，
百炼平台内置了"应用观测"功能，允许用户端到端查看业务空间内应用的处理
流程，并获取延迟、Token 量等关键性能指标，数据更新频率为分钟级。需要注意
的是，应用观测功能目前暂无 API，主要通过控制台界面进行操作。

应用观测功能的使用前提是完成三项初始化配置：第一，授权可观测链路
OpenTelemetry 服务角色权限，使百炼平台能够将 Trace 数据写入阿里云
可观测体系；第二，开通可观测链路 OpenTelemetry 服务，建立 Trace 数据的
采集与存储通道；第三，初始化可观测链路 OpenTelemetry 存储 LogStore，
为后续数据查询与分析提供持久化存储。上述操作需要使用阿里云主账号执行，
开通后通常分钟级生效，高峰期可能稍有延迟。若使用子账号，则需主账号预先
配置必要的权限策略。这一前置流程反映了云原生可观测性方案的典型特征：
观测能力的启用往往涉及跨产品的服务授权和资源初始化，需要在平台设计和
用户体验上做好引导，否则容易成为用户采纳的隐性门槛。

在具体使用方法上，应用观测遵循"选择应用 → 自动追踪 → 查看详情 → 导出数据
→ 监控统计"的五步流程。用户首先在应用观测页面添加被观测的应用，未发布的
应用或不在当前业务空间的应用不会出现在列表中。添加后该应用中的所有 Prompt
输入及相应数据和指标将被自动追踪并同步至应用观测。用户可随时关闭观测以
停止数据同步，重新添加后仅同步新增数据，历史数据不会回溯补录。在应用观测
列表中，用户可查看应用总量、应用平均延时等运营和成本优化指标。点击
"查看详情"后，系统展示最长 30 天内的所有 Prompt 内容、输出、延时、调用
时间以及 Token 量等概要信息。

列表中的 CHAIN 节点表示一次完整的应用内部调用追踪，支持展开查看子节点，
状态分为正常和错误两种。系统支持基于 Request ID、Trace ID 或 Span ID
进行精准搜索，也支持按指定时间范围筛选。如何获取这些 ID 的交互设计也体现了
产品的易用性考量：单击指定节点的名称（如 AgentApp），展开节点详情后再单击
"查看 ID"，即可查看 Request ID、Trace ID 和 Span ID 的完整信息。表头支持
自定义编辑，用户可根据需要调整显示的字段列。点击任意节点名称可查看详情、
原始数据和标注记录，若存在下属节点也支持逐层展开，形成树状的调用层级视图。

数据导出功能允许用户在 Trace 列表页签将当前筛选条件下的数据导出为 JSONL
或 EXCEL 格式，便于离线分析、团队共享和第三方工具处理。监控统计页签则提供
更宏观的性能监控视图，包括调用次数趋势图、失败次数和失败率统计、Token 总量
趋势（含全部、输入、输出）、平均单次请求 Token 量、平均首 Token 耗时（针对
流式调用场景），以及平均调用时长。所有监控图表支持按时间范围（最长 30 天）
和聚合粒度（分钟、小时、天）查看，并支持放大、下载和复制操作，兼顾了工程
排查和汇报展示两种使用场景。

此外，应用观测还提供了两个与数据质量闭环密切相关的高级功能。一是
"添加到评测集"：用户可将任意 Span 数据直接添加至评测集，用于后续的应用
评测，将真实线上调用数据转化为可复现的评估样本。二是"数据标注"：支持对
观测到的数据进行人工标注和反馈，为模型迭代和 Prompt 优化提供高质量的人工
监督信号。这两个功能的存在表明，百炼团队已经意识到"观测"只是第一步，将
观测数据转化为可行动的改进输入才是最终目标。这种从"看见"到"行动"的思维
跃迁，正是当前 AI 可观测性领域从"监控工具"向"优化平台"演进的具体体现。

在支持的节点类型方面，百炼应用观测覆盖了应用运行过程中的多种关键节点，
使得开发者可以深入了解每个环节的输入输出和执行状态。这种细粒度的节点级
观测能力，对于诊断复杂的多步骤 AI 应用尤为关键，因为问题往往出现在某个
特定的中间环节（如向量检索返回了不相关的结果，或工具调用传入了错误的
参数），而非最终的大模型生成环节。能够逐层展开查看每个子节点的详情，是
百炼应用观测相比简单日志输出的核心优势所在。

## 关键要点

1. **端到端多节点追踪**
   - 百炼应用观测覆盖向量生成、向量检索、大模型调用等全部内部节点。
   - 以 CHAIN → Span 的层级结构呈现，解决了多节点架构下"黑盒"运行的
     观测难题。
   - 每个节点的输入输出、延迟、Token 消耗均可独立查看，支持逐层展开的
     树状视图。
   - 状态标记（正常 / 错误）帮助快速识别异常节点，缩短故障定位时间。
   - 多节点追踪能力是理解复杂 AI 应用行为的必要条件，单节点日志往往
     无法揭示跨组件的交互问题。

2. **OpenTelemetry 标准集成**
   - 底层基于阿里云可观测链路 OpenTelemetry 服务构建，采用行业统一的
     Trace/Span 语义模型。
   - 兼容云原生可观测生态，便于与现有运维体系对接。
   - 这种标准化选择降低了厂商锁定风险，也为后续跨平台数据关联分析
     奠定了基础。
   - 通过 SLS LogStore 进行数据持久化，展示了 OpenTelemetry 与日志服务
     的典型云原生结合方式。
   - 采用开放标准也意味着团队内部已有的 OpenTelemetry 知识和工具可以复用。

3. **分钟级实时同步**
   - 观测数据以分钟级频率自动同步，在实时性和系统开销之间取得了较好的
     平衡。
   - 关闭观测后数据停止同步，重新添加仅同步新增数据，体现了对存储成本
     和用户控制权的尊重。
   - 满足生产环境日常监控需求，但对于需要秒级甚至毫秒级响应的实时告警
     场景可能需要结合其他工具。
   - 分钟级同步对于大多数 AI 应用场景而言已经足够，因为 LLM 调用本身的
     耗时通常在秒级。

4. **多维度性能指标**
   - 覆盖调用次数、失败率、Token 总量（输入/输出/全部）、平均单次请求
     Token 量。
   - 首 Token 耗时（流式场景下的用户体验指标）、平均调用时长等 LLM 应用
     特有的关键指标。
   - 为成本优化（FinOps）提供数据基础，帮助企业精细化管控模型调用支出。
   - 这些指标反映了 LLM 应用与传统后端服务在性能建模上的本质差异。
   - Token 相关指标对于成本控制尤为重要，因为 Token 消耗直接决定了
     模型调用的费用。

5. **精准定位与搜索能力**
   - 支持基于 Request ID、Trace ID、Span ID 的精确搜索。
   - 支持时间范围筛选和表头自定义，支持 Span 筛选模式和过滤器添加。
   - 在大规模数据场景下能够快速定位目标调用记录，降低故障排查的时间成本。
   - ID 获取的交互设计（点击节点 → 查看详情 → 查看 ID）体现了对开发者
     使用路径的深入理解。
   - 自定义表头使用户可以根据关注重点灵活调整信息展示方式。

6. **数据闭环能力**
   - "添加到评测集"：将观测数据直接转化为评估资产，打通"运行观测 →
     质量评估"的通路。
   - "数据标注"：提供人工反馈入口，为模型迭代和 Prompt 优化提供高质量
     监督信号。
   - 这是从"被动监控"走向"主动优化"的关键设计，也是构建数据飞轮的
     基础能力。
   - 评测集的自动积累解决了评估数据稀缺的问题，使持续评估成为可能。

7. **灵活的数据导出与可视化**
   - 支持 JSONL / EXCEL 格式导出，便于离线分析和团队共享。
   - 监控图表支持多时间粒度聚合（分钟、小时、天）和交互式操作
     （放大、下载、复制）。
   - 最长 30 天的历史数据保留满足了一般性的回溯分析需求。
   - 兼顾了工程排查和汇报展示两种使用场景。
   - JSONL 格式特别适合程序化处理和机器学习流水线的下游消费。

8. **应用类型与限制**
   - 当前支持智能体应用、工作流应用和高代码应用。
   - 暂不支持通过 Assistant API 创建的智能体应用，产品仍在持续迭代中。
   - 暂无 API，主要依赖控制台界面操作，这在一定程度上限制了与自动化
     流水线的深度集成。
   - 这些限制反映了不同应用形态在内部架构上的差异对观测方案带来的
     技术挑战。
   - Assistant API 的缺失支持可能与该接口的内部实现方式与标准应用有
     显著差异有关。

## 与综述的关联

百炼应用观测是阿里云在大模型服务平台层面对可观测性能力的直接内建，与
AgentLoop（云监控层面的可观测性平台）形成了"平台内置 + 云监控增强"的
双层观测体系。这一架构设计为综述中"Agent 可观测性的部署形态"子节提供了
典型的工业案例——可观测性既可以作为独立平台产品存在（如 AgentLoop、
Langfuse、Datadog LLM Observability），也可以作为大模型服务平台的原生
能力直接嵌入（如百炼应用观测、Google Vertex AI Agent Builder 的内置 Trace、
百度千帆 AppBuilder 的 Trace 功能）。这两种形态各有优劣：独立平台通常功能
更全面、更灵活，但接入成本较高；平台内置方案接入成本低、使用便捷，但功能
和定制化空间相对有限。

从技术标准角度，百炼应用观测选择基于 OpenTelemetry 实现底层 Trace 采集和
存储，这与综述中"基于开放标准的可观测性框架"方向完全一致。OpenTelemetry
作为 CNCF 毕业项目，正在快速成为 AI 应用可观测性的事实标准，百炼的实践
验证了这一标准在中文大模型应用生态中的适用性。同时，百炼通过 SLS LogStore
进行数据持久化，也展示了在云环境中 OpenTelemetry 与日志服务的典型结合方式，
为综述中"云原生 AI 可观测架构"的讨论提供了具体的技术实现参考。

从功能演进角度，百炼应用观测从最初的基础 Trace 查看，逐步扩展到评测集
集成、数据标注、监控统计等高级功能，体现了工业界对"可观测性不应止步于
看见，而应驱动改进"这一理念的认同。这与 AgentLoop 的数据飞轮设计、
LangSmith 的评估驱动迭代、Phoenix 的实验对比等功能形成了跨产品的趋势
共振，支撑综述中"从可观测到可评估、可优化"的演进主线。值得指出的是，
百炼目前暂无 API，主要依赖控制台操作，这在一定程度上限制了其与企业内部
DevOps 流水线的深度集成，也是未来产品演进的一个观察点。

从用户体验角度，百炼应用观测的交互设计（如 CHAIN 节点的展开收起、ID 的
一键查看、表头的自定义编辑）反映了工业界对"降低可观测性使用门槛"的重视。
学术研究中往往假设使用者具备足够的技术背景来理解和操作复杂的观测数据，
而工业产品必须在易用性和功能性之间取得平衡。百炼的设计选择为综述中
"可观测性的人机交互"讨论提供了实践参考，也提示我们在讨论技术方案时不能
忽视用户体验这一关键变量。

在综述的"中文大模型生态"专题中，百炼应用观测与 AgentLoop 的协同关系也
值得专门分析。阿里云在百炼（应用开发平台）和云监控（运维平台）两个层面
同时布局可观测性，形成了对开发者和运维人员的双重覆盖。这种"双平台策略"
体现了阿里云对 AI 可观测性市场的高度重视，也为其他云厂商的产品规划提供了
参考范式。综述可以从这一案例出发，讨论云厂商在 AI 可观测性领域的布局逻辑
和竞争策略。

## 我的笔记

百炼应用观测文档的价值在于展示了一种"平台原生可观测性"的落地形态。与
独立的可观测性产品（如 Langfuse、AgentLoop、Datadog）不同，百炼将观测
能力直接嵌入到大模型应用开发和运行的核心路径中，用户在无需额外接入任何
第三方 SDK 的情况下即可获得基础的 Trace 和 Metric 能力。这种"开箱即用"
的体验对于降低 AI 应用可观测性的 adoption 门槛具有重要意义，尤其是在中国
市场大量企业开发者尚未建立系统化的 AI 运维体系的背景下。对于许多初次接触
大模型应用的企业而言，百炼的这种内置方案可能是他们接触 Agent 可观测性的
第一入口。

文档中提到的三个初始化步骤（授权、开通、初始化 LogStore）揭示了一个重要
事实：即使是平台原生的观测能力，在云环境中仍然需要跨产品的权限和资源配置。
这提示综述在讨论"可观测性落地成本"时，不应只考虑技术对接成本，还需要将
云平台权限管理、存储资源预置、服务开通流程等组织层面的成本纳入考量。对于
中小企业而言，这些前置步骤可能成为实际使用中的隐性障碍，产品设计上需要
提供更自动化的开通路径，例如一键开通模板或基于 Terraform 的基础设施即代码
方案。

"平均首 Token 耗时"这一指标的出现非常有意思。它是流式调用（streaming）
场景下特有的用户体验指标，直接反映了用户感知到的"响应速度"——即从发送
请求到开始看到模型输出的时间间隔。在传统后端服务监控中几乎没有对应概念，
但在 LLM 应用中它已经成为与 P99 延迟同等重要的核心指标。这类"AI 原生指标"
的涌现，说明可观测性领域正在经历一次由生成式 AI 驱动的范式扩展，综述应当
专门讨论这一趋势，并梳理当前已经出现的 AI 特有指标家族（如首 Token 耗时、
Token 吞吐量、幻觉率、RAG 命中率、上下文窗口利用率等）。

数据标注和评测集集成功能的引入，让我看到了百炼团队在"从观测到迭代"这一
闭环上的深度思考。将线上真实数据一键转为评测样本，是打通"生产环境 →
评估环境"数据通路的关键设计。相比之下，许多开源工具仍然需要用户手动导出
数据再导入评估框架，百炼的内建集成在工程效率上具有明显优势。未来在综述的
"工业实践对比"章节中，可以将百炼的这一设计与 Amazon Bedrock 的评估作业
（Evaluation Jobs）、Google Vertex AI 的评测管道（Evaluation Pipelines）
进行横向对比，分析各家在"数据闭环"设计上的异同，以及这些设计选择背后的
产品策略差异。

最后，百炼应用观测目前暂无 API 的限制值得关注。在现代化的 DevOps 实践中，
API 优先（API-first）是基础设施产品的基本要求。缺乏 API 意味着观测数据难以
被纳入企业的自动化运维体系，也难以与 CI/CD 流水线、自动告警系统、成本分摊
系统等企业内部系统进行深度集成。我推测这一限制可能是产品早期阶段的临时状态，
随着百炼平台的成熟，开放 API 应当会列入路线图。综述在评估各产品的企业就绪度
时，可以将 API 完备性作为一个重要的评估维度，并讨论 API 缺失对可观测性方案
在生产环境中规模化部署的影响。

从更宏观的视角看，百炼应用观测和 AgentLoop 的并存反映了阿里云在 AI 可观测性
领域的全面布局。百炼面向应用开发者提供轻量、便捷的内置观测能力，AgentLoop
面向专业运维团队提供深度、全面的独立观测平台，两者形成了良好的能力互补。
这种"轻量内置 + 专业增强"的双层架构，可能是未来云厂商在 AI 可观测性领域的
标准产品组合模式，值得在综述中作为行业趋势进行分析和预判。

此外，百炼应用观测对"思考过程"的可视化支持也是一个亮点。文档提到可以
"查看模型思考过程"，这对于理解 LLM 的推理链条和诊断逻辑错误至关重要。
当前学术界和工业界都在探索如何提高 LLM 的可解释性，而百炼通过 Trace 层面的
思考过程展示提供了一种工程化的解决方案。这一方向与综述中"Agent 可解释性"
的讨论密切相关，值得进一步展开。综述可以讨论：思考过程的可视化在多大程度上
能够替代传统的可解释性方法？它对于调试复杂 Agent 系统的实际效果如何？
这些问题对于理解可观测性与可解释性之间的边界和交集非常重要。
