## 内容摘要

OpenAI Swarm 是一个由 OpenAI Solutions Team 维护的实验性教育框架。

旨在探索符合人体工学、轻量级的多智能体协调（multi-agent orchestration）模式。

需要特别注意的是，该仓库目前已被官方标记为"由 OpenAI Agents SDK 取代"。

后者是 Swarm 的生产就绪演进版本，包含关键改进并由 OpenAI 团队积极维护。

因此，Swarm 更适合作为理解多智能体编排基础概念的教育资源。

而非生产环境的首选方案。

官方明确建议所有生产用例迁移到 Agents SDK。

Swarm 要求 Python 3.10+ 环境，可以通过 pip 直接从 GitHub 仓库安装。

Swarm 的设计哲学非常明确：让智能体协调变得轻量、高度可控且易于测试。

它通过两个原始抽象（primitive abstractions）来实现这一目标：**Agent** 和 **handoff**（交接）。

其中，Agent 封装了一组指令（instructions）和一组可用函数（functions）。

并能在任何时候选择将对话交接给另一个 Agent。

这两个原语足够强大，可以表达工具与智能体网络之间的丰富动态关系。

使开发者能够在避免陡峭学习曲线的前提下，构建可扩展的真实世界解决方案。

Swarm 强调，虽然人们倾向于将 Agent 拟人化为"做某事的人"。

但它也可以被用来表示由一组指令定义的非常具体的工作流或步骤。

例如一组步骤、复杂检索、单步数据转换等。

这使得 Agent 可以被组合成由相同原语表示的"智能体"、"工作流"和"任务"网络。

这种统一的原语设计降低了概念复杂度，同时也提高了组合灵活性。

Swarm 中的 Agent 与 Assistants API 中的 Assistant 完全不同。

虽然名称相似，但二者没有任何关联。

Swarm 完全基于 Chat Completions API 构建，因此在调用之间是无状态的（stateless）。

这种设计意味着 Swarm 不会像 Assistants API 那样提供托管的线程（thread）和内置的记忆管理与检索功能。

但换来了更高的透明度和可控性。

Swarm 几乎完全在客户端运行，与 Chat Completions API 类似，不在调用之间存储任何状态。

对于需要完全托管线程和内置记忆管理的开发者，Assistants API 是一个更好的选择。

而对于希望学习多智能体编排原理、追求高度定制化和透明执行流程的开发者，Swarm 是一个理想的教育资源。

Swarm 的核心执行入口是 `client.run()` 函数，它在内部本质上只是实例化了一个 OpenAI 客户端。

`client.run()` 的接口设计类比于 Chat Completions API 中的 `chat.completions.create()`。

它接收输入消息并返回响应，且不在调用之间保存状态。

然而，关键的区别在于 `client.run()` 还负责处理 Agent 的函数执行、智能体交接（hand-off）、上下文变量引用。

并能在返回给用户之前执行多轮对话。

其核心执行循环包含五个步骤：

1. 从当前 Agent 获取补全（completion）
2. 执行工具调用并追加结果
3. 在必要时切换 Agent
4. 在必要时更新上下文变量
5. 如果没有新的函数调用则返回

这个循环设计简洁明了，所有状态转换都发生在客户端的单一控制流中，便于理解和调试。

`client.run()` 接收多个参数，包括初始 Agent、消息列表、上下文变量字典、最大对话轮次（默认 `float("inf")`）、模型覆盖字符串、是否执行工具的布尔标志、是否流式输出的标志以及是否启用调试日志的标志。

当 `client.run()` 执行完毕（可能经过多轮 Agent 调用和工具执行）后，它会返回一个 `Response` 对象。

包含对话过程中生成的所有消息（`messages`）、最后处理消息的 Agent（`agent`）以及更新后的上下文变量（`context_variables`）。

开发者可以将这些值（加上新的用户消息）传入下一次 `client.run()` 调用。

从而从上次中断的地方继续交互——这与 `chat.completions.create()` 的状态管理模型非常相似。

Swarm 还支持流式响应（streaming），通过 `delim` 开始/结束事件来标识 Agent 切换的时刻。

以及通过 `debug=True` 启用调试日志。

`run_demo_loop` 函数在 `/swarm/repl/repl.py` 中实现了一个完整的执行循环示例，可供开发者参考和扩展。

在 Agent 的定义层面，Swarm 允许为每个 Agent 指定名称、模型、指令（可以是字符串或返回字符串的可调用对象）、可用函数列表、工具选择策略等。

Agent 的指令会被直接转换为对话的系统提示词（system prompt）。

且任何时候只有当前活跃 Agent 的指令会存在于对话上下文中。

这种"单活跃指令"的设计确保了不同 Agent 之间的职责边界清晰。

避免了多智能体系统中常见的提示词污染问题。

Swarm 的示例仓库包含了多种场景的实现：

- 基础示例（setup、函数调用、交接、上下文变量）
- 分诊智能体（triage_agent）
- 天气查询智能体（weather_agent）
- 航空公司客服多智能体系统
- 支持机器人（support_bot）
- 个人购物助手（personal_shopper）

覆盖了从简单到复杂的多种业务场景，为学习者提供了丰富的参考实现。

## 关键要点

1. **两个核心原语构成全部能力**
   - Agent（封装指令和函数）和 handoff（智能体间交接）
   - 所有复杂行为均由二者组合表达
   - 统一原语降低了概念复杂度，提高了组合灵活性

2. **完全无状态设计**
   - 基于 Chat Completions API，不在调用之间保存任何状态
   - 所有上下文需由调用方显式传递
   - 确保了高透明度和可控性，但增加了调用方的状态管理负担

3. **与 Assistants API 明确区分**
   - Swarm Agent 和 Assistants API 中的 Assistant 只是名称相似
   - 架构上完全不同，前者完全客户端运行，后者由 OpenAI 托管
   - Swarm 不提供线程托管和记忆管理，但换来了完全透明

4. **client.run() 同步执行循环**
   - 获取补全 → 执行工具 → 切换智能体 → 更新上下文 → 返回结果
   - 支持多轮内部对话，所有步骤同步执行
   - 执行流程完全透明，没有隐藏的异步回调

5. **已被 Agents SDK 正式取代**
   - 官方明确建议所有生产用例迁移到 OpenAI Agents SDK
   - 后者是 Swarm 的生产就绪演进版本，包含关键改进
   - Swarm 应仅作为教育资源使用

6. **客户端主导架构**
   - 几乎完全在客户端运行，提供高透明度和可控性
   - 不提供托管的线程或记忆管理
   - 需开发者自行实现持久化和错误恢复机制

7. **流式响应与调试支持**
   - 流式响应通过 delim 事件标识智能体切换
   - `debug=True` 可输出详细日志
   - 便于开发调试和实时监控

8. **丰富的示例场景覆盖**
   - 从基础函数调用到航空公司客服、支持机器人等真实业务场景
   - 每个示例都有完整的 README 说明
   - 适合作为学习材料和项目起点

9. **上下文变量机制**
   - 通过 `context_variables` 字典在智能体之间传递额外状态
   - 函数和智能体指令均可访问和修改
   - 实现轻量级状态共享，但缺乏并发控制

10. **指令即系统提示词**
    - 每个 Agent 的 instructions 直接映射为 system prompt
    - 活跃 Agent 的指令独占对话上下文
    - 避免了多智能体系统中常见的提示词污染问题

11. **显式 handoff 语义**
    - 智能体交接是同步且显式的，通过函数调用实现
    - 便于理解和调试多智能体协作流程
    - 每次 handoff 都是清晰的追踪边界点

12. **max_turns 无限默认**
    - 默认值为 `float("inf")`
    - 允许对话在工具调用和智能体切换之间进行无限轮次
    - 直到没有新的函数调用为止，需注意潜在的无限循环风险

13. **模型覆盖灵活性**
    - 通过 `model_override` 参数可以在运行时覆盖 Agent 使用的模型
    - 实现灵活的模型路由策略
    - 便于实验不同模型对同一任务的表现

14. **工具执行中断能力**
    - 当 `execute_tools` 设为 False 时，Agent 尝试调用函数会立即中断执行
    - 并返回对应消息，便于实现人机协同审批流程
    - 为敏感操作提供了人工介入的钩子

15. **Python 3.10+ 现代语法**
    - 使用现代 Python 类型系统和语法特性
    - 确保代码的简洁性和可维护性
    - 充分利用了模式匹配等 newer 语言特性

16. **100% Python 纯实现**
    - 源码完全由 Python 编写，无外部运行时依赖
    - 易于阅读、修改和理解
    - 符合教育框架的定位，也便于二次开发

## 与综述的关联

Swarm 在综述中代表了**轻量级多智能体编排框架**的早期探索。

其设计哲学和架构选择对理解后续演进（尤其是 OpenAI Agents SDK）具有重要参考价值。

综述中多个来源（如 a-020 OpenAI Agents SDK 官方 cookbook、a-021 智能体可观测性指南）都将 Swarm 作为多智能体系统的入门级案例进行引用。

说明其在概念普及方面的历史地位。

Swarm 的"少即是多"设计哲学也影响了后续框架对核心原语的取舍。

Agents SDK 保留了 Agent 和 handoff 的概念，但增加了 guardrails、tracing 和并行执行等企业级功能。

从可观测性视角看，Swarm 的无状态设计和客户端主导架构使得追踪相对简单。

所有 Agent 切换、工具调用和上下文变更都发生在客户端的 `client.run()` 循环内。

因此可以在客户端侧完整地捕获执行轨迹，而无需处理服务端黑箱。

这与 Assistants API 的托管模式形成了鲜明对比。

后者需要在服务端和客户端之间进行分布式追踪。

综述中提到的 Codex CLI 追踪（a-013, a-014）和 Claude Code 遥测（a-007）虽然目标不同。

但在"客户端捕获完整执行轨迹"这一设计原则上与 Swarm 一致。

对于构建多智能体可观测性方案的开发者而言，Swarm 的显式 handoff 和同步执行模型提供了一个理想的"简单基线"。

用于验证追踪逻辑的正确性。

如果能在 Swarm 上正确捕获和可视化执行轨迹。

那么向更复杂的异步框架迁移时就有了可靠的对照基准。

此外，Swarm 中 Agent 的 handoff 机制和上下文变量传递。

为综述中讨论的多智能体故障归因问题（如 c-006 MASPrism、p-022 Which Agent Causes Task Failures and When?）提供了一个简洁的实验平台。

由于 Swarm 的 handoff 是显式且同步的，故障传播路径相对清晰。

这使其成为验证失败归因算法的理想基线系统。

在多智能体系统中，当一个任务失败时，确定"哪个智能体在何时导致失败"是一个核心难题。

Swarm 的架构使得每个决策点和交接点都可以被精确记录，从而简化了事后分析。

研究者可以利用 Swarm 构建可控的失败场景，系统地评估不同归因算法的准确性。

最后，Swarm 被 Agents SDK 取代这一事实本身也反映了多智能体框架从"教育探索"向"生产就绪"快速演进的行业趋势。

Agents SDK 保留了 Swarm 的核心原语（Agent、handoff、工具调用）。

但增加了追踪（tracing）、守卫（guardrails）、并行执行等企业级功能。

综述在讨论框架选型时需要明确区分这两个阶段的工具能力边界。

避免将教育框架的能力假设不恰当地推广到生产场景。

同时，Swarm 的开源代码和简洁实现使其成为理解 Agents SDK 底层原理的最佳入门材料。

## 我的笔记

Swarm 的最大价值在于其**极简主义设计哲学**。

仅用 Agent 和 handoff 两个原语就支撑起了完整的多智能体编排能力。

这种设计选择证明了多智能体系统不一定需要复杂的编排引擎。

关键在于原语的语义是否足够清晰、组合性是否足够强。

然而，正是这种极简也带来了明显的局限：

没有内置的记忆管理、没有持久化线程、没有错误恢复机制。

这些都需要开发者自行实现。

对于教育目的而言，这种"裸机"式的抽象恰到好处。

因为它迫使学习者理解每个概念的本质。

但对于生产系统，这些缺失的能力往往意味着大量的重复建设。

Swarm 的示例代码虽然覆盖了多种场景。

但并未提供通用的持久化、重试或容错模式。

这是教育框架与生产框架之间的天然鸿沟。

从工程实践角度，Swarm 的 `client.run()` 循环设计非常值得玩味。

它将 LLM 调用、工具执行、智能体切换和上下文更新封装在一个同步循环中。

使得执行流程完全透明可控。

开发者可以清楚地看到每一步发生了什么，没有隐藏的状态转换或异步回调。

但这种设计也意味着所有操作都在一个阻塞式循环中完成。

对于需要长时间运行或高并发场景的智能体系统来说，可能需要额外的异步包装。

Swarm 的 Python 实现（100% Python）也决定了其性能特征。

对于高吞吐场景可能不是最优选择。

如果要在生产环境中处理大量并发请求，开发者需要考虑在 Swarm 之上添加异步任务队列或进程池。

官方明确将 Swarm 定位为"教育性"资源并推荐迁移到 Agents SDK。

这是一个诚实且负责任的做法。

对于综述写作而言，Swarm 更适合作为**概念阐释的历史参考**。

而非当前生产实践的首选推荐。

在对比不同多智能体框架时，可以将 Swarm/Agents SDK 的演进路径与 LangGraph、CrewAI、AutoGen 等其他框架的发展轨迹进行并列分析。

从而揭示"轻量级编排"到"企业级编排"的共性演进规律。

值得注意的是，Swarm 的"stateless"设计理念与 LangGraph 的"state machine"理念形成了有趣的对比。

代表了多智能体编排的两种不同哲学取向：

前者强调简单和透明，后者强调状态管理和工作流控制。

一个有趣的观察是：Swarm 的 handoff 机制虽然简单。

但其语义与综述中提到的 OpenAI Agents SDK Traces（a-020）中的 agent 决策追踪天然契合。

每一次 handoff 都是一个清晰的追踪边界点。

这说明早期的框架设计即使出于简化目的，也可能无意中为后续的可观测性需求预留了良好的结构基础。

如果 Swarm 在设计之初就内建追踪能力，其教育价值可能会进一步提升。

因为学习者可以直接观察多智能体协作的完整执行轨迹，而不仅仅是阅读代码逻辑。

这种"可观测性友好"的设计原则应该在未来的教育框架中得到更多重视。

对于希望基于 Swarm 理解多智能体系统的研究者。

建议重点关注其 handoff 的触发条件设计、上下文变量的传递语义以及指令隔离机制。

这三个设计决策共同定义了 Swarm 中"智能体自治"与"系统协调"之间的平衡方式。

也是后续框架（包括 Agents SDK）在这些维度上进行扩展和优化的起点。

特别是上下文变量的"共享字典"模式，虽然简单。

但在多智能体系统中如何实现安全、可控的状态共享，仍然是一个值得深入研究的课题。

对于综述写作，建议将 Swarm 与 a-020（Agents SDK）配对分析。

展示同一团队从教育原型到生产框架的演进思路。
