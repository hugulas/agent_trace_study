# SWE-Chain: Benchmarking Coding Agents on Chained Release-Level Package Upgrades

## 核心信息

- 标题: SWE-Chain: Benchmarking Coding Agents on Chained Release-Level Package Upgrades
- 作者: Man Ho Lam, Chaozheng Wang, Hange Liu, Jingyu Xiao, Haau-sing Li, Jen-tse Huang, Terry Yue Zhuo, Michael R. Lyu
- 年份: 2026
- 会议/期刊: ArXiv.org
- DOI: 10.48550/arXiv.2605.14415
- 论文链接: https://arxiv.org/abs/2605.14415

## 原文摘要翻译

由大语言模型驱动的编码智能体日益被期望执行超越孤立问题修复的真实软件维护任务。现有基准测试已逐步转向逼真的软件演化场景，但它们很少以软件包发布的粒度来捕捉持续维护过程。在这个粒度上，变更被打包、发布并被后续版本继承。

本文提出 SWE-Chain，一个用于评估智能体在链式发布级软件包升级任务上的基准测试，其中每一次版本过渡都建立在智能体此前修改过的代码库之上。为了生成升级规格说明，作者设计了一套分治式合成管线，将每个版本过渡对应的发布说明与代码差异进行对齐，从而确保需求基于实际代码变更、对智能体具有信息量且可实际实现。

数据集包含横跨九个真实 Python 软件包的十二条升级链，共一百五十五次版本过渡和一千六百六十条有依据的升级需求。在九种前沿智能体配置下，智能体在允许一次修复的制度下平均解决率为 44.8%，精确率为 65.4%，F1 值为 50.2%。其中表现最佳的闭源模型解决率达到 60.8%，精确率为 80.6%，F1 值为 68.5%。这些结果表明 SWE-Chain 既可行又具有区分度，同时揭示当前智能体在跨链式软件包发布进行正确升级时仍然困难，且容易破坏既有功能。

## 创新点

1. **链式发布级维护评估范式**。SWE-Chain 首次将评估粒度定位在官方软件包发布边界，而非任意提交对或合成检查点。每次升级步骤都基于智能体上一轮修改后的代码库，真实模拟了前序错误会向后传递的维护动力学。这比孤立问题修复和孤立端到端过渡更能反映持续维护场景。

2. **DecompSynth 分治式规格合成管线**。现有基准直接将原始发布说明与代码仓库问题内容拼接，导致智能体需要处理终端输出、复现片段等噪声。DecompSynth 先将代码差异块与发布说明任务进行多标签匹配，再为每个任务合成结构化规格，确保规格基于真实代码变更、信息完整且可实施。

3. **双制度评估与多维指标**。作者设计了仅升级实现和升级后允许一次修复两种评估制度，并引入解决率、精确率、F1 值三个链级指标。解决率衡量升级能力，精确率通过惩罚既有功能回归来度量维护安全性，F1 值平衡二者。该制度缓解了因环境不兼容导致的过度惩罚问题。

4. **面向真实生态的大规模链式数据集**。数据集覆盖九个真实软件包、十二条升级链、一百五十五次版本过渡和一千六百六十条有依据需求，并通过难度公式结合代码库规模与升级密度进行排序，确保基准具有区分度。

## 一句话总结

SWE-Chain 通过 DecompSynth 将软件包发布说明与代码差异对齐，首次在十二条真实升级链上系统评估了前沿编码智能体的链式发布级维护能力，发现当前智能体平均只能解决不到一半的升级需求且频繁破坏既有功能。

## 研究问题

随着大语言模型编码智能体从孤立任务走向仓库级操作，评估范式也从任务级问题演进到真实项目问题修复。然而，一个关键缺口始终存在：现有基准的任务边界与上游维护者实际发布软件版本的时机不对齐。

具体而言，SWE-CI 使用基准自选的基础到目标提交对，SlopCodeBench 使用合成检查点，EvoClaw 从提交历史重建里程碑。这些分段方式虽然揭示了迭代代码退化和持续集成式可维护性的失败，却未触及发布边界处的真实维护动力学。SWE-EVO 首次将发布版本作为评估单元，但其任务仍是孤立的端到端过渡，规格由发布说明与代码仓库问题原始内容拼接而成，充斥着终端输出和图像引用等噪声，且无法评估智能体能否在自身修改的基础上持续维护代码库。

因此，本文的核心研究问题是：智能体能否在一条发布链上持续维护真实软件包，在携带自身变更向前推进的同时不破坏既有功能？

## 数据与任务定义

**数据集规模**。SWE-Chain 包含十二条升级链，覆盖九个真实 Python 软件包，包括网络框架、测试工具、数据处理和包管理等不同领域，共一百五十五次版本过渡和一千六百六十条有依据的升级需求。每条链由一系列连续的官方发布版本组成，智能体需要按顺序完成每一次版本升级。

**任务设置**。对于每次版本过渡，作者预先定义与该升级相关的测试集合。这些测试在原始旧版本上失败，在原始新版本上通过。智能体的目标是在其维护的旧版本代码库上实施升级，使得升级后的代码库满足这些升级相关测试通过，同时不破坏此前已通过的测试。

**评估制度**。作者采用两种制度：
- **Build**：智能体仅根据规格实现升级，然后运行测试。
- **Build+Fix**：Build 之后，若出现收集错误或导入错误等环境兼容性问题，智能体获得一次基于错误反馈的修复机会。

**反作弊控制**。为防止智能体走捷径，作者实施多层防护：显式禁止智能体访问已发布的源代码或复制官方源码；限制可用工具和命令以防止外部依赖安装和远程仓库访问；在独立容器中运行评估以避免工作区污染；覆盖目标版本的测试配置以防止通过修改测试配置来绕过评估。

## 方法主线

SWE-Chain 的核心方法论包含两部分：一是 DecompSynth 分治式规格合成管线，用于从发布说明与代码差异中生成结构化、有依据且可实施的升级规格；二是顺序版本升级评估协议，用于在链式场景下公平评估智能体的维护能力。

### DecompSynth 规格合成

DecompSynth 的目标是将粗粒度的发布级升级分解为一组细粒度、结构化且每个都与具体代码变更对齐的任务规格。其输入为某次版本过渡的发布说明和代码差异集合，输出为一组结构化规格。

**分治策略的动机**。直接让智能体从原始发布说明和完整代码差异中推断任务会导致信息过载，且发布说明中往往一条列表项对应多处代码变更。分治策略先将差异块与发布说明条目进行对齐，再逐任务合成规格，显著降低了单次推理的复杂度。

### 机制流程

DecompSynth 与评估协议的核心执行链可分为以下三步：

1. **差异分块与批量匹配（Divide）**。该步骤的输入为任务化发布说明（已按条目拆分为任务列表）和文件级代码差异批次。匹配智能体对每一批次执行多标签差异块匹配，提取每个差异块的语义特征，并将其对齐到一个或多个发布说明任务，未匹配的块则标记为待处理。匹配采用贪心打包策略，约束每批次最多二十个文件、一百五十个差异块、三千行差异。输出为按任务分组的差异块桶和未匹配差异块集合。

2. **逐任务规格合成（Conquer）**。该步骤的输入为发布说明、已匹配的差异块桶和合成智能体。合成智能体对每个发布说明任务进行编码，融合该任务对应的代码差异和测试差异信息，解码生成包含问题陈述、期望与约束、行为描述和验收标准的结构化规格。若合成失败，可在预算内重试。输出为每个升级步骤的结构化任务规格集合。

3. **顺序版本升级执行与评估（Build+Fix）**。该步骤的输入为初始代码库（原始起始版本）和按顺序排列的各步骤规格。智能体依次执行每个升级步骤，在每一步基于当前代码库和该步规格更新代码。评估在独立容器中进行，使用目标版本的完整测试套件。若 Build 阶段出现环境兼容性问题，智能体获得一次基于错误反馈的修复机会。评估系统对每个升级步骤的所有测试在升级前后代码库上分别执行，按测试是否属于升级相关集合以及状态转移进行分类。输出为每条链的解决率、精确率和 F1 值指标。

## 关键结果

**主实验结果**。在九种前沿智能体配置下，允许一次修复的制度平均表现为：解决率 44.8%，精确率 65.4%，F1 值 50.2%。相比之下，仅实现升级的制度平均解决率为 39.7%，精确率为 54.0%。额外修复将精确率提升了十一点四个百分点，而解决率仅提升五点一个百分点，说明修复步骤主要处理了脆弱性失败和回归问题，而非直接解决更多升级需求。

**最佳配置表现**。表现最优的闭源模型解决率为 60.8%，精确率 80.6%，F1 值 68.5%。排在其后的另一闭源模型解决率为 57.5%，精确率 80.1%，F1 值 64.8%。同系列较早版本和轻量版本表现明显落后，F1 值分别为 50.8% 和 45.9%。

**开源与闭源差距**。闭源前沿模型整体优于开源权重模型。表现最差的配置解决率仅 20.2%，精确率 34.2%，F1 值 21.2%。该配置与最佳配置的解决率差距高达四十点六个百分点。

**效率与成本**。性能最强的配置也是成本最高的：最佳模型平均每条链耗时三点四三小时、花费一百五十点三九美元、消耗约三点五亿词元。次优模型为三点四八小时、一百三十一点三四美元、约一点八亿词元。值得注意的是，更高资源消耗未必带来更好结果。某轻量闭源模型运行时间最长（四点八四小时）但 F1 值仅 45.9%。各模型的工具调用成功率都很高，说明性能差异并非由工具执行失败驱动。

**规格粒度消融**。在测试工具 8.0 链上，作者对比了五级规格粒度。最佳模型在粗粒度下的解决率为 44.5%，精确率仅 8.9%；而在最细粒度下，解决率升至 73.9%，精确率升至 93.5%。这表明细粒度规格能显著降低智能体的理解负担并减少误操作。

## 深度分析

**链间性能差异显著，难度由多因素共同决定**。从热图可以看出，即使在较简单的链上，智能体的平均解决率也低于 70%；而在较难的链上，解决率可降至 30% 以下。代码库规模只能部分解释这一趋势：例如某包管理工具的代码库比测试框架小，但解决率更低，因为该工具每次升级的平均代码差异规模更大。这说明链的难度取决于代码库大小、升级规模和升级密度的联合作用。

**没有单一模型在所有链上占优**。最佳闭源模型在数据处理库和测试框架等链上更强，而另一系列在网页框架和模板引擎等链上领先，反映出不同模型对软件包结构具有特异性优势。开源权重模型在简单链上仍具竞争力，但在困难链上性能急剧下降，这进一步验证了 SWE-Chain 的区分度。

**额外修复的设计哲学值得推敲**。该制度将平均精确率从 54.0% 提升到 65.4%，但解决率仅提升 5.1 个百分点，说明修复阶段主要用于处理环境兼容性问题而非实质性解决升级需求。这种设计是必要的——在黑盒评估中，智能体的实现可能因环境不兼容触发广泛失败，允许一次基于错误反馈的修正可缓解过度惩罚。但从另一面看，这也说明当前智能体在正确理解升级意图并精准实施这一核心能力上仍有巨大提升空间。

**规格质量是性能瓶颈**。消融实验显示，从最粗粒度到最细粒度，最佳模型的精确率从 8.9% 跃升至 93.5%，解决率从 44.5% 升至 73.9%。这意味着如果规格过于粗略，即使是当前最强智能体也会产生大量误操作和回归。DecompSynth 的价值在此凸显：它通过将发布级升级分解为与代码差异对齐的细粒度任务，显著提升了智能体的可执行性。

## 局限

1. **语言局限**。SWE-Chain 目前仅覆盖 Python 软件包，未涉及 JavaScript、Java、Go 等其他主流语言生态。不同语言的包管理、发布节奏和测试文化差异显著，结论能否迁移尚待验证。

2. **规格合成的固有不确定性**。DecompSynth 的差异匹配步骤依赖大语言模型进行差异块分配，作者指出这是管线中最易受随机性影响的部分。虽然附录显示在最大版本过渡上运行十次仍保持稳定，但对于边界模糊的变更，匹配错误可能导致规格与代码脱节。

3. **未覆盖并发与冲突场景**。真实维护中常出现多个依赖包同时升级、版本冲突、破坏性变更需要协调处理的情况。SWE-Chain 的评估是顺序单包升级，尚未涉及这些更复杂的维护动力学。

4. **修复制度的边界**。额外修复仅允许一次基于错误反馈的修正，且主要针对环境兼容性问题。对于更深层的逻辑错误或未完成的升级实现，修复阶段无能为力。这使得 44.8% 的解决率更接近当前智能体能力的上限估计。

5. **测试覆盖的隐含假设**。评估假设目标版本的测试套件能够充分捕捉升级需求。如果某些升级行为未被测试覆盖，则智能体可能通过评估却未真正满足升级要求。

## 我的笔记

**与 SWE-Bench 生态的关系**。SWE-Bench 开创了在真实代码仓库问题上评估编码智能体的范式，其后续变种持续改进评估可靠性。SWE-Chain 不是 SWE-Bench 的替代，而是对其时间维度的自然延伸——从修复单个问题延伸到维护整条发布链。两者结合可以构成更完整的智能体能力画像：SWE-Bench 测局部诊断与修复能力，SWE-Chain 测长期维护与演化能力。

**对未来工作的启示**。SWE-Chain 的结果表明，当前编码智能体在持续维护场景下远未达到实用水平。几个值得探索的方向包括：一是将版本间依赖关系显式建模到智能体的推理过程中；二是开发能够识别并回滚自身破坏性变更的自纠错机制；三是探索多智能体协作维护，让不同智能体分别负责升级实施、回归测试和代码审查。

**作为训练信号的价值**。SWE-Chain 不仅是一个评估基准，其 DecompSynth 合成的细粒度规格和链式代码库状态也可以作为训练数据，用于微调专门面向软件维护的智能体。相比于从原始提交历史或问题讨论中学习，结构化规格提供的监督信号更明确、噪声更低。
