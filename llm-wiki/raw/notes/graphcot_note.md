# Scaling Graph Chain-of-Thought Reasoning: A Multi-Agent Framework with Efficient LLM Serving

## 原文摘要翻译

图链式思维（Graph-CoT）使大语言模型能够对图结构知识进行逐步推理，但现有流水线存在准确率低下、token 消耗过度、延迟高且吞吐量低等问题，根源在于单智能体单体提示、重复上下文重新编码以及低效的推理执行。

本文提出 GLM，首个与优化大语言模型服务架构协同设计的多智能体 Graph-CoT 系统。GLM 将推理分解为分类、推理、动作生成和图检索四类专用智能体，通过分支执行与选择性上下文共享来缩短提示长度并减少推理轮次，同时保持推理质量，从而提升准确率并降低整体 token 消耗。

为扩展推理规模，本文引入 Graph-CoT 感知的大语言模型推理机制，包含图特定的 KV 缓存管理、基于优先级的淘汰策略与流水线执行，以提升服务效率。

实验表明，与最先进的 Graph-CoT 基线相比，GLM 的答案准确率最高提升 38%，token 成本最高降低 95.7%，推理延迟降低 90.3%，吞吐量最高提升 15.1 倍，从而支持大规模复杂现实世界推理的高效部署。

## 创新点

1. **多智能体 Graph-CoT 推理框架**：突破单智能体单体提示的局限，将推理拆分为分类智能体（C-Agent）、推理智能体（R-Agent）、动作智能体（A-Agent）和图检索器四类协作模块。不同智能体各司其职，使确定性查询只需一次分类和一次动作调用，非确定性查询则通过笔记本迭代收集信息，显著减少无效推理轮次和上下文膨胀。

2. **推理与服务协同设计**：首次将 Graph-CoT 推理框架与底层大语言模型服务架构联合优化，而非仅在应用层改进提示策略。系统级优化包括图特定的 KV 缓存复用、基于优先级的缓存淘汰以及检索与解码的流水线重叠，直接作用于推理瓶颈。

3. **顶点中心的 KV 缓存复用模型**：针对图检索结果缺乏固定文本分块边界的特性，提出以顶点为粒度的 KV 缓存复用策略。当不同查询请求共享相同顶点时，可直接复用对应 KV 缓存，缓解多智能体高频调用下的缓存碎片化问题。

4. **显著的效率与准确率双提升**：在 GRBench 基准上，GLM 相较 Graph-CoT 基线实现准确率最高提升 38%、token 消耗降低 95.7%、端到端延迟降低 90.3%、吞吐量提升 15.1 倍。相比基础大语言模型、纯文本检索增强、图检索增强的提升幅度分别为 60%、62%、55%。

## 一句话总结

GLM 通过将 Graph-CoT 推理分解为多个专用智能体并协同设计底层服务架构，在保持高准确率的同时将 token 消耗、延迟和吞吐量推进了一个数量级。

## 研究问题

现有 Graph-CoT 系统在复杂图结构知识推理中面临三重瓶颈。

**单智能体能力边界**：现有框架采用单智能体单体提示，每一步都需重新编码完整上下文和中间轨迹，导致长链条推理不稳定。实验显示，随着推理步数增加，Graph-CoT 的 GPTScore 从 3 步时的 0.42 跌至 9 步时的 0.17，准确率急剧衰减。

**重复编码带来的 token 浪费**：为了维持一定准确率，现有系统依赖多轮迭代推理，每一步都重复编码前缀和不断增长的中间历史，单次查询成本高达 1.9 至 3.6 美元（GPT-4.5），在商业部署中不可接受。

**系统级执行低效**：单智能体串行执行模式下，图检索延迟占总时间最高达 23%，且 KV 缓存命中率在并发场景下偏低，进一步限制了吞吐量和响应速度。

因此，论文核心问题是：如何在保持或提升 Graph-CoT 推理准确率的前提下，系统性地降低 token 成本、端到端延迟，并提升服务吞吐量？

## 数据与任务定义

实验基于 GRBench 基准开展。该基准包含 1,740 条人工筛选的问答对，覆盖学术、电商、文献、医疗和法律 5 个领域，共 10 张真实世界图数据。每个问题需要以下三类推理之一：事实检索、多跳推理、归纳推断。

评估指标采用 ROUGE-L（R-L）与 GPTScore。

R-L 衡量推理步骤中正确子序列的比例；GPTScore 则由 GPT-4 判断输出语义正确性的百分比。

主实验主干模型为 Qwen3-235B-A22B，解码温度固定为 $t = 0$ 以保证输出确定性。

## 方法主线

GLM 的整体架构由两大模块组成：基于大语言模型的智能体模块和图检索增强生成（Graph RAG）检索器模块，均基于约 3,000 行 Python 代码在 vLLM（v0.8.5）之上实现。

### 智能体定义

每个智能体由输入空间、提示模板与输出空间三元组刻画。

分类智能体 C-Agent 的输入为自然语言问题 $q$，输出为二分类结果 $o_c$，用于判断查询是否为确定性问题。

推理智能体 R-Agent 的输入为当前笔记本 $n$ 与问题 $q$，输出为推理结论或 finish 信号。笔记本维护已收集的中间信息。

动作智能体 A-Agent 的输入为缺失信息或完整笔记本，输出为可执行的图检索代码片段。

### 机制流程

核心推理流程遵循算法 1 的多智能体协作范式：

1. **输入与分类**：系统接收自然语言问题 $q$ 作为输入，由分类智能体 C-Agent 基于提示模板进行编码与判断，输出二分类结果 $o_c$，用于区分确定性查询与非确定性查询。输出结果送入后续分支模块。

2. **确定性路径的快速响应**：若 $o_c$ 为确定性标记，动作智能体 A-Agent 直接对输入问题进行解码，生成图检索代码；Graph RAG 检索器执行该代码并提取结果，直接返回答案，无需迭代推理。

3. **非确定性路径的迭代信息收集**：若 $o_c$ 为非确定性标记，系统初始化空笔记本 $n \leftarrow \{\}$。推理智能体 R-Agent 对当前笔记本与问题进行查询与推理，输出缺失信息；动作智能体将其编码为检索代码；检索器提取对应信息后更新笔记本。该循环持续直至推理智能体输出 finish 信号。

4. **最终答案生成与调度输出**：基于完整笔记本中的全部信息，系统对汇总状态进行编码并生成最终答案，输出到用户端。并发场景下，各查询线程共享智能体与检索器实例，由 vLLM 后端动态批处理并在多 GPU 间调度解码。

### 服务层优化

为将上述推理框架扩展至高并发场景，GLM 在 vLLM 服务层引入三项针对性优化。

**顶点中心 KV 缓存复用**：图检索结果不以固定文本块为边界，而以顶点为粒度组织。当多个请求涉及同一顶点时，直接复用对应 KV 缓存，减少重复前缀计算。

**基于优先级的缓存淘汰**：为不同智能体的缓存条目分配优先级，在显存受限时优先保留高复用概率的顶点缓存，提升高并发下的缓存命中率。

**流水线执行**：将检索器的图数据获取与 LLM 解码阶段的剩余 token 生成重叠执行。当动作智能体输出的检索代码包含 RetrieveNode 调用时，系统在等待 LLM 解码的同时启动图检索，隐藏检索延迟。

## 关键结果

GLM 与四类基线对比：基础大语言模型、纯文本检索增强、图检索增强以及单智能体 Graph-CoT。

### 准确率

在 GRBench 五个领域上，GLM 的 ROUGE-L 与 GPTScore 均显著优于所有基线。

具体地，GLM 相较基础大语言模型准确率提升 60%，相较纯文本检索增强提升 62%，相较图检索增强提升 55%，相较 Graph-CoT 提升 38%。

在电商领域，GLM 的 R-L 达到 0.77，而 Graph-CoT 仅为 0.39。

## 深度分析

### 准确率随推理步数衰减的原因

单智能体 Graph-CoT 的准确率衰减根源在于长上下文中的信息丢失。单智能体需在单条提示中同时维护原始问题、历史动作和中间结果，随着步数增加，模型对早期上下文的注意力稀释，导致错误决策。GLM 通过将推理拆分为 R-Agent 与 A-Agent，使每个智能体仅在最小任务相关上下文中运行，从根本上抑制了这种衰减。

### 各阶段 token 分布与系统瓶颈

在非确定性问题中，分类、推理、动作三个智能体的 token 使用比例约为 2.5% : 82.3% : 15.1%。推理智能体承担最重的输入 token 量，增加了预填充和解码开销；动作智能体则生成更长的输出序列，需要更多解码步数。这一分布说明，未来若需进一步优化延迟，应优先压缩推理智能体的上下文长度或提升其解码效率。

### 错误分类与容错机制

Graph-CoT 最主要的失败类型是意外智能体输出，源于单智能体处理长上下文时丢失中间信息；其次是步数超限，发生在需要从多节点获取信息的查询中。引入代码生成优化后，步数超限从 38% 降至 8%，但代码执行错误从 2% 升至 20%，因为大语言模型需要合成完整代码而非从预定义函数中选择。GLM 的多智能体设计将意外输出错误从 41% 压至 2%，同时通过超时与恢复机制提升了高负载下的可靠性。

### 消融与系统优化贡献

在无流水线优化时，平均响应延迟为 4.3 至 10.2 秒。流水线执行通过重叠检索与解码显著降低端到端延迟；顶点中心 KV 缓存复用提升了并发场景下的缓存命中率；优先级淘汰策略在高吞吐负载下维持了较高的缓存命中率。三者结合使得 GLM 在延迟和吞吐量上均取得一个数量级的改进。

## 局限

论文明确承认两项主要局限。

**准确率受限于主干模型能力**：GLM 的最终准确率仍受底层大语言模型能力的上限约束。当问题需要复杂归纳或跨领域迁移时，分类和推理智能体仍可能出错。未来可通过知识蒸馏、轻量微调以及更紧密的检索与推理校准来缓解。

**图检索延迟仍是瓶颈**：在无 GPU 加速的情况下，图检索过程可能引入显著延迟，在某些场景下构成系统主导瓶颈。论文指出，未来需要优化检索流水线和整体系统架构，以在保持推理质量的同时进一步降低延迟和资源消耗。

此外，实验仅在 GRBench 的五个领域内进行，虽然领域跨度较大，但尚未验证在动态图、时序图或超大规模图上的表现。代码生成带来的执行错误率上升也提示，动作智能体的代码合成可靠性仍有提升空间。

## 我的笔记

GLM 的核心价值在于提出了推理框架与推理服务协同设计的范式。过往 Graph-CoT 研究多聚焦于提示工程或检索策略，而将底层大语言模型服务视为黑盒。GLM 证明，当上层多智能体分解与下层 KV 缓存管理、流水线调度联合优化时，可以获得远超纯算法改进的系统性收益。这种协同设计思路对以下方向具有借鉴意义。

**长记忆智能体**：将复杂任务拆分为分类、推理、动作三类专用模块，降低单智能体的上下文长度需求。

**结构化 RAG 服务**：顶点中心的 KV 缓存复用和优先级淘汰策略不仅适用于图数据，也可推广到任何具有结构化依赖的检索增强生成系统。

**多智能体系统的成本优化**：通过确定性查询的快速路径，GLM 将简单查询的 token 消耗压至极低，这对商业部署中的成本控制具有直接意义。

一个值得后续跟踪的问题是：当主干模型从 Qwen3-235B-A22B 切换为更小规模的模型时，分类和动作智能体的错误率是否会显著上升，以及 LoRA 微调能否在保持效率增益的同时弥补准确率损失。
