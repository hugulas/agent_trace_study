# Process-Centric Analysis of Agentic Software Systems

## 核心信息

- **标题**: Process-Centric Analysis of Agentic Software Systems
- **作者**: Shuyang Liu, Yang Chen, Rahul Krishna, Saurabh Sinha, Jatin Ganhotra, Reyhaneh Jabbarvand
- **年份**: 2026
- **来源**: Proceedings of the ACM on Programming Languages (PACMPL, OOPSLA1)
- **DOI**: 10.1145/3798271
- **arXiv**: 2512.02393
- **PDF**: `[c-009]-process-centric-analysis-of-agentic-software-systems-graphec.pdf`
- **证据质量**: medium

## 内容摘要

这篇论文针对当前智能体系统评估中普遍存在的"结果中心主义"偏见提出了系统性的批判与改进方案。
作者指出，现有的智能体评估方法往往只关注最终任务是否成功完成，即通过最后一步的成败来判断性能，
而这种狭隘的视角完全忽略了中间执行过程中蕴含的丰富信息。
结果中心评估无法解释智能体如何推理、如何规划、如何行动以及如何动态调整策略，
也无法区分成功是源于系统性的推理能力还是纯粹的偶然运气。
为了填补这一关键空白，论文提出了 Graphectory 这一创新性的图结构表示方法，
用于系统性地编码智能体执行轨迹中的时间关系与语义关系。

Graphectory 的设计灵感来源于传统软件工程中将程序表示为控制流图、数据依赖图等结构化图表示的做法。
它将原本仅以线性时间序列形式存在的原始轨迹日志，自动转换为一种有向循环图。
在这个图中，每个节点代表智能体在任务执行过程中所采取的一个具体行动，
节点之间的边则编码两种不同性质的关系：
时序边（temporal edge, TE）捕捉行动之间的先后执行顺序，反映智能体的问题解决推进过程；
结构边（structural edge, SE）则表示问题空间中实体之间的层级包含关系，
例如在软件工程场景中可以表示为目录到文件、文件到代码块的导航层级，
在机器人场景中可以表示为房间到区域、区域到物体的空间层级。
通过这两种边的组合，Graphectory 能够远比线性日志更准确地刻画智能体行为的执行流与问题解决策略。

与 Graphectory 配套的是 Langutory，一种人类可读的轨迹语言抽象。
Langutory 通过将 Graphectory 中的图节点投影到不同的字母表上，生成紧凑的符号序列。
例如，当字母表符号对应于问题特定的逻辑阶段时，
Langutory 可以提供智能体在解决问题过程中所遵循的相位序列的紧凑概览，
从而支持快速识别不同智能体之间共享或 divergent 的问题解决策略。
更重要的是，Langutory 能够与系统提示中向智能体规定的预期计划进行直接比对，
从而检测智能体的实际执行轨迹是否偏离了预期规划。
这种能力对于发现智能体的策略缺陷和不必要的探索行为非常有价值。

论文的实验设计以 SWE-agent 和 OpenHands 两个主流的软件工程智能体为研究对象，
这是当前在代码修复任务上表现最突出的两类智能体架构。
作者结合 DeepSeek-V3、DeepSeek-R1、Devstral-small-2505 和 Claude Sonnet 4 四种具有代表性的骨干大语言模型，
在 SWE-Bench Verified 基准的 12 个真实代码仓库上系统性地收集了约 4000 条执行轨迹进行分析。
这些轨迹涵盖了从简单到复杂的各类 issue，涉及缺陷定位、代码修改、测试验证等多个环节，
具有高度的多样性和代表性。
所有的 Graphectory 构建、Langutory 生成和过程中心分析均完全自动化，
处理 4000 条轨迹仅需不到四分钟，充分证明了该方法的可扩展性。

实验揭示了多个重要发现。
首先，使用更强模型或更丰富提示的智能体，其 Graphectory 结构明显更复杂，
反映出更深的探索深度、更广泛的上下文收集范围以及更充分的提交前验证行为。
其次，对于成功解决的 issue，智能体的策略往往遵循定位→修补→验证的连贯步骤模式，
而未成功的实例则普遍表现出混乱、重复编辑同一位置、无目的回溯等行为特征。
第三，即使最终成功修复了 issue，智能体也常常表现出显著的过程低效性，
例如不必要的文件浏览、重复的测试运行、冗长的思考过程等，导致轨迹时间被不必要地拉长。
这些发现直接证明了过程中心评估的必要性：只看最终结果会严重低估智能体的效率问题。

论文还提出了一种新颖的在线监控与干预技术。
该技术在智能体执行过程中实时构建并分析 Graphectory 和 Langutory，
一旦检测到轨迹异常模式，如重复编辑、缺少验证步骤、过度回溯等，
系统会立即向智能体发送针对性的诊断消息，建议其调整当前策略；
在必要时，系统还可以执行轨迹回滚操作。
实验表明，对 86 个在多次执行中一致表现不佳的困难实例应用在线干预后，
94.1% 的轨迹转变为非问题轨迹，平均问题解决率提升 11.9%，
而额外的时间开销不到 10 毫秒，几乎实现了零成本的质量改进。
这一结果极具实践价值，说明通过精细的过程监控和及时反馈，
可以在不更换模型、不增加计算资源的前提下显著提升智能体的实际表现。

## 关键要点

1. **Graphectory 图表示**：
   将线性轨迹日志自动转换为有向循环图 G=(V, TE ⊎ SE)，
   节点为行动，时序边编码执行顺序，结构边编码问题空间导航层级关系。

2. **Langutory 语言抽象**：
   Graphectory 的紧凑人类可读抽象，通过投影 π: V→Φ 生成符号序列，
   支持策略比较、规划偏差检测和快速语义概览。

3. **过程中心指标**：
   包括图规模、回边数量、相位流分析（Phase Flow Analysis）、模式检测（Pattern Detection）等，
   用于量化轨迹复杂度与效率特征。

4. **实验规模与自动化**：
   4000 条轨迹、8 个〈agent, model〉组合、12 个代码仓库；
   全自动分析，4 分钟内完成处理，克服了手工分析的人类偏见与不可扩展性问题。

5. **核心实证发现**：
   - 未解决 issue 的 Graphectory 显著大于已解决 issue，更多回边表示更多重复和无效探索；
   - 任务难度增加时，轨迹复杂度和策略切换频率同步上升；
   - 成功与效率之间存在显著不平衡，很多成功轨迹包含大量冗余步骤；
   - 不同模型在相同任务上的策略模式存在系统性差异。

6. **在线监控与干预**：
   实时构建 Graphectory/Langutory，检测异常后通知智能体调整策略或回滚；
   对困难实例问题解决率提升 6.9%–23.5%，轨迹显著缩短，开销近零。

7. **方法论贡献**：
   首次将程序分析中的图表示方法系统性地应用于智能体轨迹分析，
   为社区提供了一个可扩展的、自动化的过程中心评估工具箱。

8. **开源数据集**：
   论文提供了 4000 条轨迹的 Graphectory 和 Langutory 表示，
   以及相应的分析代码，便于后续研究者在此基础上开展更深入的过程中心研究。

## 与综述的关联

这篇论文与本综述关于智能体可观测性与失败诊断的主题高度相关。
Graphectory 提供了一种从过程视角理解智能体行为的结构化方法，
直接补充了传统结果中心评估的盲区。
在综述讨论的智能体评估方法维度上，
Graphectory 代表了一种从"黑箱测试"向"白箱分析"演进的重要方向，
与 AI-NativeBench（c-008）的白箱 trace-based 评估理念形成呼应，
但 Graphectory 更侧重于通用的图结构表示和跨智能体的可比性分析。

论文中提出的在线监控与干预机制，
直接对应综述中讨论的"运行时可观测性"与"自适应纠偏"方向。
当前业界对智能体的监控主要集中在 API 调用链延迟和错误率等指标上，
而 Graphectory 的在线分析则深入到策略层面的异常检测，
这种细粒度的过程监控代表了下一代智能体运维技术的可能形态。
此外，论文对 SWE-agent 和 OpenHands 的大规模轨迹分析，
为综述中关于软件工程智能体可靠性瓶颈的讨论提供了高质量的实证数据支撑。

在可观测性技术栈的坐标系中，
Graphectory 与 AgentTrace（c-011）处于互补位置：
Graphectory 侧重于轨迹的离线结构化表示与在线分析，
而 AgentTrace 侧重于运行时的结构化日志采集与遥测基础设施。
两者可以形成上下层配合：
AgentTrace 负责在运行时采集高质量的结构化事件流，
Graphectory 负责将这些事件流转换为语义图并进行深度分析。
同时，论文中提出的过程中心指标（如回边数、相位流转移概率、模式检测命中率）
也可直接作为综述建议的"细粒度评估维度"被后续工作所采用。

## 我的笔记

Graphectory 的核心价值在于将"轨迹"从线性序列提升为语义图，
这一视角转变对智能体评估具有方法论层面的重要意义。
传统评估将智能体视为黑箱，只看输入输出；
Graphectory 则打开了这个黑箱，让我们能看到智能体在问题空间中的导航路径、策略切换点和效率瓶颈。
这种图表示方法借鉴了程序分析领域长期积累的控制流图、依赖图等技术，
但将其创造性地应用于智能体轨迹这一新的数据对象上，体现了跨领域知识迁移的力量。

特别值得注意的是，论文揭示了"成功不等于高效"这一现象。
很多成功修复 issue 的轨迹实际上包含大量冗余步骤和重复编辑，
比如 SWE-agent Dev 在修复 django-10973 时对 client.py 进行了多次重复编辑，
而 SWE-agent DSK-V3 则一次性完成修改。
从结果看两者都成功了，但从过程看效率差异巨大。
这种低效在结果中心评估中完全不可见，
却直接影响了实际部署中的成本、延迟和用户体验。
对于企业级智能体应用而言，这种过程低效可能意味着数倍的 token 成本和数倍的响应延迟，
是一个不可忽视的工程问题。

在线监控机制的设计非常巧妙：
它不是简单地限制智能体的步数或设置超时阈值，
而是通过实时构建 Graphectory 识别具体的策略缺陷类型，
如缺少验证步骤、重复编辑同一文件、无目的的项目结构浏览等，
然后给出针对性的诊断反馈。
这种方法比粗暴的截断更智能，也更尊重智能体的自主性——
它不是在限制智能体的能力，而是在帮助智能体更好地发挥自己的能力。
实验结果也证明了这种精细干预的有效性，
11.9% 的平均问题解决率提升和近零的时间开销，
使得这一技术具有很强的工程落地价值。

从局限性和未来方向来看，
当前 Graphectory 主要针对软件工程领域的智能体进行设计和验证，
其结构边中的层级关系定义（目录→文件→代码块）具有领域特定性。
将 Graphectory 推广到网页导航、科学发现、多智能体协作等其他领域时，
需要重新定义结构边的语义和层级体系。
此外，Langutory 的字母表设计目前主要依赖人工定义的问题语法，
未来可以探索自动化的字母表学习或自适应投影方法。
另一个有趣的方向是将 Graphectory 与学习算法结合，
让智能体不仅从最终结果的反馈中学习，
还能从过程图的模式中学习更优的问题解决策略，
类似于强化学习中的 credit assignment，但作用在图结构而非标量奖励上。
